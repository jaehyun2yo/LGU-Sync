# 023. 깊은 폴더 스캔 누락 + 다운로드 실패 수정

- **날짜:** 2026-03-10
- **브랜치:** master

## 변경 요약
외부웹하드 전체 스캔 시 12GB만 감지되는 문제(실제 13.99GB)와 깊은 폴더(3단계+) 파일 다운로드 실패 버그를 수정했다. 근본 원인은 공유 폴더 LIST API의 ITEM_ID(래핑 ID)를 다운로드 API에 전달한 것으로, ITEM_SRC_NO(원본 파일 ID)를 사용하도록 수정했다.

## 변경 파일
- `src/core/types/lguplus-client.types.ts` — `LGUplusFileItem`에 `itemSrcNo?: number` 필드 추가
- `src/core/lguplus-client.ts` — getFileList에 ITEM_SRC_NO 파싱 추가, getDownloadUrlInfo에 data.file null 방어, getAllFilesDeep 실패 폴더 재시도 2회+지수백오프로 강화
- `src/core/sync-engine.ts` — scanFolder에서 `lguplus_file_id: String(file.itemSrcNo ?? file.itemId)` 로 변경

## 주요 결정사항

### ITEM_ID vs ITEM_SRC_NO
- 실시간 감지 경로(히스토리 API)는 ITEM_SRC_NO를 반환하여 정상 동작
- 전체 스캔 경로(폴더 LIST API)는 ITEM_ID만 반환했으나, ITEM_SRC_NO도 존재하므로 파싱 추가
- `itemSrcNo ?? itemId` fallback으로 하위 호환성 유지

### pageSize 고정(20) 시도 → 원복
- DATA-8 이슈 대응으로 pageSize=20 고정을 시도했으나, 테스트 mock과 호환 불가
- 원래 `firstPage.items.length || 20` 방식이 과대 추정(빈 페이지 extra fetch)만 발생시키므로 안전
- 원래 코드로 복원

### 서킷 브레이커 연쇄 차단
- 513건의 동일 원인 실패(ITEM_ID→다운로드 API)가 failureThreshold=15 도달 후 OPEN
- 근본 원인(ITEM_SRC_NO) 수정으로 실패 자체가 해소됨
- getDownloadUrlInfo data.file null 방어로 TypeError 크래시 방지
