# 020. 다운로드 중복 체크 버그 수정

- **날짜:** 2026-03-10
- **브랜치:** fix/realtime-detection-download-bugs

## 변경 요약

`test:download-only`와 `test:full-sync` IPC 핸들러의 중복 파일 체크 로직이 완전히 깨져 있던 버그를 수정. `getFileByHistoryNo(file.itemId)` 호출이 항상 null을 반환하여 매 실행마다 DB에 중복 레코드가 생성되던 문제를 `getFileByLguplusFileId()`로 교체하여 해결.

## 변경 파일

- `src/core/types/state-manager.types.ts` — `IStateManager`에 `getFileByLguplusFileId()` 메서드 추가
- `src/core/state-manager.ts` — `getFileByLguplusFileId()` 구현 (lguplus_file_id 기반 조회, 최신 레코드 반환)
- `src/main/ipc-router.ts` — `test:download-only` 핸들러: 중복 체크를 `getFileByLguplusFileId`로 교체, `downloaded`/`completed` 상태 스킵, 기존 레코드 재활용
- `src/main/ipc-router.ts` — `test:full-sync` 핸들러: 동일 패턴 적용, `downloaded` 상태 파일은 업로드만 실행
- `tests/core/state-manager.test.ts` — 새 메서드 테스트 3건 추가
- `tests/core/sync-engine.test.ts` — mock에 `getFileByLguplusFileId` 추가
- `tests/core/file-detector.test.ts` — mock 추가
- `tests/core/file-detector-limitations.test.ts` — mock 추가
- `tests/core/folder-discovery.test.ts` — mock 추가
- `tests/core/pipeline-integration.test.ts` — mock 추가
- `tests/integration/setup.ts` — InMemoryStateManager에 메서드 추가

## 주요 결정사항

- `getFileByHistoryNo(file.itemId)`가 항상 null을 반환한 근본 원인: `file.itemId`는 LGU+ `ITEM_ID`이고, `history_no`는 폴링 API의 `HISTORY_NO`로 완전히 다른 값. `test:download-only`에서 `saveFile()` 시 `history_no`를 설정하지 않아 매칭 불가.
- 기존 DB 레코드가 있으면 `existing?.id ?? saveFile(...)` 패턴으로 재활용하여 중복 INSERT 방지.
- download-only에서는 `downloaded`와 `completed` 모두 스킵 (이전에는 `completed`만 체크).
- full-sync에서 `downloaded` 상태 파일은 다운로드를 건너뛰고 업로드만 실행하여 부분 재개 지원.
