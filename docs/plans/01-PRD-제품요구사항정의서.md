## 1. 프로젝트 개요

### 1.1 배경

유진레이저목형은 도무송 목형 제작 전문 업체로, 거래처로부터 도면 파일(DXF, DWG, AI, PDF 등)을 수신하여 목형을 제작, 납품하는 B2B 제조업체이다. 거래처는 LGU+ 웹하드(`only.webhard.co.kr`)의 `올리기전용/{업체명}/` 폴더에 도면을 업로드하고, 유진레이저는 이를 자체 웹하드(Cloudflare R2 기반, yjlaser_website에서 관리)로 옮겨 작업한다.

현재 v1 프로그램이 이 동기화를 수행하고 있으나, 아래와 같은 구조적 문제로 인해 v2 재구축이 필요하다.

### 1.2 v1의 문제점

| 구분 | 문제 | 상세 |
|------|------|------|
| **코드 구조** | 스파게티 코드 | `sync-cli.ts` 4,087줄 단일 파일, 70개 이상의 소스 파일이 복잡하게 얽힘 |
| **유지보수성** | 수정 불가 수준 | 양방향 동기화로 설계했다가 단방향으로 전환하면서 불필요한 코드가 대량 잔존, 77개의 작업 문서가 누적된 패치 히스토리가 복잡성을 증명 |
| **감지 신뢰성** | 간헐적 누락 | Upload History API 폴링 방식의 한계로 간헐적 파일 감지 누락 발생, 이를 보완하기 위해 이중 감지(폴링 + 스냅샷) 전략을 도입했으나 복잡도만 증가 |
| **UI 부재** | CLI 전용 | 터미널 기반 CLI로만 운영 가능, 비개발자인 실 사용자가 상태 확인이나 설정 변경 불가 |
| **배포/운영** | 불편함 | Docker/PM2 기반 운영으로 Windows 데스크톱 환경에서 관리 어려움 |
| **설정 관리** | 환경변수 의존 | `.env` 파일 직접 편집 필요, GUI 없음 |
| **오류 대응** | 수동 개입 | 세션 만료, 네트워크 장애 등에서 수동 CLI 명령 필요 |

### 1.3 v2의 목적

1. **신뢰성 향상**: 파일 감지 누락을 근본적으로 해소하여 100% 동기화 보장
2. **사용성 개선**: Electron + React 기반 GUI로 비개발자도 쉽게 운영 가능
3. **유지보수성 확보**: 계층형 아키텍처(Renderer → Main → Core)로 명확한 관심사 분리
4. **운영 편의성**: Windows 시작 시 자동 실행, 시스템 트레이 상주, 자동 복구

### 1.4 방향성

- **단방향 동기화 전용**: LGU+ 외부웹하드 → 자체웹하드 (한 방향만)
- **GUI 우선**: Electron + React 기반 데스크톱 앱 (대시보드, 파일 탐색기, 설정, 로그, 통계 차트)
- **계층형 아키텍처**: IPC 기반 Renderer ↔ Main ↔ Core 분리
- **인터페이스/어댑터 패턴**: 내부 웹하드 API가 추후 제공될 예정이므로 교체 가능한 설계
- **SDD 기반 개발**: 스펙(인터페이스/타입) 우선 정의 후 구현하는 Specification-Driven Development 방법론 적용 → 10-SDD-개발방법론.md 참조

### 1.5 범위

**포함 (In Scope)**:
- LGU+ 외부웹하드 파일 감지 및 다운로드
- 자체웹하드로의 자동 업로드
- Electron 기반 GUI (대시보드, 파일 탐색기, 설정, 로그 뷰어, 통계)
- Windows 자동 시작 및 시스템 트레이
- 인앱 알림 및 Windows 토스트 알림
- 폴더 선택적 동기화
- SQLite 기반 상태 관리

**제외 (Out of Scope)**:
- 역방향 동기화 (자체웹하드 → LGU+)
- 모바일/웹 버전
- 다중 사용자 지원
- LGU+ 웹하드 파일 삭제/이동/이름변경의 자체웹하드 반영 (v2 초기 범위 외, 향후 확장)

## 2. 용어 정의

### 2.1 비즈니스 도메인 용어

| 용어 | 정의 |
|------|------|
| **도무송** | Die-cutting의 한국어 표현. 종이, 골판지 등을 특정 형태로 절단하는 가공 방식 |
| **목형** | 도무송 작업에 사용되는 절단 틀. 합판에 레이저로 절단선(칼선)과 접는선(오시선)을 가공하여 제작 |
| **거래처** | 유진레이저목형에 목형 제작을 의뢰하는 B2B 고객사 (포장재 제조업체, 디자인 에이전시 등) |
| **도면** | 거래처가 제공하는 제품 설계 파일 (DXF, DWG, AI, PDF 등). 목형 제작의 원본 자료 |
| **DXF** | Drawing Exchange Format. AutoCAD 호환 벡터 도면 파일 형식. 목형 제작의 주요 파일 포맷 |
| **올리기전용** | LGU+ 웹하드에서 거래처가 유진레이저에게 파일을 전달하기 위한 업로드 폴더 (거래처 → 유진레이저 방향) |
| **내리기전용** | LGU+ 웹하드에서 유진레이저가 거래처에게 수정 도면을 전달하기 위한 다운로드 폴더 (유진레이저 → 거래처 방향) |
| **네스팅** | Nesting. 합판 위에 여러 목형을 최적 배치하여 재료 낭비를 최소화하는 공정 |
| **오시** | 접는선(Crease line). 종이를 접기 위해 눌러서 만드는 선 |
| **칼선** | 절단선(Cut line). 종이를 자르기 위한 선 |

### 2.2 기술 용어

| 용어 | 정의 |
|------|------|
| **외부웹하드** | LGU+ 웹하드 서비스 (`only.webhard.co.kr`). 거래처와 파일을 주고받는 외부 저장소 |
| **자체웹하드** | 유진레이저 자체 웹 서비스의 파일 저장소. Cloudflare R2 기반, yjlaser_website(Next.js + NestJS)에서 관리 |
| **동기화** | 외부웹하드의 파일을 자체웹하드로 자동 복제하는 과정. 본 프로그램에서는 단방향(외부 → 자체)만 수행 |
| **폴링(Polling)** | 서버에 주기적으로 요청을 보내 변경사항을 확인하는 방식 |
| **스냅샷** | 특정 시점의 웹하드 전체 폴더/파일 구조를 캡처한 데이터 |
| **체크포인트** | 마지막으로 처리 완료한 지점을 기록하여 중복 처리를 방지하고 재시작 시 이어받기를 가능하게 하는 메커니즘 |
| **historyNo** | LGU+ 웹하드 Upload History API에서 반환하는 고유 시퀀스 번호. 이벤트의 순서와 중복을 판별하는 핵심 식별자 |
| **operCode** | LGU+ 웹하드 API에서 파일 작업 유형을 나타내는 코드 (UP: 업로드, D: 삭제, MV: 이동, RN: 이름변경, FC: 폴더생성, FD: 폴더삭제, FMV: 폴더이동, FRN: 폴더이름변경, CP: 복사) |
| **세션 쿠키** | LGU+ 웹하드 로그인 후 발급되는 인증 토큰. 모든 API 호출에 Cookie 헤더로 포함해야 함 |
| **Presigned URL** | 일정 시간 동안만 유효한 서명된 URL. Cloudflare R2에 직접 업로드할 때 사용 |
| **IPC** | Inter-Process Communication. Electron에서 Renderer 프로세스(UI)와 Main 프로세스(비즈니스 로직) 간 통신 방식 |
| **시스템 트레이** | Windows 작업 표시줄 우측 알림 영역. 프로그램을 최소화하여 상주시킬 수 있는 공간 |
| **DLQ** | Dead Letter Queue. 재시도에 실패한 이벤트를 별도로 보관하는 큐 |
| **서킷 브레이커** | 연속 실패 시 일시적으로 요청을 차단하여 시스템 과부하를 방지하는 패턴 |

## 3. 사용자 정의

### 3.1 주 사용자 프로필

| 항목 | 내용 |
|------|------|
| **역할** | 유진레이저목형 내부 관리자 (1인) |
| **기술 수준** | 비개발자. Windows PC 일상 사용 수준. CLI 사용 불가 |
| **사용 환경** | Windows 10/11 데스크톱 PC, 사무실 내 유선 네트워크 |
| **사용 시간** | 업무 시간(09:00~18:00) 중심이나, PC가 켜져 있는 동안 24시간 자동 동기화 |
| **주요 관심사** | "파일이 빠짐없이 잘 동기화되고 있는가", "오류가 발생하면 어떻게 하는가" |
| **현재 불편사항** | CLI만 있어서 프로그램 상태 확인 불가, 오류 발생 시 개발자 호출 필요 |

### 3.2 사용 패턴

| 시나리오 | 설명 |
|----------|------|
| **일상 운영** | PC 부팅 시 자동 시작 → 시스템 트레이 상주 → 별도 조작 없이 자동 동기화 |
| **상태 확인** | 트레이 아이콘 클릭 → 대시보드에서 동기화 현황 확인 (오늘 동기화 건수, 최근 파일 목록 등) |
| **오류 대응** | Windows 토스트 알림 수신 → 프로그램 열어 로그 확인 → 재시도 버튼 클릭 또는 개발자에게 로그 전달 |
| **설정 변경** | 설정 화면에서 동기화 대상 폴더 선택/해제, 폴링 간격 조절 등 |
| **이력 조회** | 특정 파일이 언제 동기화되었는지 검색, 날짜별/업체별 동기화 통계 확인 |

### 3.3 사용자 요구사항 요약

1. **무간섭 운영**: 한 번 설정하면 PC를 끄기 전까지 자동으로 동작해야 함
2. **직관적 상태 표시**: 현재 동기화가 정상인지 한눈에 알 수 있어야 함
3. **즉시 알림**: 오류 발생 시 Windows 알림으로 즉시 인지 가능해야 함
4. **간편한 복구**: 오류 시 버튼 하나로 재시도/재연결이 가능해야 함
5. **이력 추적**: 특정 파일의 동기화 시점, 성공/실패 여부를 조회할 수 있어야 함

## 5. 기능별 상세 요구사항

### F-01. LGU+ 웹하드 로그인 및 세션 관리

#### 설명

LGU+ 웹하드(`only.webhard.co.kr`)에 관리자 계정으로 로그인하여 세션 쿠키를 획득하고, 세션이 만료되지 않도록 자동 갱신한다. LGU+ 웹하드는 공식 API가 없으므로 역공학으로 분석한 비공식 API를 사용한다.

#### 사용자 스토리

- 사용자로서, 프로그램에 LGU+ 웹하드 아이디와 비밀번호를 한 번만 입력하면 이후 자동으로 로그인이 유지되기를 원한다.
- 사용자로서, 세션이 만료되었을 때 자동으로 재로그인되어 동기화가 중단되지 않기를 원한다.
- 사용자로서, 로그인 실패 시 알림을 받고 비밀번호를 재입력할 수 있기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-01-AC01 | HTTP API 직접 로그인(`POST /login-process`)으로 세션 쿠키를 획득할 수 있다 | 로그인 후 폴더 목록 API 호출이 성공하는지 확인 |
| F-01-AC02 | Playwright 브라우저 로그인이 HTTP 로그인 실패 시 폴백으로 작동한다 | HTTP 로그인을 의도적으로 실패시킨 후 브라우저 로그인이 시도되는지 확인 |
| F-01-AC03 | 15분 간격으로 세션 유효성을 검증하고, 만료 시 자동 재로그인한다 | 15분 타이머 동작 및 세션 만료 후 재로그인 성공 확인 |
| F-01-AC04 | API 호출 중 세션 만료 감지(302 리다이렉트, 401, RESULT_CODE 9999) 시 즉시 재로그인 후 API를 재시도한다 | 세션 만료 시뮬레이션 후 자동 복구 확인 |
| F-01-AC05 | 자격증명을 안전하게 저장한다 (평문 저장 금지) | 저장소에 암호화된 자격증명이 기록되는지 확인 |
| F-01-AC06 | 로그인 실패가 3회 연속 발생하면 사용자에게 알림을 보내고 재입력을 요청한다 | 3회 실패 후 알림 및 설정 화면 이동 확인 |
| F-01-AC07 | CAPTCHA 감지 시 사용자에게 수동 로그인을 안내한다 | CAPTCHA 감지 후 알림 내용 확인 |

#### 엣지 케이스

- LGU+ 웹하드 서버 점검 시간(새벽 시간대) 동안 로그인 불가 상황 처리
- 비밀번호 변경 후 기존 세션 무효화 감지
- 동시에 여러 곳에서 같은 계정 로그인 시 세션 충돌
- LGU+ 웹하드 로그인 페이지 HTML 구조 변경 시 Playwright 셀렉터 실패

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 네트워크 타임아웃 | 5초, 10초, 30초 간격으로 3회 재시도 |
| 잘못된 자격증명 | 사용자에게 알림 + 설정 화면으로 유도 |
| CAPTCHA 출현 | 사용자에게 수동 로그인 안내 (브라우저 창 표시) |
| LGU+ 서버 다운 | 지수 백오프로 재시도 (최대 30분 간격), 사용자 알림 |
| 쿠키 파싱 실패 | 로그 기록 + Playwright 폴백 시도 |

### F-02. 실시간 파일 변경 감지

#### 설명

LGU+ 웹하드에 새로운 파일이 업로드되면 이를 실시간으로 감지한다. v1에서 사용한 Upload History API 폴링 방식을 기반으로 하되, 감지 신뢰성을 개선한다.

#### 사용자 스토리

- 사용자로서, 거래처가 외부웹하드에 파일을 올리면 1분 이내에 자동으로 동기화가 시작되기를 원한다.
- 사용자로서, 어떤 파일도 누락 없이 모두 동기화되기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-02-AC01 | Upload History API를 5초 간격으로 폴링하여 신규 파일 업로드(operCode: UP)를 감지한다 | 외부웹하드에 파일 업로드 후 5초 이내 이벤트 생성 확인 |
| F-02-AC02 | historyNo 기반 체크포인트로 중복 감지를 방지한다 | 동일 historyNo가 2회 이상 처리되지 않음을 확인 |
| F-02-AC03 | 감지된 이벤트를 SyncQueue에 추가하여 순서대로 처리한다 | 이벤트 발생 순서와 처리 순서가 일치하는지 확인 |
| F-02-AC04 | 이벤트 감지 후 500ms 이내 즉시 재폴링하여 연속 업로드를 놓치지 않는다 (최대 3회 연속) | 연속 3개 파일 업로드 시 모두 감지되는지 확인 |
| F-02-AC05 | 폴링 실패 시 자동 재시도하며, 실패가 지속되면 간격을 늘린다 (지수 백오프) | 네트워크 단절 시뮬레이션 후 재시도 패턴 확인 |
| F-02-AC06 | 보조 감지로 주기적 스냅샷 비교(10분 간격)를 수행하여 폴링이 놓친 변경을 보완한다 | 폴링 비활성 상태에서 10분 후 스냅샷 비교로 파일 감지 확인 |
| F-02-AC07 | 동기화 대상으로 선택된 폴더의 파일만 감지한다 (선택되지 않은 폴더는 무시) | 비대상 폴더의 파일 업로드 시 이벤트가 생성되지 않음을 확인 |
| F-02-AC08 | 프로그램 재시작 후에도 마지막 체크포인트부터 이어서 감지한다 | 프로그램 종료 → 외부 파일 업로드 → 재시작 → 해당 파일 감지 확인 |

#### 엣지 케이스

- 같은 파일명으로 다른 폴더에 동시 업로드
- 대량 업로드 (100개 이상 파일을 한 번에 업로드)
- 업로드 직후 삭제 (감지 전에 삭제됨)
- 날짜 경계(자정) 시점의 이력 조회
- 폴링 중 세션 만료
- LGU+ 서버 측 Upload History API 페이지네이션 변동

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| API 응답 타임아웃 | 5초 후 재시도, 3회 실패 시 30초 대기 후 재시도 |
| 세션 만료 (RESULT_CODE: 9999) | 자동 재로그인 후 폴링 재개 |
| 빈 응답 반환 | 정상으로 간주하고 다음 폴링 진행 |
| historyNo 역전/중복 | 메모리 캐시 + SQLite 이중 검증으로 필터링 |
| LGU+ API 형식 변경 | 파싱 오류 로그 + 사용자 알림 |

### F-03. 전체 동기화 (초기/수동)

#### 설명

프로그램 최초 실행 시 또는 사용자가 수동 요청 시, 선택된 폴더의 전체 파일을 스캔하여 아직 동기화되지 않은 파일을 모두 동기화한다. 델타 비교(파일명 + 크기)로 이미 동기화된 파일은 건너뛴다.

#### 사용자 스토리

- 사용자로서, 프로그램을 처음 설치하면 기존 파일을 모두 자동 동기화하기를 원한다.
- 사용자로서, "전체 동기화" 버튼을 눌러 누락된 파일을 수동으로 보충할 수 있기를 원한다.
- 사용자로서, 전체 동기화 진행률(스캔 중, 비교 중, 다운로드 중, 업로드 중)을 실시간으로 확인하기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-03-AC01 | 선택된 폴더의 전체 파일/폴더 구조를 재귀적으로 스캔한다 | 스캔 완료 후 파일 수가 외부웹하드 실제 파일 수와 일치하는지 확인 |
| F-03-AC02 | 로컬 상태(SQLite)와 비교하여 신규/변경 파일만 동기화 대상으로 선별한다 | 이미 동기화된 파일이 재다운로드되지 않음을 확인 |
| F-03-AC03 | 전체 동기화 진행률(단계별: 스캔 → 비교 → 다운로드 → 업로드)을 UI에 실시간 표시한다 | UI에 단계, 진행률 퍼센트, 현재 파일명이 표시되는지 확인 |
| F-03-AC04 | 전체 동기화 중에도 실시간 감지가 병행된다 (새 파일 놓치지 않음) | 전체 동기화 중 새 파일 업로드 시 해당 파일도 동기화되는지 확인 |
| F-03-AC05 | 전체 동기화를 중간에 취소할 수 있으며, 재시작 시 이어받기가 가능하다 | 50% 진행 후 취소 → 재시작 시 나머지 50%만 처리하는지 확인 |
| F-03-AC06 | 빈 폴더도 자체웹하드에 동일 구조로 생성한다 | 외부웹하드의 빈 폴더가 자체웹하드에도 존재하는지 확인 |
| F-03-AC07 | 전체 동기화 시작 시 startHistoryNo를 기록하여 갭 없는 동기화를 보장한다 | 전체 동기화 시작~실시간 감지 시작 사이에 올라온 파일이 누락 없이 처리되는지 확인 |

#### 엣지 케이스

- 매우 큰 폴더 (10,000개 이상 파일)
- 전체 동기화 중 네트워크 단절
- 전체 동기화 중 LGU+ 웹하드에서 파일 삭제
- 동일 파일명이 다른 폴더에 존재
- 특수문자가 포함된 파일명/폴더명
- 0바이트 파일

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 스캔 중 API 실패 | 실패한 폴더 건너뛰기 + 나중에 재시도 |
| 디스크 공간 부족 | 사전 검사 + 용량 부족 시 알림 후 중단 |
| 다운로드 실패 (개별 파일) | 실패 파일 DLQ에 추가 + 나머지 계속 진행 |
| 업로드 실패 (개별 파일) | 3회 재시도 후 실패 파일 기록 + 나머지 계속 진행 |

### F-04. 파일 다운로드

#### 설명

LGU+ 웹하드에서 파일을 다운로드한다. 2단계 프로세스를 따른다: (1) 다운로드 URL 정보 획득, (2) 실제 파일 다운로드.

#### 사용자 스토리

- 사용자로서, 외부웹하드의 파일이 빠르게 다운로드되기를 원한다.
- 사용자로서, 대용량 파일도 안정적으로 다운로드되기를 원한다.
- 사용자로서, 다운로드 중인 파일의 진행 상황을 알 수 있기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-04-AC01 | `GET /downloads/{fileId}/server?fileStatus=1`로 다운로드 URL 정보를 획득한다 | API 응답에 session, nonce, url 등 필수 필드가 포함되는지 확인 |
| F-04-AC02 | 획득한 정보로 다운로드 URL을 구성하여 파일을 다운로드한다 | 다운로드된 파일이 원본과 동일한지 크기 비교 확인 |
| F-04-AC03 | 파일 크기에 따른 동적 타임아웃을 적용한다 (10MB 미만: 2분, 100MB 미만: 5분, 1GB 미만: 15분, 1GB 이상: 30분) | 대용량 파일 다운로드 시 타임아웃이 적절히 설정되는지 확인 |
| F-04-AC04 | 배치 다운로드 시 최대 동시 다운로드 수를 제한한다 (기본값: 5) | 6개 이상 파일 동시 요청 시 5개만 병렬 진행하는지 확인 |
| F-04-AC05 | 다운로드 실패 시 최대 3회 재시도한다 (1초 → 3초 → 10초 간격) | 네트워크 오류 시뮬레이션 후 재시도 횟수 및 간격 확인 |
| F-04-AC06 | 서킷 브레이커로 연속 실패 시 일시적으로 다운로드를 중단한다 (실패율 40% 초과 시 10초 차단) | 연속 실패 시 차단 → 복구 패턴 확인 |
| F-04-AC07 | 다운로드된 파일명에서 특수문자를 안전하게 처리한다 (경로 구분자, 제어문자 등 제거) | 특수문자 파일명 다운로드 후 파일 시스템에 정상 저장 확인 |
| F-04-AC08 | 파일별 다운로드 진행률(바이트 기반)을 UI에 전달한다 | 대용량 파일 다운로드 시 진행률 업데이트가 UI에 반영되는지 확인 |

#### 엣지 케이스

- 다운로드 중 LGU+ 서버에서 파일 삭제
- 다운로드 URL 만료 (시간 초과)
- 0바이트 파일
- 파일명이 매우 긴 경우 (255자 초과)
- 한글/일본어/중국어 파일명
- 동시 다운로드 중 하나만 실패

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 다운로드 URL 획득 실패 (404) | 파일 삭제로 판단, 건너뛰기 + 로그 기록 |
| 다운로드 URL 획득 실패 (500) | 3회 재시도 후 DLQ에 추가 |
| 다운로드 중 네트워크 단절 | 부분 파일 삭제 + 재시도 |
| 다운로드 파일 크기 불일치 | 파일 삭제 + 재다운로드 |
| 로컬 디스크 공간 부족 | 즉시 중단 + 사용자 알림 |

### F-05. 자체웹하드 업로드

#### 설명

다운로드된 파일을 자체웹하드(yjlaser_website API)에 업로드한다. 자체웹하드 API는 추후 변경될 수 있으므로 인터페이스/어댑터 패턴으로 교체 가능하게 설계한다.

#### 사용자 스토리

- 사용자로서, 다운로드된 파일이 자동으로 자체웹하드에 업로드되어 웹사이트에서 확인할 수 있기를 원한다.
- 사용자로서, 외부웹하드의 폴더 구조가 자체웹하드에 그대로 보존되기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-05-AC01 | 다운로드 완료된 파일을 자체웹하드 API로 업로드한다 | 업로드 후 자체웹하드 웹 UI에서 파일 확인 |
| F-05-AC02 | 업로드 전 MD5 체크섬을 계산하고, 서버 응답의 체크섬과 비교하여 무결성을 검증한다 | 체크섬 불일치 시 재업로드가 시도되는지 확인 |
| F-05-AC03 | 외부웹하드의 폴더 경로를 자체웹하드에 동일하게 재현한다 (폴더 자동 생성) | `/올리기전용/업체A/서브폴더/` 경로가 그대로 생성되는지 확인 |
| F-05-AC04 | 100MB 이상 파일은 멀티파트 업로드를 사용한다 | 150MB 파일 업로드 시 멀티파트 API가 호출되는지 확인 |
| F-05-AC05 | 중복 파일(체크섬 동일)은 업로드를 건너뛴다 | 동일 파일 재업로드 시도 시 스킵되는지 확인 |
| F-05-AC06 | 업로드 실패 시 3회 재시도한다 (지수 백오프: 1초 → 2초 → 4초) | 서버 오류 시뮬레이션 후 재시도 패턴 확인 |
| F-05-AC07 | 내부웹하드 API 인터페이스가 정의되어 있어, 다른 구현체로 교체 가능하다 | 인터페이스 정의서 검토 + Mock 구현체 테스트 |
| F-05-AC08 | 폴더 404 오류 시 폴더 재생성 후 업로드를 재시도한다 (타이밍 이슈 대응) | 폴더 생성 직후 업로드 시도 시 정상 처리 확인 |

#### 엣지 케이스

- 자체웹하드 서버 다운/점검 시
- 업로드 중 자체웹하드 서버 재시작
- Presigned URL 만료
- 동일 파일명이 이미 자체웹하드에 존재 (체크섬은 다름)
- 매우 깊은 폴더 경로 (10단계 이상)

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 자체웹하드 서버 연결 불가 | 30초 간격으로 재시도, 5분 후에도 불가하면 사용자 알림 |
| 업로드 타임아웃 | 재시도 (멀티파트의 경우 실패 파트만 재업로드) |
| 인증 실패 (API Key 오류) | 사용자에게 API Key 확인 알림 |
| 폴더 생성 실패 | 상위 폴더부터 순차 생성 재시도 |
| 체크섬 불일치 | 파일 재다운로드 후 재업로드 |

### F-06. 대시보드 UI

#### 설명

프로그램의 메인 화면으로, 동기화 상태를 한눈에 파악할 수 있는 대시보드를 제공한다.

#### 사용자 스토리

- 사용자로서, 프로그램을 열면 현재 동기화 상태(정상/오류/일시중지)를 즉시 파악하기를 원한다.
- 사용자로서, 오늘 몇 개의 파일이 동기화되었는지 한눈에 보기를 원한다.
- 사용자로서, 최근 동기화된 파일 목록을 확인하기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-06-AC01 | 동기화 상태 표시기를 표시한다: 정상(초록), 경고(노랑), 오류(빨강), 일시중지(회색) | 각 상태에서 올바른 색상과 아이콘이 표시되는지 확인 |
| F-06-AC02 | 오늘의 동기화 요약을 표시한다: 총 파일 수, 성공 수, 실패 수, 전송 용량 | 실제 동기화 수치와 UI 표시 수치 일치 확인 |
| F-06-AC03 | 최근 동기화 파일 목록(최대 50건)을 시간 역순으로 표시한다 | 최신 파일이 상단에 표시되는지 확인 |
| F-06-AC04 | 현재 진행 중인 동기화 작업의 진행률을 실시간 표시한다 | 파일 다운로드/업로드 중 진행률 바가 업데이트되는지 확인 |
| F-06-AC05 | 외부웹하드 연결 상태(세션 유효 여부)를 표시한다 | 세션 만료 시 UI에 "연결 끊김" 상태가 표시되는지 확인 |
| F-06-AC06 | 자체웹하드 연결 상태를 표시한다 | API 서버 다운 시 UI에 "연결 끊김" 상태가 표시되는지 확인 |
| F-06-AC07 | "전체 동기화" 버튼을 제공하여 수동 전체 동기화를 시작할 수 있다 | 버튼 클릭 후 전체 동기화가 시작되는지 확인 |
| F-06-AC08 | "일시중지/재개" 버튼을 제공한다 | 일시중지 후 폴링이 멈추고, 재개 후 다시 시작되는지 확인 |

#### 엣지 케이스

- 동기화 이력이 없는 초기 상태
- 매우 많은 동기화 이력 (10,000건 이상)
- UI 렌더링 중 IPC 통신 실패
- 대시보드 열려 있는 상태에서 대량 동기화 진행 (UI 성능)

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| IPC 통신 실패 | "상태 로딩 실패" 메시지 + 새로고침 버튼 |
| SQLite 조회 실패 | 빈 대시보드 표시 + 오류 메시지 |
| 데이터 불일치 | 자동 새로고침 (5초 간격) |

### F-07. 시스템 트레이 상주

#### 설명

프로그램 창을 닫아도 시스템 트레이에 상주하며 백그라운드에서 동기화를 계속한다.

#### 사용자 스토리

- 사용자로서, 프로그램 창을 닫아도 동기화가 계속 진행되기를 원한다.
- 사용자로서, 트레이 아이콘을 통해 동기화 상태를 빠르게 확인하기를 원한다.
- 사용자로서, 트레이 아이콘 우클릭으로 빠른 조작(열기/일시중지/종료)을 하기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-07-AC01 | 창 닫기(X 버튼) 시 트레이로 최소화된다 (프로세스 종료 아님) | X 클릭 후 트레이 아이콘 존재 + 동기화 계속 진행 확인 |
| F-07-AC02 | 트레이 아이콘이 동기화 상태를 색상으로 표시한다 (정상: 초록, 오류: 빨강, 일시중지: 회색) | 각 상태에서 아이콘 색상 변경 확인 |
| F-07-AC03 | 트레이 아이콘 더블클릭으로 메인 창을 연다 | 더블클릭 후 창 복원 확인 |
| F-07-AC04 | 트레이 아이콘 우클릭 메뉴: 열기, 일시중지/재개, 전체 동기화, 종료 | 각 메뉴 항목 동작 확인 |
| F-07-AC05 | 트레이 아이콘 툴팁에 간략한 상태 정보를 표시한다 (예: "동기화 중 - 오늘 15건 완료") | 마우스 호버 시 툴팁 내용 확인 |
| F-07-AC06 | "종료" 메뉴 선택 시 확인 대화상자 표시 후 프로세스를 완전 종료한다 | 종료 확인 후 프로세스 종료 확인 |

#### 엣지 케이스

- Windows 탐색기 재시작 시 트레이 아이콘 복구
- 다중 모니터 환경에서 트레이 아이콘 위치
- 트레이 아이콘 영역 오버플로우 (많은 프로그램이 트레이에 있을 때)

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 트레이 아이콘 생성 실패 | 일반 창 모드로 폴백 |
| Windows 탐색기 크래시 | 탐색기 재시작 감지 시 트레이 아이콘 재등록 |

### F-08. Windows 자동 시작

#### 설명

Windows 부팅 시 프로그램이 자동으로 시작되어 트레이에 상주한다.

#### 사용자 스토리

- 사용자로서, PC를 켜면 별도 조작 없이 자동으로 동기화가 시작되기를 원한다.
- 사용자로서, 자동 시작 기능을 설정 화면에서 켜고 끌 수 있기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-08-AC01 | 설정에서 "Windows 시작 시 자동 실행"을 활성화하면 레지스트리(또는 시작 폴더)에 등록된다 | 레지스트리 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run` 확인 |
| F-08-AC02 | 자동 시작 시 메인 창 없이 트레이 모드로 시작한다 | PC 재부팅 후 트레이 아이콘만 표시되는지 확인 |
| F-08-AC03 | 자동 시작 후 자동 로그인 → 실시간 감지가 시작된다 | 부팅 후 1분 이내에 동기화 폴링이 시작되는지 확인 |
| F-08-AC04 | 설정에서 비활성화하면 자동 시작 등록이 해제된다 | 비활성화 후 레지스트리 항목 제거 확인 |

#### 엣지 케이스

- Windows 빠른 시작(Fast Startup) 활성화 상태
- 여러 사용자 계정이 있는 PC
- 프로그램 업데이트 후 경로 변경

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 레지스트리 쓰기 권한 없음 | 시작 폴더 바로가기 방식으로 폴백 |
| 실행 파일 경로 변경됨 | 다음 실행 시 자동 시작 경로 갱신 |

### F-10. 동기화 대상 폴더 선택

#### 설명

외부웹하드의 폴더 목록을 조회하여 사용자가 동기화할 폴더를 선택/해제할 수 있게 한다.

#### 사용자 스토리

- 사용자로서, 외부웹하드의 폴더 중 필요한 폴더만 선택하여 동기화하기를 원한다.
- 사용자로서, 새 거래처 폴더가 생기면 자동으로 감지되어 선택 목록에 추가되기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-10-AC01 | 외부웹하드의 `올리기전용/` 하위 폴더 목록을 트리 형태로 표시한다 | UI에 실제 폴더 구조가 표시되는지 확인 |
| F-10-AC02 | 각 폴더에 체크박스를 제공하여 동기화 대상을 선택/해제할 수 있다 | 체크 후 해당 폴더만 동기화 대상이 되는지 확인 |
| F-10-AC03 | 상위 폴더 선택 시 하위 폴더도 자동 선택된다 | 상위 폴더 체크 후 하위 항목도 체크되는지 확인 |
| F-10-AC04 | 선택 상태는 SQLite에 영속적으로 저장된다 | 프로그램 재시작 후 선택 상태 유지 확인 |
| F-10-AC05 | "새로고침" 버튼으로 폴더 목록을 최신으로 갱신할 수 있다 | 새 폴더 생성 후 새로고침 시 목록에 나타나는지 확인 |
| F-10-AC06 | 신규 폴더가 감지되면 알림을 보내고 기본값은 '선택됨'으로 설정한다 | 새 폴더 생성 후 자동 선택 + 알림 확인 |
| F-10-AC07 | 각 폴더 옆에 마지막 동기화 시간과 파일 수를 표시한다 | 동기화 후 해당 정보가 갱신되는지 확인 |

#### 엣지 케이스

- 외부웹하드에 폴더가 하나도 없는 경우
- 폴더명에 특수문자 포함
- 매우 깊은 폴더 계층 (5단계 이상)
- 폴더 목록 조회 중 세션 만료
- 동기화 중에 폴더 선택 변경

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 폴더 목록 조회 실패 | 마지막으로 캐시된 목록 표시 + 오류 알림 |
| 선택한 폴더가 외부웹하드에서 삭제됨 | "삭제된 폴더" 표시 + 동기화 목록에서 자동 제거 |

|----|------|-----------|

| F-11-AC02 | 로그 레벨 필터: DEBUG, INFO, WARN, ERROR | 각 레벨 필터 적용 후 해당 레벨만 표시되는지 확인 |
| F-11-AC03 | 텍스트 검색으로 로그를 필터링할 수 있다 | 파일명 검색 시 관련 로그만 표시되는지 확인 |
| F-11-AC04 | 날짜/시간 범위 필터를 제공한다 | 특정 날짜 범위 선택 후 해당 기간 로그만 표시 확인 |
| F-11-AC05 | 로그를 텍스트 파일로 내보내기할 수 있다 | 내보내기 후 파일 내용이 화면 표시와 일치하는지 확인 |
| F-11-AC06 | 오래된 로그를 자동 삭제한다 (기본 30일 보관) | 30일 이상 된 로그가 자동 정리되는지 확인 |
| F-11-AC07 | 로그 항목 클릭 시 상세 정보(스택 트레이스, 컨텍스트 등)를 펼쳐 표시한다 | 오류 로그 클릭 후 상세 정보 표시 확인 |

#### 엣지 케이스

- 로그가 초당 100건 이상 발생하는 대량 동기화 상황 (UI 성능)
- 매우 긴 로그 메시지
- 로그 DB 파일 크기가 매우 큰 경우 (1GB 이상)

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|

| 내보내기 경로에 쓰기 권한 없음 | 다른 경로 선택 유도 |

### F-15. 체크포인트 기반 이어받기

#### 설명

프로그램 종료/크래시 후 재시작 시 마지막 처리 지점부터 이어서 동기화한다.

#### 사용자 스토리

- 사용자로서, 프로그램이 비정상 종료되어도 이미 처리된 파일을 다시 처리하지 않기를 원한다.
- 사용자로서, 재시작 후 누락 없이 나머지 파일부터 이어서 동기화하기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-15-AC01 | 처리 완료된 이벤트의 historyNo를 SQLite에 원자적으로 기록한다 | DB에 체크포인트가 기록되는지 확인 |
| F-15-AC02 | 프로그램 재시작 시 마지막 체크포인트부터 폴링을 재개한다 | 재시작 후 마지막 처리 이후 이벤트만 처리되는지 확인 |
| F-15-AC03 | 전체 동기화 중단 시 진행 상황(처리된 폴더/파일 목록)을 저장하여 이어받기한다 | 전체 동기화 50% 중단 → 재시작 → 나머지 50% 처리 확인 |
| F-15-AC04 | 3단계 중복 검사: historyNo 비교 → 메모리 캐시 → SQLite DB 조회 | 동일 이벤트가 중복 처리되지 않음을 확인 |
| F-15-AC05 | 비정상 종료 시 처리 중이던 파일(부분 다운로드)을 정리한다 | 재시작 후 불완전 파일이 삭제 + 재다운로드되는지 확인 |

#### 엣지 케이스

- SQLite DB 파일 손상
- 체크포인트 기록 직전 프로세스 Kill
- 재시작 사이에 LGU+ 웹하드에서 대량 변경 발생

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| SQLite DB 손상 | 새 DB 생성 + 전체 동기화 자동 시작 + 사용자 알림 |
| 체크포인트 불일치 | 보수적으로 이전 체크포인트부터 재처리 (중복 허용) |

### F-16. 자동 오류 복구

#### 설명

네트워크 단절, 세션 만료 등 일시적 오류를 자동으로 복구한다.

#### 사용자 스토리

- 사용자로서, 일시적 네트워크 오류로 인해 동기화가 영구 중단되지 않기를 원한다.
- 사용자로서, 세션 만료 시 자동 재로그인되기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-16-AC01 | 네트워크 오류 시 지수 백오프로 자동 재시도한다 (1초 → 3초 → 10초 → 30초 → 최대 5분) | 네트워크 단절 시뮬레이션 후 재시도 패턴 확인 |
| F-16-AC02 | 세션 만료 감지 시 자동 재로그인 후 작업을 재개한다 | 세션 만료 후 폴링이 자동 재개되는지 확인 |
| F-16-AC03 | 실패한 개별 파일을 DLQ에 보관하고, 주기적으로 재시도한다 (1시간 간격) | DLQ에 쌓인 파일이 1시간 후 재시도되는지 확인 |
| F-16-AC04 | 5분 이상 stuck 상태인 이벤트를 자동 감지하여 재처리한다 | stuck 이벤트 자동 복구 확인 |
| F-16-AC05 | 자동 복구 시도 횟수와 결과를 로그에 기록한다 | 복구 시도 로그 확인 |

#### 엣지 케이스

- 장시간(1시간 이상) 네트워크 단절
- 자체웹하드 서버 장시간 다운
- 동시에 여러 종류의 오류 발생

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 자동 복구 최대 횟수 초과 | 사용자에게 수동 개입 필요 알림 |
| 복구 시도 중 다른 오류 발생 | 기존 오류 보존 + 새 오류 병행 처리 |

| 원본 파일이 이미 삭제됨 | "파일 없음" 상태로 변경 + 재시도 비활성화 |

| F-18-AC02 | 체크포인트 테이블: historyNo, 처리 시간, 상태 저장 | 체크포인트 CRUD 동작 확인 |

| F-18-AC04 | 설정 테이블: 모든 사용자 설정 영속 저장 | 설정 변경 후 프로그램 재시작 시 유지 확인 |
| F-18-AC05 | 통계 테이블: 일별 동기화 건수, 용량, 성공률 집계 | 통계 조회 결과가 정확한지 확인 |
| F-18-AC06 | DLQ 테이블: 실패 이벤트 보관, 재시도 이력 | DLQ CRUD 및 재시도 이력 확인 |
| F-18-AC07 | 자동 마이그레이션: 스키마 버전 관리 및 업그레이드 | 스키마 변경 후 자동 마이그레이션 동작 확인 |
| F-18-AC08 | 자동 백업: 일 1회 DB 파일 백업 (최근 7개 보관) | 백업 파일 생성 및 보관 주기 확인 |

#### 엣지 케이스

- DB 파일 크기가 1GB를 초과하는 경우
- 디스크 공간 부족으로 쓰기 실패
- 동시에 여러 스레드에서 쓰기 시도

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| DB 파일 손상 | 최근 백업에서 복원 + 사용자 알림 |
| 디스크 공간 부족 | 사용자 알림 + 오래된 이력 자동 정리 |
| 마이그레이션 실패 | 이전 버전 DB 백업 보존 + 오류 보고 |

### F-19. 자동 업데이트

#### 설명

프로그램의 새 버전을 자동으로 감지하고 업데이트한다.

#### 사용자 스토리

- 사용자로서, 새 버전이 나오면 자동으로 알림을 받고 업데이트하기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-19-AC01 | GitHub Releases 또는 지정 서버에서 새 버전을 주기적으로 확인한다 | 새 버전 발행 후 프로그램에서 감지하는지 확인 |
| F-19-AC02 | 업데이트 가능 시 사용자에게 알림을 보낸다 (강제 아님) | 알림 표시 확인 |
| F-19-AC03 | "지금 업데이트" 버튼으로 다운로드 → 설치 → 재시작을 자동 수행한다 | 업데이트 완료 후 새 버전 확인 |
| F-19-AC04 | 업데이트 중 동기화를 일시중지하고, 업데이트 후 자동 재개한다 | 업데이트 전후 동기화 상태 확인 |

#### 엣지 케이스

- 업데이트 다운로드 중 네트워크 단절
- 업데이트 파일 무결성 검증 실패
- 업데이트 후 DB 스키마 마이그레이션 필요

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 업데이트 서버 접속 불가 | 무시하고 현재 버전으로 계속 운영 |
| 설치 실패 | 이전 버전으로 롤백 + 오류 보고 |

### F-20. 내부웹하드 API 어댑터

#### 설명

자체웹하드 API와의 통신을 인터페이스로 추상화하여, API가 변경되거나 다른 백엔드로 교체되어도 핵심 로직에 영향이 없도록 한다.

#### 사용자 스토리

- 개발자로서, 자체웹하드 API가 변경되어도 어댑터 구현체만 교체하면 되기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-20-AC01 | `IInternalWebhardApi` 인터페이스가 정의되어 있다 (폴더 조회, 폴더 생성, 파일 업로드, 파일 존재 확인, 파일 삭제) | 인터페이스 정의 코드 리뷰 |
| F-20-AC02 | 현재 yjlaser_website API 용 구현체가 존재한다 (`YjlaserWebhardAdapter`) | 구현체 코드 리뷰 + 통합 테스트 |
| F-20-AC03 | Mock 구현체가 존재하여 단위 테스트에 사용할 수 있다 | Mock을 사용한 테스트 실행 확인 |
| F-20-AC04 | 설정에서 API 어댑터를 선택할 수 있다 (현재는 yjlaser만 제공) | 설정 UI에서 어댑터 선택 옵션 확인 |
| F-20-AC05 | 어댑터 교체 시 동기화 핵심 로직(`SyncEngine`)에 코드 변경이 필요 없다 | 다른 어댑터로 교체 후 테스트 통과 확인 |

#### 엣지 케이스

- API 응답 형식이 예상과 다른 경우
- 어댑터 초기화 실패

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| API 응답 파싱 실패 | 상세 오류 로그 + 원본 응답 보존 |
| 인터페이스 미구현 메서드 호출 | TypeScript 컴파일 타임에 차단 |

## 7. 제약사항

### 7.1 기술적 제약사항

| 제약 | 설명 | 영향 |
|------|------|------|
| **LGU+ 비공식 API** | LGU+ 웹하드는 공식 API를 제공하지 않음. 역공학으로 분석한 HTTP API를 사용함 | API 형식이 예고 없이 변경될 수 있으며, 이 경우 긴급 패치가 필요함 |
| **쿠키 기반 인증** | LGU+ 웹하드는 세션 쿠키 기반 인증만 지원. OAuth 등 표준 인증 미지원 | 세션 만료 관리, 쿠키 파싱 등 추가 로직 필요 |
| **폴링 방식 감지** | LGU+ 웹하드는 WebSocket/Webhook을 지원하지 않음. Upload History API 폴링만 가능 | 실시간성에 한계 (최소 5초 지연), 서버 부하 고려 필요 |
| **단일 계정** | LGU+ 웹하드 관리자 계정 1개만 사용 | 다중 세션/병렬 접근 제한 |
| **Windows 전용** | 사용 환경이 Windows 데스크톱으로 한정됨 | Electron으로 크로스 플랫폼 가능하나 Windows만 지원/테스트 |

| **내부웹하드 API 미확정** | 자체웹하드 API 사양이 확정되지 않았으며 추후 변경 가능 | 인터페이스/어댑터 패턴으로 교체 가능하게 설계 필수 |

## 9. 향후 확장 계획

> 아래 항목들은 v2 초기 출시 범위에 포함되지 않으나, 아키텍처 설계 시 확장 가능성을 고려해야 한다.

### 9.1 양방향 동기화

- **내용**: 자체웹하드 → LGU+ 외부웹하드 방향 동기화 추가 (수정 도면 전달 자동화)
- **고려사항**: LGU+ 업로드 API 역공학 필요, 충돌 해결 전략(LWW, 수동 등) 설계 필요
- **아키텍처 영향**: `SyncDirection` enum, `ConflictResolver` 인터페이스를 미리 확장 가능하게 설계

### 9.2 파일 변경 이벤트 전파 (삭제/이동/이름변경)

- **내용**: LGU+ 웹하드에서 파일 삭제, 이동, 이름변경 시 자체웹하드에도 반영
- **고려사항**: operCode(D, MV, FMV, RN, FRN) 처리 로직 추가, 삭제의 경우 휴지통 개념 도입 검토
- **아키텍처 영향**: `SyncEventType` 확장, `EventHandler` 전략 패턴 추가

### 9.3 웹사이트 연동 강화

- **내용**: 동기화된 파일을 yjlaser_website의 주문/접수 시스템과 자동 연결
- **고려사항**: 파일명에서 업체명 추출 → 거래처 자동 매칭 → 주문 건 자동 생성
- **아키텍처 영향**: 웹사이트 API 연동 모듈 추가, Webhook 수신 기능

### 9.4 다중 외부웹하드 지원

- **내용**: LGU+ 외에 다른 웹하드 서비스 지원 (ex. 센드애니웨어, 네이버 MYBOX 등)
- **아키텍처 영향**: `IExternalWebhardApi` 인터페이스 정의, 외부웹하드도 어댑터 패턴 적용

### 9.5 모니터링 대시보드 (웹)

- **내용**: 외부에서 동기화 상태를 모니터링할 수 있는 웹 대시보드
- **고려사항**: yjlaser_website에 관리자 전용 모니터링 페이지 추가
- **아키텍처 영향**: Main 프로세스에 REST API 서버 추가 (또는 yjlaser_website에 상태 리포팅)

### 9.6 모바일 알림

- **내용**: 심각한 오류 발생 시 모바일(카카오톡, SMS 등)로 알림
- **고려사항**: Popbill API 또는 Slack Webhook 활용
- **아키텍처 영향**: 알림 인터페이스 확장, 알림 채널 어댑터 추가

### 주요 엔드포인트

| 용도 | 엔드포인트 | 메서드 | 요청 형식 |
|------|-----------|--------|----------|

### API 응답 공통 형식
```json
{
  "RESULT_CODE": "0000",
  "RESULT_MSG": "OK",
  "TOTAL": 150,
  "COUNT": 100,
  "ITEMS": [...]
}
```

- `RESULT_CODE`: `"0000"` = 성공, `"9999"` = 세션 만료
- 세션 만료 감지: HTTP 302 + Location에 'login' 포함, HTTP 401, HTML 응답(JSON 대신), RESULT_CODE 9999
