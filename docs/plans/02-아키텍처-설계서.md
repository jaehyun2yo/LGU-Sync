## 1. 아키텍처 개요

### 1.1 설계 원칙

| 원칙 | 적용 |
|------|------|
| **SRP** | 모듈당 하나의 책임. SyncEngine=오케스트레이션, LGUplusClient=API 통신 |
| **OCP** | WebhardUploader 인터페이스로 백엔드 교체 시 기존 코드 수정 불필요 |
| **DIP** | Core는 구체 구현이 아닌 인터페이스에 의존. 외부 의존성은 추상화 |
| **ISP** | 인터페이스를 역할 단위로 분리. 사용하지 않는 메서드 의존 금지 |
| **Clean Architecture** | 의존성은 바깥→안쪽 단방향. Core는 Main/Renderer에 무지 |
| **SDD** | Specification-Driven Development — 인터페이스/타입 스펙을 먼저 정의하고, 구현은 스펙을 따른다. 모든 Core 모듈은 인터페이스 기반 계약을 가진다 |

### 1.2 전체 아키텍처

```
+---------------------------------------------------------------------+
|  Renderer (React + Tailwind + shadcn/ui + Zustand)                  |
|  Pages: Dashboard | FileExplorer | Settings | LogViewer | Stats     |
+---------------------------------------------------------------------+
        |  preload.ts (contextBridge)                         ^
        v  ipcRenderer.invoke                                 | webContents.send
========================= IPC Bridge (type-safe) ========================
        |  ipcMain.handle                                     ^
        v                                                     |
+---------------------------------------------------------------------+
|  Main Process (Electron)                                            |
|  WindowManager | TrayManager | AutoStart | IPC Router | Lifecycle  |
+---------------------------------------------------------------------+
        |  직접 함수 호출 (동일 프로세스)
        v
+---------------------------------------------------------------------+
|  Core Layer (순수 TypeScript, Electron 의존성 없음)                 |
|                                                                     |
|  SyncEngine ─┬─ FileDetector     (폴링+스냅샷 감지)                |
|              ├─ LGUplusClient    (LGU+ 비공식 API)                  |
|              ├─ WebhardUploader  (인터페이스, 구현체 교체 가능)      |
|              ├─ RetryManager     (재시도+서킷브레이커+DLQ)           |
|              ├─ StateManager     (SQLite WAL)                       |
|              └─ EventBus         (타입 안전 이벤트)                  |
|                                                                     |
|  Logger | ConfigManager | NotificationService                      |
+---------------------------------------------------------------------+
```

### 1.3 레이어 의존성 규칙

| 규칙 | 방향 |
|------|------|
| Renderer -> Main | IPC를 통해서만 통신 |
| Main -> Core | Core의 public API만 호출 |
| Core -> 없음 | Electron/React 의존 금지. Node.js 표준 모듈은 인터페이스 래핑 후 사용 |
| **금지** | Core -> Main, Renderer -> Core 직접 import |

## 2. 레이어 상세

### 2.1 Renderer Layer

| 구성 | 내용 |
|------|------|
| **Pages** | Dashboard(현황), FileExplorer(파일탐색), Settings(설정), LogViewer(로그), Statistics(차트) |
| **Components** | Layout(사이드바/헤더), SyncStatus(상태 뱃지), FileList(가상 스크롤), FolderTree(체크박스), NotificationCenter, ProgressBar |
| **Stores (Zustand)** | `syncStore`(동기화 상태/진행률), `settingsStore`(설정 캐시), `logStore`(필터/검색), `notificationStore`(알림 목록), `uiStore`(사이드바/페이지) |
| **Hooks** | `useIpc`(invoke 래퍼+로딩/에러), `useIpcEvent`(이벤트 구독), `useSyncStatus`(상태 구독) |

**설계 결정**: Zustand은 보일러플레이트 최소화 + IPC 패턴과 자연스러운 결합. shadcn/ui는 복사-붙여넣기 방식으로 의존성 최소화. 대량 목록(1,000건+)은 `@tanstack/react-virtual`로 가상 스크롤.

### 2.2 Main Process Layer

| 구성 | 역할 |
|------|------|
| **App Lifecycle** | `app.whenReady`, `before-quit`, `window-all-closed` 처리 |
| **Window Manager** | BrowserWindow 생성/관리, 위치/크기 저장/복원 |
| **Tray Manager** | 시스템 트레이 아이콘, 컨텍스트 메뉴(열기/일시중지/전체동기화/종료), 상태별 색상 변경 |
| **AutoStart** | `app.setLoginItemSettings()` 우선, 실패 시 레지스트리 폴백 |
| **IPC Router** | 채널별 핸들러 등록, Core 메서드 위임, EventBus→IPC 이벤트 브릿지 |
| **Service Layer** | Core 모듈 DI 팩토리 호출, 생명주기 바인딩 |

**생명주기**: `app.whenReady()` -> initCore -> registerIpcHandlers -> createTray -> createWindow -> startSync. `before-quit` -> stopSync(5초 타임아웃) -> saveState -> closeDb.

### 2.3 Core Layer

Electron, React 의존성 없는 순수 TypeScript. `vitest`로 단독 테스트 가능. Electron 전용 API(`Notification`, `dialog` 등)는 인터페이스로 추상화하고 Main에서 구현체를 주입.

## 3. 모듈 구조

### 3.1 SyncEngine

**책임**: 동기화 오케스트레이션. 감지→다운로드→업로드 파이프라인 관리.
**상태**: `idle` | `syncing` | `paused` | `error`
**API**: `start()`, `stop()`, `pause()`, `resume()`, `fullSync()`, `syncFile(fileId)`
**의존**: FileDetector, LGUplusClient, WebhardUploader, StateManager, RetryManager, EventBus, Logger

```
FileDetector.onNewFile -> SyncQueue(우선순위큐, 중복제거)
  -> Download(LGUplusClient, 동시5) -> Upload(WebhardUploader, 동시3)
  -> StateManager.markCompleted -> EventBus.emit('sync:completed')
```

### 3.2 LGUplusClient

**책임**: LGU+ 웹하드 비공식 API HTTP 통신 전체.
**API**: `login()`, `validateSession()`, `refreshSession()`, `getUploadHistory()`, `listFolder()`, `getDownloadInfo()`, `downloadFile()`
**의존**: ConfigManager, Logger, RetryManager

세션 흐름: `login()` -> 쿠키 획득 -> 15분 타이머 -> `validateSession()` -> 만료 시 `refreshSession()` -> 실패 시 Playwright 폴백.

### 3.3 WebhardUploader (인터페이스 + 구현체)

**패턴**: Strategy. 인터페이스를 Core에 정의하고 구현체를 교체 가능하게 설계.

```typescript
interface IWebhardUploader {
  checkConnection(): Promise<boolean>;
  createFolder(path: string): Promise<FolderInfo>;
  uploadFile(params: UploadParams): Promise<UploadResult>;
  fileExists(path: string, checksum?: string): Promise<boolean>;
  deleteFile(path: string): Promise<void>;
}
// 구현체: YjlaserWebhardUploader (Presigned URL + R2, 100MB+ 멀티파트, MD5 검증)
```

### 3.4 FileDetector

**책임**: LGU+ 웹하드 신규 파일 감지. 이중 감지 전략.
**API**: `start()`, `stop()`, `setPollingInterval()`, `forceCheck()`

| 전략 | 간격 | 방식 |
|------|------|------|
| **Primary** | 5초 | Upload History API 폴링, historyNo 체크포인트, 이벤트 발견 시 즉시 재폴링(최대 3회) |
| **Secondary** | 10분 | 전체 폴더 스냅샷 비교(파일명+크기+수정일), 폴링 누락 보완 |

### 3.5 EventBus

**책임**: Core 내부 모듈 간 느슨한 결합. Observer 패턴, 타입 안전 이벤트.
**의존**: 없음 (독립 모듈)

주요 이벤트: `sync:started`, `sync:completed`, `sync:failed`, `sync:progress`, `session:expired`, `session:renewed`, `detection:found`, `engine:status`

### 3.6 RetryManager

**책임**: 재시도 정책, 서킷 브레이커, DLQ 관리.
**API**: `execute(fn, options)`, `getCircuitState()`, `getDlqItems()`, `retryDlqItem(id)`, `retryAllDlq()`

- **재시도**: 1초 -> 3초 -> 10초, 3회 실패 시 DLQ 이동 (1시간 후 자동 재시도)
- **서킷 브레이커**: CLOSED --(실패율 40%)--> OPEN --(10초)--> HALF_OPEN --(성공)--> CLOSED

### 3.7 StateManager

**책임**: SQLite 기반 상태 영속화. WAL 모드.
**API**: `getCheckpoint()`, `saveCheckpoint()`, `saveEvent()`, `getEvents()`, `getStats()`, `getDlqItems()`

**핵심 테이블**:

| 테이블 | 용도 | 주요 컬럼 |
|--------|------|-----------|
| `checkpoints` | 폴링 재개 지점 | type, value(historyNo), updated_at |
| `sync_events` | 파일별 동기화 이력 | file_id, file_name, file_path, file_size, status, error_msg, retry_count |
| `dead_letter_queue` | 실패 큐 | event_id, error_type, retry_count, next_retry |
| `sync_folders` | 동기화 폴더 선택 | folder_id, folder_path, enabled, last_sync, file_count |
| `settings` | 사용자 설정 | key, value |
| `daily_stats` | 일별 통계 | date, total_files, success, failed, total_bytes |
| `logs` | 로그 (GUI 조회용) | level, message, context(JSON), timestamp |
| `schema_version` | 마이그레이션 버전 | version, applied_at |

### 3.8 Logger

**책임**: 구조화 로깅 (파일 JSON Lines + SQLite + 콘솔 + EventBus 스트리밍).
**API**: `debug()`, `info()`, `warn()`, `error()`, `child(context)`
상세는 [10. 로깅](#10-로깅) 참조.

### 3.9 ConfigManager

**책임**: 설정 읽기/쓰기/검증/암호화, 변경 이벤트 발행.
**API**: `get(key)`, `set(key, value)`, `getAll()`, `validate()`, `reset()`, `onChanged(key, handler)`

주요 설정: `lguplusId`, `lguplusPassword`(암호화), `webhardApiUrl`, `webhardApiKey`(암호화), `pollingIntervalSec`(기본5), `maxConcurrentDownloads`(기본5), `maxConcurrentUploads`(기본3), `tempDownloadPath`, `enableInAppNotification`, `enableToastNotification`, `autoStart`, `logRetentionDays`(기본30), `startMinimized`

### 3.10 NotificationService

**책임**: 인앱 알림 관리 + OS 토스트 알림 추상화 + 알림 그룹핑.
**API**: `notify()`, `getNotifications()`, `getUnreadCount()`, `markRead()`, `markAllRead()`, `clearOld()`
OS 알림은 `IOsNotifier` 인터페이스로 추상화. Main에서 Electron `Notification` 구현체를 주입.

### 3.11 모듈 스펙 → 구현 매핑

## 4. IPC 통신

### 4.1 타입 안전 IPC 패턴

`shared/ipc-types.ts`에 모든 채널의 요청/응답 타입을 공유 정의하여 컴파일 타임에 계약을 보장.

```typescript
// 요청/응답 (invoke/handle)
interface IpcHandlers {
  'sync:start':         { req: void;              res: void };
  'sync:stop':          { req: void;              res: void };
  'sync:pause':         { req: void;              res: void };
  'sync:resume':        { req: void;              res: void };
  'sync:fullSync':      { req: FullSyncOptions;   res: FullSyncResult };
  'sync:getStatus':     { req: void;              res: SyncEngineStatus };
  'sync:syncFile':      { req: { fileId: string }; res: SyncResult };
  'config:getAll':      { req: void;              res: AppConfig };
  'config:set':         { req: { key: string; value: unknown }; res: void };
  'config:testLguplus': { req: Credentials;       res: TestResult };
  'config:testWebhard': { req: WebhardConfig;     res: TestResult };
  'files:listFolder':   { req: { folderId: string }; res: FolderContents };
  'folders:getAll':     { req: void;              res: SyncFolder[] };
  'folders:toggle':     { req: { folderId: string; enabled: boolean }; res: void };
  'logs:query':         { req: LogQuery;          res: LogEntry[] };
  'logs:export':        { req: ExportOptions;     res: string };
  'stats:daily':        { req: StatsQuery;        res: DailyStat[] };
  'dlq:getAll':         { req: void;              res: DlqItem[] };
  'dlq:retry':          { req: { id: number };    res: SyncResult };
  'dlq:retryAll':       { req: void;              res: BatchResult };
  'notification:getAll': { req: NotificationFilter; res: AppNotification[] };
  'notification:read':   { req: { id: string };   res: void };
}

// 이벤트 스트리밍 (send/on) - Main -> Renderer 단방향
interface IpcEvents {
  'sync:progress':      SyncProgress;
  'sync:statusChanged': SyncEngineStatus;
  'sync:fileCompleted': CompletedFile;
  'sync:fileFailed':    FailedFile;
  'log:entry':          LogEntry;
  'notification:new':   AppNotification;
  'session:status':     SessionStatus;
}
```

### 4.2 preload (contextBridge)

```typescript
contextBridge.exposeInMainWorld('electronAPI', {
  invoke: <K extends keyof IpcHandlers>(
    channel: K, data?: IpcHandlers[K]['req']
  ): Promise<IpcHandlers[K]['res']> =>
    ipcRenderer.invoke(channel, data),

  on: <K extends keyof IpcEvents>(
    channel: K, handler: (data: IpcEvents[K]) => void
  ): (() => void) => {
    const listener = (_: any, data: IpcEvents[K]) => handler(data);
    ipcRenderer.on(channel, listener);
    return () => ipcRenderer.removeListener(channel, listener);
  },
});
```

### 4.3 채널 분류

| 카테고리 | 패턴 | 방향 | 예시 |
|----------|------|------|------|
| 명령 | invoke/handle | Renderer -> Main | `sync:start`, `config:set` |
| 쿼리 | invoke/handle | Renderer -> Main | `sync:getStatus`, `logs:query` |
| 이벤트 | send/on | Main -> Renderer | `sync:progress`, `log:entry` |

## 5. 프로세스 모델

### 5.1 역할 분리

| Main Process (Node.js) | Renderer Process (Chromium) |
|------------------------|-----------------------------|
| Core Layer 전체 실행 | React UI 렌더링 |
| SQLite 읽기/쓰기 | Zustand 상태 관리 |
| HTTP 네트워크 요청 | 사용자 입력 처리 |
| 파일 시스템 접근 | 차트/그래프 렌더링 |
| 시스템 트레이, Worker Thread 관리 | 가상 스크롤 (대량 목록) |

### 5.2 Worker Thread 활용

| 작업 | Worker 사용 | 사유 |
|------|------------|------|
| MD5 체크섬 계산 | O | 대용량 파일(100MB+) CPU 집약적 |
| 스냅샷 비교 | O | 10,000+ 파일 트리 diff 연산 |
| 파일 다운로드 | X | I/O 바운드, 비동기 스트림 충분 |
| HTTP API 호출 | X | 비동기 I/O |
| SQLite 쿼리 | X | 경량 쿼리, 대량 INSERT 시에만 검토 |

### 5.3 동시성 제어

- **다운로드 큐**: Semaphore, 최대 5 동시 (설정 가능)
- **업로드 큐**: Semaphore, 최대 3 동시 (설정 가능)
- **폴링 루프**: 단일 타이머, 5초 간격 `setInterval`

## 8. 의존성 주입

### 8.1 팩토리 함수 패턴

DI 컨테이너 없이 팩토리 함수로 모듈을 조립한다. 코드 투명성과 디버깅 용이성 우선.

```typescript
// src/main/services/core-factory.ts
export function createCoreModules(options: CoreOptions): CoreModules {
  // 1. 인프라 (의존 없음)
  const eventBus = new EventBus();
  const logger = new Logger({ level: options.logLevel, logDir: options.logDir, eventBus });

  // 2. 데이터
  const stateManager = new StateManager({ dbPath: options.dbPath, logger });

  // 3. 설정
  const configManager = new ConfigManager({
    stateManager, logger, eventBus, encryptionKey: options.encryptionKey
  });

  // 4. 비즈니스
  const retryManager = new RetryManager({ logger, stateManager, eventBus });
  const lguplusClient = new LGUplusClient({ configManager, logger, retryManager });
  const webhardUploader = new YjlaserWebhardUploader({ configManager, logger });
  const fileDetector = new FileDetector({ lguplusClient, stateManager, eventBus, logger });
  const notificationService = new NotificationService({
    stateManager, eventBus, configManager,
    osNotifier: options.osNotifier  // Main에서 Electron Notification 구현체 주입
  });

  // 5. 오케스트레이터
  const syncEngine = new SyncEngine({
    fileDetector, lguplusClient, webhardUploader,
    stateManager, retryManager, eventBus, logger, notificationService
  });

  return { syncEngine, configManager, stateManager, logger, notificationService, retryManager };
}
```

### 8.2 의존 관계

```
EventBus (독립)
Logger -> EventBus
StateManager -> Logger
ConfigManager -> StateManager, Logger, EventBus
RetryManager -> Logger, StateManager, EventBus
LGUplusClient -> ConfigManager, Logger, RetryManager
YjlaserWebhardUploader -> ConfigManager, Logger
FileDetector -> LGUplusClient, StateManager, EventBus, Logger
NotificationService -> StateManager, EventBus, ConfigManager, IOsNotifier
SyncEngine -> (모두)
```

### 8.3 테스트 시 Mock 주입

모든 의존성을 인터페이스로 정의하여 `vi.fn()`으로 Mock 객체를 주입. Core 전체를 Electron 없이 `vitest`로 단독 테스트.

### 8.4 스펙 우선 DI 규칙

SDD 원칙에 따른 의존성 주입 순서:

1. **인터페이스 정의**: `types/*.types.ts`에 모듈 인터페이스 정의
2. **팩토리 타입 선언**: 팩토리 함수의 파라미터와 반환 타입을 인터페이스로 명시
3. **구현체 작성**: 인터페이스를 `implements`하는 클래스/함수 작성
4. **팩토리 조립**: `createCoreModules()`에서 구현체를 인터페이스 타입으로 주입

```typescript
// [SPEC] 인터페이스 정의 (먼저)
interface ISyncEngine {
  start(): Promise<void>;
  stop(): Promise<void>;
}

// [IMPL] 구현체 (나중)
class SyncEngine implements ISyncEngine {
  constructor(deps: SyncEngineDeps) { /* ... */ }
  async start(): Promise<void> { /* ... */ }
  async stop(): Promise<void> { /* ... */ }
}

// [IMPL] 팩토리에서 인터페이스 타입으로 주입
function createCoreModules(): { syncEngine: ISyncEngine; /* ... */ } {
  const syncEngine: ISyncEngine = new SyncEngine(deps);
  return { syncEngine };
}
```

> 📌 DI와 SDD의 통합 전략은 [10-SDD-개발방법론](./10-SDD-개발방법론.md) §6 스펙 → 구현 매핑 규칙을 참조한다.

## 9. 에러 처리

### 9.1 에러 분류

```typescript
abstract class SyncAppError extends Error {
  abstract readonly code: string;
  abstract readonly category: ErrorCategory;  // NETWORK|AUTH|API|FILE|DB|CONFIG|INTERNAL
  abstract readonly retryable: boolean;
  readonly timestamp = Date.now();
  context?: Record<string, unknown>;
}
```

### 9.2 에러별 처리 정책

| 에러 | 코드 | 재시도 | 사용자 알림 |
|------|------|--------|------------|
| 네트워크 타임아웃 | `NETWORK_TIMEOUT` | 3회 지수백오프 | 3회 실패 시 |
| 네트워크 단절 | `NETWORK_DISCONNECTED` | 무제한 (최대 5분) | 즉시 |
| 세션 만료 | `AUTH_SESSION_EXPIRED` | 자동 재로그인 | 실패 시에만 |
| 잘못된 자격증명 | `AUTH_INVALID_CREDENTIALS` | 불가 | 즉시, 설정 유도 |
| CAPTCHA | `AUTH_CAPTCHA_REQUIRED` | 불가 | 즉시, 수동 로그인 안내 |
| API 404 | `API_NOT_FOUND` | 불가 | 로그만 |
| API 500 | `API_SERVER_ERROR` | 3회 -> DLQ | 3회 실패 시 |
| API 형식 변경 | `API_PARSE_ERROR` | 불가 | 즉시 (긴급) |
| 디스크 부족 | `FILE_DISK_FULL` | 불가 | 즉시, 동기화 중지 |
| 체크섬 불일치 | `FILE_CHECKSUM_MISMATCH` | 1회 재다운로드 | 2회 실패 시 |
| DB 손상 | `DB_CORRUPTED` | 불가 | 즉시, 백업 복원 |

### 9.3 에러 전파

```
Core 모듈 --throw SyncAppError--> SyncEngine
  ├─ retryable=true  --> RetryManager (재시도 -> 성공/DLQ)
  └─ retryable=false --> EventBus.emit('sync:failed')
       --> Main IPC --> Renderer UI 표시 + 알림
```

미처리 예외 안전망: `process.on('uncaughtException')` + `process.on('unhandledRejection')`에서 긴급 상태 저장 후 로그 기록.

## 10. 로깅

### 10.1 레벨 및 출력 대상

| 레벨 | 용도 |
|------|------|
| DEBUG | 개발 상세 (API 요청/응답, 상태 변경) |
| INFO | 주요 동작 (동기화 시작/완료, 세션 갱신) |
| WARN | 주의 (재시도, 세션 만료, 디스크 경고) |
| ERROR | 오류 (다운로드/업로드 실패, DB 오류) |

| 출력 대상 | 레벨 | 보관 |
|-----------|------|------|
| 파일 (JSON Lines) | 전체 | 일별 로테이션, 30일, 최대 50MB/파일 |
| SQLite (`logs` 테이블) | INFO+ | 30일 (GUI 조회용) |
| 콘솔 | 개발모드 DEBUG+ | 없음 |
| EventBus -> IPC | INFO+ | 없음 (UI 실시간 스트리밍) |

### 10.2 형식

```
# 파일 (JSON Lines)
{"ts":"2026-02-23T10:30:15.123Z","level":"INFO","msg":"파일 동기화 완료","ctx":{"fileId":"abc","fileName":"도면.dxf","size":1024000}}

# 콘솔 (개발용)
[10:30:15] INFO  파일 동기화 완료 | fileId=abc fileName=도면.dxf
```

### 10.3 민감정보 마스킹

자동 마스킹 대상: 비밀번호(`****`), API Key(`sk-***...***`), 세션 쿠키(`SESS=***...`), URL 내 `token`/`key`/`session` 파라미터.

### 10.4 GUI 연동

`Logger` -> `EventBus.emit('log:entry')` -> Main IPC `'log:entry'` -> Renderer `LogViewer`(가상 스크롤, 레벨/텍스트 필터, 날짜 범위, 내보내기).

## 11. 보안

### 11.1 Electron 보안 설정

| 설정 | 값 | 설명 |
|------|-----|------|
| `contextIsolation` | `true` | Renderer JS 컨텍스트 분리 |
| `nodeIntegration` | `false` | Renderer에서 Node.js 직접 사용 금지 |
| `sandbox` | `true` | Renderer 샌드박싱 |
| `webSecurity` | `true` | Same-Origin Policy |

추가: 외부 URL 네비게이션 차단(`will-navigate`), 새 창 열기 차단(`setWindowOpenHandler`).

### 11.2 자격증명 저장

1. **Electron safeStorage** (우선): OS 자격증명 관리자(Windows Credential Manager) 활용 암호화
2. **AES-256-GCM 폴백**: 머신 고유 키(MAC+컴퓨터명+사용자명 해시)로 암호화 -> SQLite 저장

### 11.3 네트워크 보안

- 모든 외부 통신 HTTPS(TLS 1.2+) 강제
- 세션 쿠키는 메모리만 보관, 디스크 저장 금지
- API Key는 `X-API-Key` 헤더 전송, URL 파라미터 포함 금지

### 11.4 CSP
