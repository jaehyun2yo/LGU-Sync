## 1. 데이터베이스 개요

### 1.1 기술 선택

| 항목 | 결정 | 근거 |
|------|------|------|
| DBMS | SQLite 3 (`better-sqlite3`) | 단일 사용자 데스크톱 앱, 서버 불필요, 동기 API |
| 저널 모드 | WAL | 읽기/쓰기 동시성, Electron 환경 안정성 |
| 인코딩 | UTF-8 | 한글 파일명/폴더명 지원 |
| 외래 키 | `PRAGMA foreign_keys = ON` | 참조 무결성 보장 |

### 1.2 DB 파일 위치

```
%APPDATA%/외부웹하드동기화프로그램/
├── sync.db              # 메인 DB
├── sync.db-wal / -shm   # WAL 자동 생성
└── backups/             # 일별 백업 (최근 7개 보관)
```

### 1.3 초기화 PRAGMA

```sql
PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 5000;
PRAGMA cache_size = -8000;       -- 8MB
PRAGMA synchronous = NORMAL;
PRAGMA temp_store = MEMORY;
PRAGMA mmap_size = 268435456;    -- 256MB
```

### 1.4 마이그레이션 전략

`app_settings` 테이블의 `system.schema_version` 키로 버전을 관리한다. 앱 시작 시 코드 요구 버전과 비교하여 순차 마이그레이션을 트랜잭션 내에서 실행한다. 실행 전 자동 백업, 실패 시 백업에서 복원.

### 1.5 SDD 데이터 스펙 원칙

본 프로젝트는 SDD(Specification-Driven Development)에 따라 데이터 계층도 스펙 우선으로 정의한다.

**SQL + TypeScript Row 타입 쌍 규칙:**
- 모든 테이블은 SQL DDL과 대응하는 TypeScript Row 인터페이스를 **쌍으로** 정의한다
- SQL 컬럼 타입과 TS 필드 타입은 다음 매핑을 따른다:

| SQL 타입 | TypeScript 타입 |
|---|---|
| `TEXT` | `string` |
| `INTEGER` | `number` |
| `REAL` | `number` |
| `BOOLEAN` (INTEGER 0/1) | `boolean` (런타임 변환) |
| `DATETIME` (TEXT) | `string` (ISO 8601) |
| `TEXT CHECK(...)` | union 리터럴 타입 |

**Zod 검증 스키마 규칙:**
- 외부 입력(IPC, API 응답)으로부터 DB에 삽입되는 데이터는 Zod 스키마로 검증한다
- Zod 스키마는 Row 인터페이스와 동일한 파일에 정의하거나 `validators/` 디렉토리에 배치한다
- `z.infer<typeof schema>`로 타입을 추출하여 Row 인터페이스와 일치시킨다

> 📌 SDD 데이터 스펙 체계의 전체 규칙은 [10-SDD-개발방법론](./10-SDD-개발방법론.md) §4.3을 참조한다.

## 2. 테이블 설계

### 2.1 sync_folders (동기화 대상 폴더)

LGU+ 웹하드 폴더와 자체웹하드 폴더 간 매핑. 사용자의 동기화 대상 선택 상태를 관리.

```sql
CREATE TABLE sync_folders (
    id                  TEXT PRIMARY KEY,
    lguplus_folder_id   TEXT NOT NULL UNIQUE,
    lguplus_folder_name TEXT NOT NULL,
    lguplus_folder_path TEXT,
    self_webhard_path   TEXT,
    company_name        TEXT,
    enabled             INTEGER NOT NULL DEFAULT 1,
    auto_detected       INTEGER NOT NULL DEFAULT 0,
    files_synced        INTEGER NOT NULL DEFAULT 0,
    bytes_synced        INTEGER NOT NULL DEFAULT 0,
    last_synced_at      TEXT,
    created_at          TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at          TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_sf_enabled ON sync_folders(enabled);
CREATE INDEX idx_sf_path ON sync_folders(lguplus_folder_path);
```

**핵심 컬럼:** `lguplus_folder_id` -- LGU+ API 폴더 고유 ID (UNIQUE). `enabled` -- 1=동기화 대상. `auto_detected` -- 1=스캔으로 자동 발견. `files_synced`/`bytes_synced` -- 누적 동기화 통계. 모든 시각은 ISO 8601 TEXT.

```sql
-- 예시
INSERT INTO sync_folders (id, lguplus_folder_id, lguplus_folder_name, lguplus_folder_path, company_name, enabled)
VALUES ('a1b2c3d4-...', '50821', '원컴퍼니', '/올리기전용/원컴퍼니', '원컴퍼니', 1);
```

**Zod 검증 스키마 (SDD Level 1):**

```typescript
import { z } from 'zod';

export const SyncFolderInsertSchema = z.object({
  lguplus_folder_id: z.string().min(1),
  lguplus_folder_name: z.string().min(1),
  lguplus_parent_id: z.string().nullable().default(null),
  enabled: z.boolean().default(true),
  auto_detected: z.boolean().default(false),
});

export const SyncFolderRowSchema = SyncFolderInsertSchema.extend({
  id: z.number().int().positive(),
  files_synced: z.number().int().nonnegative().default(0),
  bytes_synced: z.number().int().nonnegative().default(0),
  last_sync_at: z.string().datetime().nullable().default(null),
  created_at: z.string().datetime(),
  updated_at: z.string().datetime(),
});

export type SyncFolderInsert = z.infer<typeof SyncFolderInsertSchema>;
export type SyncFolderRow = z.infer<typeof SyncFolderRowSchema>;
```

### 2.2 sync_files (동기화 파일 기록)

감지된 각 파일의 동기화 생애주기를 추적. 파일당 1행.

```sql
CREATE TABLE sync_files (
    id                    TEXT PRIMARY KEY,
    folder_id             TEXT NOT NULL REFERENCES sync_folders(id) ON DELETE CASCADE,
    history_no            INTEGER,
    file_name             TEXT NOT NULL,
    file_path             TEXT NOT NULL,
    file_size             INTEGER NOT NULL DEFAULT 0,
    file_extension        TEXT,
    lguplus_file_id       TEXT,
    status                TEXT NOT NULL DEFAULT 'detected',
    download_path         TEXT,
    self_webhard_file_id  TEXT,
    md5_hash              TEXT,
    retry_count           INTEGER NOT NULL DEFAULT 0,
    last_error            TEXT,
    detected_at           TEXT NOT NULL,
    download_started_at   TEXT,
    download_completed_at TEXT,
    upload_started_at     TEXT,
    upload_completed_at   TEXT,
    created_at            TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at            TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE UNIQUE INDEX idx_sfi_history ON sync_files(history_no) WHERE history_no IS NOT NULL;
CREATE INDEX idx_sfi_folder ON sync_files(folder_id);
CREATE INDEX idx_sfi_status ON sync_files(status);
CREATE INDEX idx_sfi_detected ON sync_files(detected_at);
CREATE INDEX idx_sfi_folder_status ON sync_files(folder_id, status);
CREATE INDEX idx_sfi_lguplus_id ON sync_files(lguplus_file_id);
```

**핵심 컬럼:** `folder_id` -- FK(sync_folders.id). `history_no` -- LGU+ historyNo, 부분 UNIQUE 인덱스(폴링 감지 시, 스냅샷은 NULL). `status` -- 동기화 상태(섹션 3.1 참조). `lguplus_file_id` -- LGU+ ITEM_SRC_NO. `md5_hash` -- 다운로드 후 체크섬. `*_at` 컬럼 -- 각 단계별 타임스탬프(ISO 8601).

```sql
-- 예시
INSERT INTO sync_files (id, folder_id, history_no, file_name, file_path, file_size, file_extension, status, detected_at)
VALUES ('b2c3...', 'a1b2...', 1587432, 'design.dxf', '/올리기전용/원컴퍼니/design.dxf', 2457600, 'dxf', 'detected', '2026-02-23T09:30:00Z');
```

**Zod 검증 스키마 (SDD Level 1):**

```typescript
import { z } from 'zod';
import { FileSyncStatus } from './enums';

export const SyncFileInsertSchema = z.object({
  folder_id: z.number().int().positive(),
  history_no: z.number().int().positive(),
  file_name: z.string().min(1),
  file_path: z.string().min(1),
  file_size: z.number().int().nonnegative(),
  uploaded_at: z.string().datetime(),
  uploader_name: z.string().nullable().default(null),
});

export const SyncFileRowSchema = SyncFileInsertSchema.extend({
  id: z.number().int().positive(),
  status: z.nativeEnum(FileSyncStatus).default(FileSyncStatus.DETECTED),
  md5_hash: z.string().nullable().default(null),
  local_temp_path: z.string().nullable().default(null),
  retry_count: z.number().int().nonnegative().default(0),
  last_error: z.string().nullable().default(null),
  started_at: z.string().datetime().nullable().default(null),
  completed_at: z.string().datetime().nullable().default(null),
  created_at: z.string().datetime(),
  updated_at: z.string().datetime(),
});

export type SyncFileInsert = z.infer<typeof SyncFileInsertSchema>;
export type SyncFileRow = z.infer<typeof SyncFileRowSchema>;
```

### 2.3 sync_events (이벤트 로그)

불변 이벤트 로그. 감사 추적, 디버깅, 통계 집계에 사용.

```sql
CREATE TABLE sync_events (
    sequence_id   INTEGER PRIMARY KEY AUTOINCREMENT,
    event_id      TEXT NOT NULL UNIQUE,
    event_type    TEXT NOT NULL,              -- 'upload', 'folder_create'
    source        TEXT NOT NULL DEFAULT 'polling', -- 'polling', 'snapshot', 'manual'
    file_id       TEXT,
    folder_id     TEXT,
    history_no    INTEGER,
    file_name     TEXT,
    file_path     TEXT,
    file_size     INTEGER,
    status        TEXT NOT NULL DEFAULT 'logged', -- 'logged','processing','completed','failed'
    result        TEXT,                       -- 'success', 'skipped', 'failed'
    error_message TEXT,
    duration_ms   INTEGER,
    metadata      TEXT,                       -- JSON
    detected_at   TEXT NOT NULL,
    processed_at  TEXT,
    created_at    TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_se_status ON sync_events(status);
CREATE INDEX idx_se_type ON sync_events(event_type);
CREATE INDEX idx_se_detected ON sync_events(detected_at);
CREATE INDEX idx_se_file ON sync_events(file_id);
CREATE INDEX idx_se_history ON sync_events(history_no);
CREATE INDEX idx_se_date ON sync_events(date(created_at));
```

**핵심 컬럼:** `sequence_id` -- 자동 증가, 이벤트 순서 보장. `event_id` -- UUID, 외부 참조/중복 검사. `source` -- polling/snapshot/manual. `status` -- logged->processing->completed/failed. `metadata` -- JSON 추가 컨텍스트.

## 3. 상태 머신

### 3.1 파일 동기화 상태 (sync_files.status)

```
                         [재시도 초과]
  detected ──> downloading ──> uploading ──> completed
                  │                │
             [실패]│           [실패]│
                  v                v
              dl_failed        ul_failed
                  │                │
             [재시도]          [재시도]
                  v                v
              downloading      uploading
  
  detected ──[이미 존재]──> skipped
  
  dl_failed / ul_failed ──[재시도 초과]──> dlq ──[수동 재시도]──> detected
```

| 현재 상태 | 트리거 | 다음 상태 | 조건 |
|-----------|--------|-----------|------|
| `detected` | 다운로드 큐 픽업 | `downloading` | - |
| `detected` | 중복 발견 | `skipped` | 전체 동기화 시 |
| `downloading` | 성공 | `uploading` | 파일 저장 완료 |
| `downloading` | 실패 | `dl_failed` | 네트워크/서버 오류 |
| `dl_failed` | 재시도 | `downloading` | retry_count < max |
| `dl_failed` | 초과 | `dlq` | retry_count >= max |
| `uploading` | 성공 | `completed` | API 응답 성공 |
| `uploading` | 실패 | `ul_failed` | API 오류 |
| `ul_failed` | 재시도 | `uploading` | retry_count < max |
| `ul_failed` | 초과 | `dlq` | retry_count >= max |
| `dlq` | 수동 재시도 | `detected` | 사용자 클릭 |

**SDD 타입 정의 (Level 1):**

```typescript
// src/core/types/sync-status.types.ts
export type SyncFileStatus =
  | 'detected'
  | 'downloading'
  | 'dl_failed'
  | 'uploading'
  | 'ul_failed'
  | 'completed'
  | 'skipped'
  | 'dlq';

export type SyncSessionStatus = 'active' | 'completed' | 'failed' | 'cancelled';
export type SyncEventStatus = 'pending' | 'processing' | 'completed' | 'failed' | 'dlq';
```

### 3.2 세션 상태 (sync_sessions.status)

```
  started ──> running ──> completed
                │
           [오류] │  [취소]
                v     v
             failed  cancelled
```

### 3.3 이벤트 상태 (sync_events.status)

```
  logged ──> processing ──> completed
                 │
                 └──> failed
```

## 4. 인덱스 설계

### 4.1 쿼리 패턴별 인덱스 매핑

| 쿼리 패턴 | 사용처 | 인덱스 |
|-----------|--------|--------|
| 활성 폴더 목록 | 폴더 선택 UI, 폴링 필터 | `idx_sf_enabled` |
| 경로로 폴더 검색 | 폴더 매핑 | `idx_sf_path` |
| historyNo 중복 검사 | 폴링 감지 | `idx_sfi_history` (UNIQUE, partial) |
| 상태별 파일 목록 | 대시보드, 진행 중 표시 | `idx_sfi_status` |
| 폴더+상태 복합 조회 | 파일 탐색기 | `idx_sfi_folder_status` |
| 날짜별 파일 | 최근 동기화 목록 | `idx_sfi_detected` |
| LGU+ 파일 ID | 중복 검사 | `idx_sfi_lguplus_id` |
| 이벤트 처리 큐 | 이벤트 프로세서 | `idx_se_status` |
| 날짜별 이벤트 집계 | 통계 차트 | `idx_se_date` |
| 재시도 대상 | DLQ 자동 재시도 | `idx_fq_retry` |
| 스냅샷 배치 | 스냅샷 비교 | `idx_fs_batch` |
| 스냅샷 경로 | 델타 감지 | `idx_fs_path` |
| 설정 카테고리 | 설정 UI | `idx_as_category` |

## 5. 데이터 흐름도

### 5.1 파이프라인 단계별 테이블 갱신

| 단계 | 트리거 | 테이블 갱신 |
|------|--------|-------------|
| **감지** | 폴링/스냅샷 | `detection_checkpoints`: last_history_no 갱신 / `sync_files`: INSERT (status=detected) / `sync_events`: INSERT (status=logged) / `sync_sessions`: INSERT or UPDATE |
| **다운로드** | 큐 픽업 | `sync_files`: status=downloading, download_started_at / `sync_events`: status=processing |
| **업로드** | 다운로드 완료 | `sync_files`: status=uploading, upload_started_at |
| **완료** | 업로드 성공 | `sync_files`: status=completed, upload_completed_at, self_webhard_file_id / `sync_events`: status=completed, result=success / `sync_folders`: files_synced+=1, bytes_synced+=size / `sync_sessions`: completed_files+=1, transferred_bytes+=size |
| **실패** | 오류 발생 | `sync_files`: status=dl_failed/ul_failed, retry_count+=1 / 재시도 초과 시: status=dlq, `failed_queue` INSERT, `sync_events` result=failed |
| **스냅샷** | 10분 주기 | `file_snapshots`: 배치 INSERT -> 이전 배치와 비교 -> 변경 발견 시 sync_files/sync_events INSERT |

## 6. 데이터 정리 정책

### 6.1 자동 정리 규칙

| 테이블 | 대상 | 보관 기간 | 주기 |
|--------|------|-----------|------|
| `sync_events` | completed | 90일 | 일 1회 |
| `sync_events` | failed | 180일 | 일 1회 |
| `sync_files` | completed | 180일 | 일 1회 |
| `sync_files` | skipped | 30일 | 일 1회 |
| `file_snapshots` | 전체 | 최근 3배치 | 스냅샷 직후 |
| `sync_sessions` | completed/cancelled | 365일 | 월 1회 |
| `failed_queue` | can_retry=0 | 30일 | 일 1회 |
| `detection_checkpoints` | - | 영구 | 없음 |
| `app_settings` | - | 영구 | 없음 |

### 6.2 DB 백업

- **주기**: 일 1회 (앱 시작 시, 24시간 경과 확인)
- **보관**: 최근 7개, 초과분 자동 삭제
- **위치**: `%APPDATA%/.../backups/sync-{YYYY-MM-DD}.db`
- **방법**: SQLite `.backup()` API (온라인 백업)

### 6.3 정리 SQL

```sql
DELETE FROM sync_events WHERE status='completed' AND created_at < datetime('now','-90 days');
DELETE FROM sync_files WHERE status='completed' AND upload_completed_at < datetime('now','-180 days');
DELETE FROM sync_files WHERE status='skipped' AND created_at < datetime('now','-30 days');
DELETE FROM file_snapshots WHERE snapshot_batch_id NOT IN (
  SELECT DISTINCT snapshot_batch_id FROM file_snapshots ORDER BY captured_at DESC LIMIT 3);
DELETE FROM failed_queue WHERE can_retry=0 AND created_at < datetime('now','-30 days');
```

### 6.4 DB 크기 예상

| 시나리오 | 일일 파일 | 정리 후 최대 행(sync_files) | 예상 크기 |
|---------|----------|---------------------------|----------|
| 소규모 | 10건 | ~1,800행 | ~5MB |
| 일반 | 30건 | ~5,400행 | ~15MB |
| 대규모 | 100건 | ~18,000행 | ~50MB |
