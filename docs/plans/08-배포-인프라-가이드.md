## 1. 개발 환경 세팅

### 1.6 개발 서버 실행

```bash
# electron-vite 개발 모드 (HMR 지원)
npm run dev
```

`electron-vite dev`가 실행되면:
1. Main Process: TypeScript 감시 모드 컴파일
2. Preload: TypeScript 감시 모드 컴파일
3. Renderer: Vite 개발 서버 (HMR, port 5173)
4. Electron 앱 자동 실행, Renderer는 `http://localhost:5173` 로드

## 2. 빌드 프로세스

### 2.1 electron-vite 빌드 파이프라인

```
소스 코드                     빌드 결과
─────────────────────────────────────────────────
src/main/**       → CJS   → dist/main/index.js
src/preload/**    → CJS   → dist/preload/index.js
src/renderer/**   → ESM   → dist/renderer/index.html + assets/
```

`electron-vite`는 세 프로세스를 독립 빌드하며, 각 프로세스의 설정은 `electron.vite.config.ts`에서 관리한다.

```typescript
// electron.vite.config.ts
import { defineConfig, externalizeDepsPlugin } from 'electron-vite'
import react from '@vitejs/plugin-react'
import tailwindcss from '@tailwindcss/vite'

export default defineConfig({
  main: {
    plugins: [externalizeDepsPlugin()],
    build: {
      rollupOptions: {
        external: ['better-sqlite3', 'playwright']
      }
    }
  },
  preload: {
    plugins: [externalizeDepsPlugin()]
  },
  renderer: {
    plugins: [react(), tailwindcss()],
    build: {
      rollupOptions: {
        input: './src/renderer/index.html'
      }
    }
  }
})
```

### 2.2 스펙 적합성 검증 단계

빌드 프로세스에 SDD 스펙 적합성 검증을 포함한다:

```bash
# 1. 스펙 적합성 검증 (타입 체크만, 출력 없음)
npx tsc --noEmit

# 2. 기존 빌드 프로세스 계속
npm run build
```

`tsc --noEmit`은 모든 인터페이스 구현체가 스펙(타입 정의)을 준수하는지 컴파일 타임에 검증한다. 이 단계에서 실패하면 빌드가 중단된다.

### 2.3 TypeScript 컴파일 설정

| 설정 파일 | 대상 | target | module |
|-----------|------|--------|--------|
| `tsconfig.node.json` | Main + Preload | ES2022 | CommonJS |
| `tsconfig.web.json` | Renderer | ES2022 | ESNext |
| `tsconfig.json` | 기본 (references) | - | - |

핵심 컴파일러 옵션:
```json
{
  "compilerOptions": {
    "strict": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true,
    "isolatedModules": true,
    "paths": {
      "@core/*": ["./src/core/*"],
      "@shared/*": ["./src/shared/*"]
    }
  }
}
```

### 2.4 빌드 결과물 구조

```
dist/
├── main/
│   └── index.js              # Main Process 번들 (CJS)
├── preload/
│   └── index.js              # Preload 스크립트 (CJS)
└── renderer/
    ├── index.html             # 엔트리 HTML
    └── assets/
        ├── index-[hash].js    # React 앱 번들
        └── index-[hash].css   # Tailwind CSS 번들
```

### 2.5 Native 모듈 처리 (better-sqlite3)

`better-sqlite3`는 C++ Addon이므로 Electron의 Node.js ABI 버전에 맞게 리빌드가 필요하다.

```bash
# 개발 환경: 수동 리빌드
npx electron-rebuild -f -w better-sqlite3

# 패키징 시: electron-builder가 자동으로 리빌드 실행
# electron-builder.yml의 extraResources로 네이티브 바인딩 포함
```

`electron-vite`에서 `better-sqlite3`를 external로 처리하여 번들에 포함하지 않고, 런타임에 node_modules에서 직접 로드한다.

### 2.6 빌드 트러블슈팅

| 증상 | 원인 | 해결 |
|------|------|------|
| `MODULE_NOT_FOUND: better-sqlite3` | Electron ABI 불일치 | `npx electron-rebuild -f -w better-sqlite3` |
| `Error: The module was compiled against a different Node.js version` | 네이티브 모듈 버전 불일치 | `npm run rebuild` 후 재빌드 |
| Renderer 빈 화면 | 경로 오류, CSP 차단 | DevTools 콘솔 확인, `electron.vite.config.ts` base 경로 점검 |
| Tailwind 스타일 미적용 | content 경로 누락 | `tailwind.config.js`의 content 배열에 `src/renderer/**` 포함 확인 |
| `playwright` 빌드 실패 | 브라우저 미설치 | `npx playwright install chromium` |
| `node-gyp` 빌드 에러 | C++ 빌드 도구 미설치 | Visual Studio Build Tools 설치 |
| EPERM 파일 잠금 | 이전 Electron 프로세스 잔존 | 작업관리자에서 Electron 프로세스 종료 |

## 3. 패키징 (electron-builder)

### 3.1 electron-builder.yml 설정

```yaml
# electron-builder.yml
appId: com.yjlaser.webhard-sync
productName: 외부웹하드동기화
copyright: Copyright (c) 2026 유진레이저목형

directories:
  buildResources: build       # 아이콘, 인스톨러 리소스
  output: release/${version}  # 빌드 결과 (버전별 폴더)

files:
  - dist/**/*
  - node_modules/better-sqlite3/**/*
  - node_modules/bindings/**/*
  - node_modules/file-uri-to-path/**/*
  - "!**/*.{ts,map,md}"

extraResources:
  - from: node_modules/better-sqlite3/build/Release/
    to: native/better-sqlite3/
    filter:
      - "*.node"

win:
  target:
    - target: nsis
      arch: [x64]
  icon: build/icon.ico
  artifactName: "${productName}-Setup-${version}.${ext}"

nsis:
  oneClick: false                    # 비원클릭 (설치 경로 선택 가능)
  allowToChangeInstallationDirectory: true
  installerLanguages:
    - ko                             # 한국어
  language: 1042                     # 한국어 LCID
  createDesktopShortcut: true
  createStartMenuShortcut: true
  shortcutName: 외부웹하드동기화
  installerIcon: build/installer.ico
  uninstallerIcon: build/uninstaller.ico
  deleteAppDataOnUninstall: false    # 설정/DB 보존

portable:
  artifactName: "${productName}-Portable-${version}.${ext}"

publish:
  provider: github
  owner: yjlaser
  repo: webhard-sync-v2
  releaseType: release
```

### 3.2 코드 사이닝

내부용 앱이므로 코드 사이닝은 선택사항이다. 미서명 시 Windows SmartScreen 경고가 표시된다.

## 4. GitHub Actions CI/CD

### 4.1 워크플로우 개요

| 파일 | 트리거 | 역할 |
|------|--------|------|
| `.github/workflows/ci.yml` | PR, push (main/develop) | Lint + TypeCheck + 단위 테스트 |
| `.github/workflows/release.yml` | tag push (`v*`) | 빌드 + 패키징 + GitHub Release 생성 |

### 4.2 CI 워크플로우 (ci.yml)

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  lint-and-test:
    runs-on: windows-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: Rebuild native modules
        run: npx electron-rebuild -f -w better-sqlite3

      - name: Lint
        run: npm run lint

      - name: TypeCheck
        run: npm run typecheck

      - name: Unit Tests
        run: npm test -- --reporter=verbose

      - name: Build (검증)
        run: npm run build
```

### 4.3 SDD 스펙 검증 CI 단계

GitHub Actions 워크플로우에 스펙 검증 단계를 추가한다:

```yaml
jobs:
  spec-validation:
    name: SDD Spec Validation
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci

      # SDD 스펙 적합성 검증
      - name: Type Check (Spec Compliance)
        run: npx tsc --noEmit

      # 단위 테스트 (스펙 계약 테스트 포함)
      - name: Unit Tests
        run: npx vitest run --reporter=verbose

      # 린트 검사
      - name: Lint
        run: npm run lint
```

이 파이프라인은 다음 순서로 실행된다:
1. **`tsc --noEmit`**: 모든 구현체가 인터페이스 스펙을 준수하는지 검증
2. **`vitest run`**: 스펙 기반 계약 테스트 포함 단위 테스트 실행
3. **`lint`**: 코드 스타일 및 규칙 검사

> 스펙 검증이 실패하면 후속 단계는 실행되지 않는다 (fail-fast).

### 4.4 Release 워크플로우 (release.yml)

```yaml
# .github/workflows/release.yml
name: Release

on:
  push:
    tags:
      - 'v*'

permissions:
  contents: write

jobs:
  release:
    runs-on: windows-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: Rebuild native modules
        run: npx electron-rebuild -f -w better-sqlite3

      - name: Lint & TypeCheck
        run: |
          npm run lint
          npm run typecheck

      - name: Unit Tests
        run: npm test

      - name: Build & Package
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: npm run dist

      - name: Upload artifacts
        uses: actions/upload-artifact@v4
        with:
          name: release-${{ github.ref_name }}
          path: |
            release/**/*.exe
            release/**/latest.yml
          retention-days: 30

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v2
        with:
          files: |
            release/**/*-Setup-*.exe
            release/**/*-Portable-*.exe
            release/**/latest.yml
          draft: false
          prerelease: ${{ contains(github.ref, 'beta') || contains(github.ref, 'alpha') }}
          generate_release_notes: true
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

### 4.5 시크릿 설정

| 시크릿 | 용도 | 설정 위치 |
|--------|------|-----------|
| `GITHUB_TOKEN` | GitHub Release 생성, 아티팩트 업로드 | 자동 제공 (별도 설정 불필요) |
| `CSC_LINK` | 코드 사이닝 인증서 (Base64, 선택) | Repository Settings > Secrets |
| `CSC_KEY_PASSWORD` | 인증서 비밀번호 (선택) | Repository Settings > Secrets |

> `GITHUB_TOKEN`은 GitHub Actions에서 자동으로 제공하므로 별도 설정이 필요 없다.
> `electron-builder`가 GitHub Release에 아티팩트를 업로드할 때 `GH_TOKEN` 환경변수를 참조한다.

### 4.6 아티팩트 업로드

CI에서 생성된 빌드 결과물은 두 곳에 보관된다:

| 위치 | 보관 기간 | 용도 |
|------|-----------|------|
| GitHub Actions Artifacts | 30일 | 빌드 검증, QA 테스트 |
| GitHub Releases | 영구 | 배포, 자동 업데이트 서버 |

## 5. 자동 업데이트 (electron-updater)

### 5.1 설정

`electron-builder.yml`의 `publish` 섹션이 GitHub Releases를 업데이트 서버로 사용하도록 설정한다. 별도 업데이트 서버가 필요 없다.

```yaml
# electron-builder.yml (위 3.1 참조)
publish:
  provider: github
  owner: yjlaser
  repo: webhard-sync-v2
  releaseType: release
```

### 5.2 업데이트 흐름

```
앱 시작 ─────────────────────────────────────────────────────────────
  │
  ├─ (1) autoUpdater.checkForUpdates()
  │      GitHub Releases의 latest.yml 조회
  │
  ├─ (2) 새 버전 발견 → 'update-available' 이벤트
  │      → Renderer에 업데이트 알림 표시
  │
  ├─ (3) 백그라운드 다운로드 → 'download-progress' 이벤트
  │      → 다운로드 진행률 표시
  │
  ├─ (4) 다운로드 완료 → 'update-downloaded' 이벤트
  │      → 사용자에게 재시작 확인 알림
  │
  └─ (5) 사용자 승인 → autoUpdater.quitAndInstall()
         → 앱 종료 → NSIS 인스톨러 실행 → 앱 재시작
         → StateManager 마이그레이션 자동 실행
```

### 5.3 Main Process autoUpdater 코드 패턴

```typescript
// src/main/auto-updater.ts
import { autoUpdater, UpdateInfo } from 'electron-updater'
import { BrowserWindow } from 'electron'
import log from 'electron-log'

export function initAutoUpdater(mainWindow: BrowserWindow): void {
  // 로깅 설정
  autoUpdater.logger = log
  autoUpdater.autoDownload = true
  autoUpdater.autoInstallOnAppQuit = true

  // 업데이트 채널 (latest | beta)
  // autoUpdater.channel = 'latest'

  // 업데이트 확인 간격: 6시간
  const CHECK_INTERVAL = 6 * 60 * 60 * 1000

  autoUpdater.on('checking-for-update', () => {
    log.info('업데이트 확인 중...')
  })

  autoUpdater.on('update-available', (info: UpdateInfo) => {
    log.info(`새 버전 발견: ${info.version}`)
    mainWindow.webContents.send('update:available', {
      version: info.version,
      releaseNotes: info.releaseNotes
    })
  })

  autoUpdater.on('update-not-available', () => {
    log.info('최신 버전입니다.')
  })

  autoUpdater.on('download-progress', (progress) => {
    mainWindow.webContents.send('update:progress', {
      percent: progress.percent,
      transferred: progress.transferred,
      total: progress.total
    })
  })

  autoUpdater.on('update-downloaded', (info: UpdateInfo) => {
    log.info(`업데이트 다운로드 완료: ${info.version}`)
    mainWindow.webContents.send('update:downloaded', {
      version: info.version
    })
  })

  autoUpdater.on('error', (err) => {
    log.error('업데이트 오류:', err.message)
    // 업데이트 실패는 앱 동작에 영향 없음 (무시)
  })

  // 앱 시작 시 즉시 확인
  autoUpdater.checkForUpdates().catch(() => {})

  // 주기적 확인
  setInterval(() => {
    autoUpdater.checkForUpdates().catch(() => {})
  }, CHECK_INTERVAL)
}

// Renderer에서 "지금 재시작" 요청 시 호출
export function installUpdateAndRestart(): void {
  autoUpdater.quitAndInstall(false, true)
}
```

### 6.4 민감 정보 관리 (safeStorage)

비밀번호, API Key 등 민감 정보는 Electron `safeStorage`를 사용하여 OS 자격증명 관리자(Windows Credential Manager)로 암호화한다.

```typescript
import { safeStorage } from 'electron'

function encryptValue(value: string): string {
  if (safeStorage.isEncryptionAvailable()) {
    const encrypted = safeStorage.encryptString(value)
    return encrypted.toString('base64')
  }
  // 폴백: AES-256-GCM (머신 고유 키)
  return aes256Encrypt(value, getMachineKey())
}

function decryptValue(encrypted: string): string {
  if (safeStorage.isEncryptionAvailable()) {
    const buffer = Buffer.from(encrypted, 'base64')
    return safeStorage.decryptString(buffer)
  }
  return aes256Decrypt(encrypted, getMachineKey())
}
```

| 항목 | 암호화 방식 |
|------|------------|
| LGU+ 비밀번호 | safeStorage (우선) / AES-256-GCM (폴백) |
| 웹하드 API Key | safeStorage (우선) / AES-256-GCM (폴백) |
| LGU+ 아이디 | 평문 (민감도 낮음) |
| 기타 설정 | 평문 |

## 7. 버전 관리 및 릴리스 절차

### 7.1 Semantic Versioning

`MAJOR.MINOR.PATCH` 규칙을 따른다.

| 변경 유형 | 버전 증가 | 예시 |
|-----------|-----------|------|
| 호환 안 되는 변경 (DB 스키마, 설정 형식) | MAJOR | 1.0.0 -> 2.0.0 |
| 새 기능 추가 (하위 호환) | MINOR | 1.0.0 -> 1.1.0 |
| 버그 수정, 성능 개선 | PATCH | 1.0.0 -> 1.0.1 |
| 사전 릴리스 (베타) | 프리릴리스 라벨 | 1.1.0-beta.1 |

### 7.2 git tag 컨벤션

```
v{MAJOR}.{MINOR}.{PATCH}          # 정식 릴리스: v1.0.0
v{MAJOR}.{MINOR}.{PATCH}-beta.{N} # 베타: v1.1.0-beta.1

```

```
   - 설정(config.json)과 DB(sync.db)는 %APPDATA%에 보존
```

### 8.3 설정/DB 호환성 주의사항

| 항목 | 하위 호환 | 상위 호환 | 비고 |
|------|-----------|-----------|------|
| config.json | O | O | 신규 키는 기본값 사용, 제거된 키는 무시 |
| sync.db | O | **X** | DB 마이그레이션은 단방향 (up만 존재) |
| electron-store | O | O | JSON 형식, 유연한 스키마 |

> **DB 롤백 주의**: 새 버전에서 DB 마이그레이션(테이블/컬럼 추가)이 실행된 후 이전 버전으로 롤백하면, 이전 버전이 새 컬럼/테이블을 인식하지 못할 수 있다. SQLite는 미지의 컬럼을 무시하므로 대부분 문제없지만, 스키마 변경이 큰 MAJOR 업데이트 롤백 시에는 DB 백업 복원이 필요하다.

### 8.4 DB 백업 복원

```
%APPDATA%/외부웹하드동기화/
├── sync.db              # 현재 DB
└── backups/
    ├── sync-2026-02-22.db   # 일별 자동 백업
    ├── sync-2026-02-21.db
    └── ...                   # 최근 7개 보관

복원 절차:
1. 앱 완전 종료
2. sync.db 삭제 (또는 이름 변경)
3. backups/sync-YYYY-MM-DD.db를 sync.db로 복사
4. sync.db-wal, sync.db-shm 파일이 있으면 삭제
5. 앱 재시작
```

```bash
#!/bin/bash
# scripts/verify-spec-compliance.sh
# SDD 스펙 적합성 배포 전 검증 스크립트

set -e

echo "=== SDD 스펙 적합성 검증 시작 ==="

# 1. TypeScript 타입 체크 (스펙 준수 검증)
echo "[1/4] TypeScript 타입 체크..."
npx tsc --noEmit
echo "✓ 타입 체크 통과"

# 2. 단위 테스트 실행
echo "[2/4] 단위 테스트 실행..."
npx vitest run --reporter=verbose
echo "✓ 단위 테스트 통과"

# 3. 린트 검사
echo "[3/4] 린트 검사..."
npm run lint
echo "✓ 린트 검사 통과"

# 4. 빌드 검증
echo "[4/4] 빌드 검증..."
npm run build
echo "✓ 빌드 성공"

echo "=== SDD 스펙 적합성 검증 완료 ==="
```

### 수동 체크리스트

| # | 항목 | 검증 방법 | 통과 기준 |
|---|---|---|---|
| 1 | 타입 체크 통과 | `npx tsc --noEmit` | 에러 0개 |
| 2 | 단위 테스트 통과 | `npx vitest run` | 전체 통과 |
| 3 | Core 커버리지 | `npx vitest run --coverage` | Line 70%+ |
| 4 | 린트 통과 | `npm run lint` | 에러 0개 |
| 5 | 빌드 성공 | `npm run build` | 에러 없이 완료 |
| 6 | 스펙 파일 변경 확인 | `git diff --name-only` | 스펙 변경 시 관련 테스트도 변경 |
| 7 | 인터페이스 호환성 | 코드 리뷰 | 기존 인터페이스 변경 시 하위 호환 유지 |

> 📌 SDD 검증 체크리스트의 상세는 [10-SDD-개발방법론](./10-SDD-개발방법론.md) §8을 참조한다.
