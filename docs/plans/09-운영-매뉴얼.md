# 외부웹하드동기화프로그램 v2 - 운영 매뉴얼

> **버전**: 1.0 | **작성일**: 2026-02-23 | **대상**: 개발자/운영자 (본인)
> **선행 문서**: [10-SDD-개발방법론](./10-SDD-개발방법론.md)

## 1. 시스템 개요

LGU+ 웹하드 `올리기전용/` → 내부 웹하드(Cloudflare R2) 단방향 자동 동기화. Electron + 시스템 트레이 상주 + 자동 시작.

### 1.1 주요 경로

| 항목 | 경로 |
|------|------|
| 데이터 루트 | `%APPDATA%/외부웹하드동기화프로그램/` |
| DB | `{루트}/sync.db` (WAL: `sync.db-wal`, `sync.db-shm`) |
| DB 백업 | `{루트}/backups/sync-{YYYY-MM-DD}.db` (최근 7개) |
| 로그 | `{루트}/logs/main-{YYYY-MM-DD}.log` (일별 롤링) |
| 쿠키 | `{루트}/data/cookie-store.json` |
| 임시 다운로드 | `{루트}/temp/downloads/` |
| 설치 경로 | `%LOCALAPPDATA%/외부웹하드동기화프로그램/` |

### 1.2 프로세스 구조

- **Main Process**: 동기화 엔진(Primary 5초 폴링 / Secondary 10분 스냅샷 / Tertiary 1시간 무결성 검증), 다운로드 관리자(최대 5 동시), 업로드 관리자(최대 3 동시), 인증 관리자(Playwright), DB(better-sqlite3 WAL), 트레이
- **Renderer Process**: 대시보드, 파일 목록, 실패 큐(DLQ), 설정

### 1.3 의존 시스템

| 시스템 | 장애 시 영향 |
|--------|-------------|
| LGU+ 웹하드 (only.webhard.co.kr) | 감지/다운로드 전면 중단 |
| 자체 웹하드 API (Cloudflare R2) | 업로드만 실패 |
| 인터넷 | 전면 중단 |

## 3. 트러블슈팅 가이드

### 3.1 인증 문제

> **기대 동작 (SDD 스펙)**: `ILGUplusClient.login()` → 성공 시 `true` 반환, 실패 시 `AuthError` throw. `ILGUplusClient.validateSession()` → 세션 유효 시 `true`. 스펙 참조: `src/core/types/lguplus-client.types.ts`

#### LGU+ 로그인 실패 (`LOGIN_FAILED`)

**원인**: ID/PW 변경, 계정 잠김, LGU+ 사이트 변경

**해결**:
1. 브라우저에서 직접 https://only.webhard.co.kr 로그인 → ID/PW 확인
2. 성공: 프로그램 설정에서 재입력 / 실패: LGU+ 고객센터 문의
3. 사이트 변경 시: Playwright 셀렉터 코드 수정 필요 (7.4절)

#### 세션 만료 반복 (`SESSION_EXPIRED` 빈번)

**원인**: 동일 계정 다중 접속, LGU+ 세션 정책 변경, cookie-store.json 손상

**해결**:
1. 다른 기기에서 동일 계정 로그인 여부 확인 (중복 로그인 제거)
2. `cookie-store.json` 삭제 후 재시작 (강제 재로그인)

#### CAPTCHA 발생 (`CAPTCHA_REQUIRED`)

**원인**: 짧은 시간 내 로그인 반복 시도

**해결**:
1. 프로그램 일시 정지
2. 브라우저에서 직접 로그인하여 CAPTCHA 수동 해결
3. 30분~1시간 대기 후 프로그램 재시작

#### Playwright 브라우저 오류

**증상**: "Browser launch failed", "Protocol error", "Target closed"

**해결**:
1. 작업 관리자에서 `chromium` / `chrome` 프로세스 강제 종료
2. `{루트}/playwright-browsers/` 삭제 후 재설치
3. 메모리 여유 512MB 이상 확인
4. 최후 수단: 프로그램 재설치 (7.2절)

### 3.2 동기화 문제

> **기대 동작 (SDD 스펙)**: `ISyncEngine.start()` → 상태 `RUNNING`으로 전환. `IFileDetector.startPolling()` → 5초 간격 폴링 시작. 스펙 참조: `src/core/types/sync-engine.types.ts`, `src/core/types/file-detector.types.ts`

#### 파일 감지 안됨 (가장 중요!)

LGU+ 웹하드에 파일이 올라왔는데 감지 안될 때 **단계적 확인**:

| 순서 | 확인 사항 | 해결 |
|------|----------|------|
| 1 | 엔진 상태가 RUNNING인가 | RUNNING 아니면 시작 클릭 |
| 2 | 서킷 브레이커가 OPEN인가 | 30초 후 자동 복구 대기 |
| 3 | 인증이 만료되었는가 | 재로그인 |
| 4 | 해당 폴더가 enabled=1인가 | 비활성화 폴더면 활성화 |
| 5 | 파일이 필터 대상인가 | `.tmp`, `Thumbs.db` 등은 자동 제외 |
| 6 | 이미 동기화 완료인가 | DB 조회: `completed`/`skipped`면 정상 |
| 7 | Primary 폴링이 진행되는가 | 로그에서 `last_history_no` 갱신 확인 |
| 8 | Secondary에서 잡히는가 | 10분 대기 후 스냅샷 비교 확인 |

**DB 직접 확인**:
```sql
SELECT file_name, status, detected_at FROM sync_files ORDER BY detected_at DESC LIMIT 20;
SELECT * FROM sync_files WHERE file_name LIKE '%파일명%';
SELECT * FROM detection_checkpoints;
```

#### 동기화 시작 안됨

**원인**: INITIALIZING 정체, 인증 실패, DB 초기화 실패

**해결**: 로그 확인 → 인증 문제면 3.1절 / DB 문제면 3.5절 / FATAL이면 7.1절 초기화

#### 동기화 hang (멈춤)

**증상**: 마지막 동기화 시간 미갱신, `downloading` 상태 5분+ 유지

**해결**:
1. 진행 중 작업 확인, 장기 정체 파일 스킵
2. 로그에서 `timeout` 검색
3. 프로그램 재시작 (체크포인트에서 이어서 처리)

#### 특정 파일만 반복 실패

DLQ 에러 메시지 확인 후:
- `FILE_NOT_FOUND` → LGU+ 웹하드에서 원본 존재 확인
- `UPLOAD_FAILED` → 내부 웹하드 API 확인
- `CHECKSUM_MISMATCH` → 네트워크 불안정, 재시도
- 반복 시: DLQ에서 삭제 + 수동 처리

#### 중복 다운로드

**해결**: DB에서 동일 파일 이벤트 중복 확인 → 프로그램 재시작 (LRU 캐시 초기화)

### 3.3 다운로드 문제

> **기대 동작 (SDD 스펙)**: `ILGUplusClient.downloadFile()` → 성공 시 `{success: true, size, filename}` 반환. `IRetryManager` → 최대 3회 재시도 후 DLQ 이동. 스펙 참조: `src/core/types/lguplus-client.types.ts`, `src/core/types/retry-manager.types.ts`

#### 속도 저하

`sync.max_concurrent_downloads` 값 감소 (3 또는 1) + 다른 앱 대역폭 확인

#### 네트워크 실패 (`NETWORK_TIMEOUT`, `CONNECTION_REFUSED`)

1. `ping only.webhard.co.kr` / `nslookup only.webhard.co.kr`
2. 방화벽/백신에서 프로그램 허용 확인
3. 자동 재시도 대기 (지수 백오프: 1→2→4→8→16초, 최대 5회)

#### 대용량 파일 타임아웃 (100MB+)

다운로드 타임아웃 증가 (300000 → 600000ms) + 동시 다운로드 1로 감소

#### 디스크 공간 부족 (`STORAGE_FULL`)

FileSystemError는 **재시도 없이 즉시 실패**. `temp/downloads/` 정리 + 디스크 확보 후 DLQ에서 수동 재시도.

### 3.4 프로그램 문제

> **기대 동작 (SDD 스펙)**: `ISyncEngine` 상태 머신: `INITIALIZING → RUNNING ↔ PAUSED → STOPPING → STOPPED`. 비정상 상태 시 `ERROR` 전환 후 자동 복구 시도. 스펙 참조: `src/core/types/sync-engine.types.ts`

#### 시작 안됨

1. 작업 관리자에서 중복 프로세스 확인 → 종료 후 재시작
2. `sync.db-wal`, `sync.db-shm` 삭제 후 재시작
3. 로그에서 시작 에러 확인

#### White screen

`Ctrl+Shift+I` → DevTools Console 에러 확인. 캐시 삭제 (`{루트}/Cache/`) 후 재시작.

#### 메모리 과다 (1GB+)

1. `chromium` 고아 프로세스 종료
2. 이벤트 큐 10,000건 이상이면 비정상
3. 프로그램 재시작

#### CPU 과다 (30%+)

폴링 간격 확인 (기본 5초) + 동시 다운로드/업로드 수 줄이기

#### 트레이 아이콘 없음

Windows 설정 > 작업 표시줄 > 시스템 트레이 > "항상 표시"로 변경

### 3.5 DB 문제

> **기대 동작 (SDD 스펙)**: SQLite WAL 모드, PRAGMA 설정은 `src/core/db/schema.ts`에 정의. DB Row 타입은 해당 스키마의 TypeScript 인터페이스와 1:1 대응. 스펙 참조: `src/core/db/schema.ts`

#### DB 손상 ("database disk image is malformed")

```bash
sqlite3 "%APPDATA%/외부웹하드동기화프로그램/sync.db" "PRAGMA integrity_check;"
```
`ok`면 다른 원인 / 오류면 5.3절 백업 복원 / 백업 없으면 DB 삭제 후 전체 재동기화

#### DB 락 (`SQLITE_BUSY`)

1. 중복 인스턴스 종료 + DB Browser 등 외부 도구 종료
2. `sync.db-wal`, `sync.db-shm` 삭제 후 재시작
3. `busy_timeout=5000`이므로 5초 내 자동 재시도됨

#### 마이그레이션 실패

로그에서 실패 버전 확인 → `backups/`에서 마이그레이션 전 백업 복원 (5.3절)

## 4. 에러 코드 빠른 참조표

| 에러 코드 | 분류 | 즉시 조치 | 재시도 정책 |
|-----------|------|----------|-----------|
| `NETWORK_TIMEOUT` | Network | 자동 재시도 대기 | 지수백오프 5회 (1→2→4→8→16초) |
| `CONNECTION_REFUSED` | Network | 서버/방화벽 확인 | 지수백오프 5회 |
| `SESSION_EXPIRED` | Auth | 자동 재인증 | 재인증 후 1회 |
| `LOGIN_FAILED` | Auth | ID/PW 확인 | 재인증 후 1회 |
| `CAPTCHA_REQUIRED` | Auth | 브라우저에서 수동 해결 | 없음 |
| `FILE_NOT_FOUND` | Server | 원본 파일 확인 | 고정 30초 3회 |
| `DOWNLOAD_FAILED` | Server | 자동 재시도 대기 | 고정 30초 3회 |
| `UPLOAD_FAILED` | Server | 내부 웹하드 API 확인 | 고정 30초 3회 |
| `STORAGE_FULL` | FileSystem | 디스크 공간 확보 | 없음 (즉시 실패) |
| `API_ERROR` | Server | 로그에서 응답 확인 | 고정 30초 3회 |
| `CHECKSUM_MISMATCH` | Download | 자동 재다운로드 | 지수백오프 5회 |
| `UNKNOWN` | Unknown | 로그 확인 | 지수백오프 5회 |

**서킷 브레이커**: 최근 20건 중 실패율 >= 40% → OPEN(모두 차단) → 30초 후 HALF_OPEN(3건 테스트) → 2/3 성공 시 CLOSED

**에러 코드 스펙 정의 위치:**

| 에러 코드 접두사 | 스펙 파일 | SDD Level |
|---|---|---|
| `AUTH_*` | `src/core/errors/auth-errors.ts` | Level 3 |
| `NET_*` | `src/core/errors/network-errors.ts` | Level 3 |
| `DL_*` | `src/core/errors/download-errors.ts` | Level 3 |
| `UL_*` | `src/core/errors/upload-errors.ts` | Level 3 |
| `SYNC_*` | `src/core/errors/sync-errors.ts` | Level 3 |
| `DB_*` | `src/core/errors/db-errors.ts` | Level 3 |
| `FS_*` | `src/core/errors/filesystem-errors.ts` | Level 3 |
| `SYS_*` | `src/core/errors/system-errors.ts` | Level 3 |

모든 에러 클래스는 `SyncAppError` 추상 클래스를 확장한다 (`src/core/errors/index.ts`).

## 5. DB 관리

### 5.1 백업/복원

**자동 백업**: 앱 시작 시 24시간 경과 확인, 최근 7개 보관

**수동 백업**:
```bash
sqlite3 "%APPDATA%/외부웹하드동기화프로그램/sync.db" ".backup '%APPDATA%/외부웹하드동기화프로그램/backups/sync-manual.db'"
```

**복원**:
1. 프로그램 종료
2. `ren sync.db sync.db.broken`
3. `copy backups\sync-{날짜}.db sync.db`
4. `del sync.db-wal sync.db-shm`
5. 재시작 (복원 이후 파일은 재감지, 중복은 스킵)

### 5.2 수동 정리 SQL

```sql
DELETE FROM sync_events WHERE status='completed' AND created_at < datetime('now','-90 days');
DELETE FROM sync_files WHERE status='completed' AND upload_completed_at < datetime('now','-180 days');
DELETE FROM sync_files WHERE status='skipped' AND created_at < datetime('now','-30 days');
DELETE FROM file_snapshots WHERE snapshot_batch_id NOT IN (
  SELECT DISTINCT snapshot_batch_id FROM file_snapshots ORDER BY captured_at DESC LIMIT 3);
DELETE FROM failed_queue WHERE can_retry=0 AND created_at < datetime('now','-30 days');
VACUUM;
```

### 5.3 자동 정리 정책 (참고)

| 테이블 | 대상 | 보관 | 주기 |
|--------|------|------|------|
| sync_events | completed | 90일 | 일 1회 |
| sync_events | failed | 180일 | 일 1회 |
| sync_files | completed | 180일 | 일 1회 |
| sync_files | skipped | 30일 | 일 1회 |
| file_snapshots | 전체 | 최근 3배치 | 스냅샷 직후 |
| failed_queue | can_retry=0 | 30일 | 일 1회 |
| detection_checkpoints | - | 영구 | - |

### 5.4 무결성 검사

```bash
sqlite3 "%APPDATA%/외부웹하드동기화프로그램/sync.db" "PRAGMA integrity_check;"
```

## 6. 로그 분석

### 6.1 구조

경로: `{루트}/logs/main-{YYYY-MM-DD}.log`
형식: `[YYYY-MM-DD HH:mm:ss.SSS] [LEVEL] [MODULE] message`

| 레벨 | 의미 | 조치 |
|------|------|------|
| `DEBUG` | 상세 디버깅 | 불필요 |
| `INFO` | 정상 동작 | 불필요 |
| `WARN` | 재시도, 서킷 변경 등 | 빈도 모니터링 |
| `ERROR` | 실패, 인증 오류, DB 오류 | 원인 파악 |

### 6.2 주요 로그 패턴

**정상**: `[INFO] [SyncEngine] Primary polling: N new events detected` → `[INFO] Download completed` → `[INFO] Upload completed`

**재시도**: `[WARN] Download failed: ... (NETWORK_TIMEOUT)` → `[WARN] Queued for retry (attempt N/5)`

**서킷 오픈**: `[WARN] [CircuitBreaker] Failure rate 45%, state: CLOSED -> OPEN` → (30초) → `HALF_OPEN` → `CLOSED`

**재인증**: `[WARN] API returned 401` → `[INFO] Re-authentication successful`

### 6.3 문제 추적 명령어

```powershell
# 최근 ERROR 20건
Select-String "$env:APPDATA\외부웹하드동기화프로그램\logs\main-*.log" -Pattern "\[ERROR\]" | Select-Object -Last 20

# 특정 파일 추적
Select-String "$env:APPDATA\외부웹하드동기화프로그램\logs\main-*.log" -Pattern "파일명.dxf"

# 실시간 모니터링
Get-Content "$env:APPDATA\외부웹하드동기화프로그램\logs\main-$(Get-Date -Format 'yyyy-MM-dd').log" -Wait -Tail 50
```

### 6.4 로그 정리

자동: `system.log_retention_days` (기본 30일). 수동:
```powershell
Get-ChildItem "$env:APPDATA\외부웹하드동기화프로그램\logs\*.log" | Where-Object { $_.LastWriteTime -lt (Get-Date).AddDays(-30) } | Remove-Item
```

## 7. 비상 대응 절차

### 7.1 프로그램 완전 초기화

**주의: 모든 이력/설정 삭제. 전체 재동기화 시작됨.**

1. 프로그램 종료 + 작업 관리자에서 관련 프로세스 모두 종료
2. `rmdir /s "%APPDATA%\외부웹하드동기화프로그램"`
3. 재시작 → 초기 설정 재입력 (LGU+ ID/PW, API 키)

### 7.2 프로그램 재설치

1. DB 백업: `copy sync.db "%USERPROFILE%\Desktop\sync-backup.db"`
2. 프로그램 제거
3. 잔여 삭제: `rmdir /s "%APPDATA%\외부웹하드동기화프로그램"` + `rmdir /s "%LOCALAPPDATA%\외부웹하드동기화프로그램"`
4. 새 버전 설치 → 백업 복원 (5.1절) 또는 초기 설정

### 7.3 데이터 복구 (백업 없을 때)

DB 삭제 후 재시작 → 전체 재동기화. 내부 웹하드에 이미 있는 파일은 중복 제거로 스킵. 이력/통계만 소실.

### 7.4 LGU+ API 변경 대응

**증상**: 갑자기 모든 동기화 실패

**임시**: 브라우저에서 LGU+ 사이트 확인 → 변경 확인 → 수동 동기화로 전환

**근본**: Playwright 셀렉터, API 엔드포인트/응답 형식, 쿠키 구조 코드 수정 후 빌드/배포

## 스펙 기반 상태 검증

운영 중 동기화 엔진의 상태가 SDD 스펙과 일치하는지 확인하는 절차:

### 엔진 상태 검증

```bash
# 1. 엔진 상태 확인 (IPC 채널을 통해)
# 기대값: SyncEngineStatus = 'running' | 'paused' | 'stopped' | 'error'
# GUI 대시보드 → 동기화 상태 카드에서 확인 가능

# 2. DB 상태 일관성 확인
sqlite3 "%APPDATA%/외부웹하드동기화프로그램/sync.db" "
  -- 비정상 상태 파일 확인 (stuck files)
  SELECT COUNT(*) as stuck_files
  FROM sync_files
  WHERE status IN ('downloading', 'uploading')
  AND updated_at < datetime('now', '-1 hour');

  -- DLQ 대기 건수 확인
  SELECT COUNT(*) as dlq_count
  FROM failed_queue
  WHERE status = 'pending';

  -- 최근 1시간 동기화 성공률
  SELECT
    ROUND(
      CAST(SUM(CASE WHEN status = 'completed' THEN 1 ELSE 0 END) AS REAL)
      / COUNT(*) * 100, 1
    ) as success_rate
  FROM sync_files
  WHERE updated_at > datetime('now', '-1 hour');
"
```

### 스펙 준수 상태 점검표

| 점검 항목 | 기대 상태 (SDD 스펙) | 확인 방법 | 비정상 시 조치 |
|---|---|---|---|
| 엔진 상태 | `RUNNING` 또는 `PAUSED` | GUI 대시보드 | §3.4 참조 |
| 폴링 간격 | 5초 (`HistoryPollingConfig.intervalMs`) | 로그 확인 | 설정 확인 |
| 서킷 브레이커 | `CLOSED` | 로그 `[CircuitBreaker]` | §3.2 참조 |
| DB WAL 모드 | `journal_mode=wal` | `PRAGMA journal_mode;` | DB 재초기화 |
| stuck 파일 수 | 0개 | 위 SQL 쿼리 | §3.3 참조 |
| DLQ 대기 건수 | 정상 범위 (< 50건) | 위 SQL 쿼리 | DLQ 재시도 |
| 동기화 성공률 | 95%+ | 위 SQL 쿼리 | §3 트러블슈팅 |

> 📌 SDD 스펙 정의와 검증 체크리스트는 [10-SDD-개발방법론](./10-SDD-개발방법론.md) §8을 참조한다.
