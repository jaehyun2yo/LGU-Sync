## 1. 테스트 전략 개요

### 1.1 테스트 피라미드

```
        /‾‾‾‾‾‾‾\
       / E2E 10% \          Playwright (Electron)
      /‾‾‾‾‾‾‾‾‾‾‾\
     / 통합 20%     \        Vitest + IPC Mock
    /‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾\
   / 단위 70%          \      Vitest + RTL
  /‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾\
```

| 레벨 | 비율 | 프레임워크 | 대상 |
|------|------|-----------|------|
| 단위 테스트 | 70% | Vitest | Core 모듈 (순수 TypeScript) |
| 통합 테스트 | 20% | Vitest + React Testing Library | Main IPC, Renderer 컴포넌트 |
| E2E 테스트 | 10% | Playwright (Electron) | 전체 사용자 시나리오 |

### 1.2 테스트 프레임워크 구성

| 도구 | 용도 |
|------|------|
| **Vitest** | 단위/통합 테스트 러너, 모킹, 커버리지 |
| **React Testing Library** | Renderer 컴포넌트 렌더링 및 인터랙션 |
| **Playwright** | Electron E2E 테스트 |
| **MSW (Mock Service Worker)** | HTTP API 모킹 (LGU+, 자체웹하드) |
| **better-sqlite3** | 인메모리 SQLite (테스트용) |

### 1.3 커버리지 목표

| 레이어 | 라인 커버리지 | 브랜치 커버리지 | 비고 |

| Main | 70%+ | 60%+ | IPC, 시스템 통합 |

### 1.6 SDD 스펙 기반 테스트 도출 원칙

SDD(Specification-Driven Development)에 따라 테스트 케이스는 인터페이스 스펙에서 체계적으로 도출한다.

**테스트 도출 규칙:**

1. **인터페이스 메서드 → 테스트 그룹**: 각 인터페이스의 공개 메서드마다 `describe` 블록을 생성한다
2. **파라미터 경계값 → 테스트 케이스**: 각 파라미터의 유효/무효 경계값에서 테스트 케이스를 도출한다
3. **반환 타입 분기 → 테스트 케이스**: `Promise<T | null>`, `WResult<T>` 등 반환 타입의 각 분기마다 테스트를 작성한다
4. **에러 코드 → 테스트 케이스**: 인터페이스 명세에 정의된 에러 코드마다 에러 시나리오 테스트를 작성한다

**도출 예시:**

```typescript
// 스펙: ILGUplusClient.downloadFile(fileId: string, destPath: string): Promise<DownloadResult>
// → 도출 테스트:
//   TC-CORE-LGU-001: 정상 다운로드 (성공 케이스)
//   TC-CORE-LGU-002: fileId가 빈 문자열 (파라미터 경계값)
//   TC-CORE-LGU-003: 존재하지 않는 fileId (에러 케이스)
//   TC-CORE-LGU-004: destPath 디렉토리 없음 (에러 케이스)
//   TC-CORE-LGU-005: 네트워크 타임아웃 (에러 코드: NETWORK_TIMEOUT)
```

**커버리지 규칙:**
- 모든 SDD Level 2 인터페이스의 공개 메서드는 최소 1개의 성공/실패 테스트 필수
- SDD Level 1 타입의 Zod 검증 스키마는 유효/무효 입력 테스트 필수

> 📌 SDD 스펙 체계와 테스트 연동 규칙은 [10-SDD-개발방법론](./10-SDD-개발방법론.md) §8을 참조한다.
