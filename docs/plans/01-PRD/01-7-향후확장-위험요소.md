## 9. 향후 확장 계획

> 아래 항목들은 v2 초기 출시 범위에 포함되지 않으나, 아키텍처 설계 시 확장 가능성을 고려해야 한다.

### 9.1 양방향 동기화

- **내용**: 자체웹하드 → LGU+ 외부웹하드 방향 동기화 추가 (수정 도면 전달 자동화)
- **고려사항**: LGU+ 업로드 API 역공학 필요, 충돌 해결 전략(LWW, 수동 등) 설계 필요
- **아키텍처 영향**: `SyncDirection` enum, `ConflictResolver` 인터페이스를 미리 확장 가능하게 설계

### 9.2 파일 변경 이벤트 전파 (삭제/이동/이름변경)

- **내용**: LGU+ 웹하드에서 파일 삭제, 이동, 이름변경 시 자체웹하드에도 반영
- **고려사항**: operCode(D, MV, FMV, RN, FRN) 처리 로직 추가, 삭제의 경우 휴지통 개념 도입 검토
- **아키텍처 영향**: `SyncEventType` 확장, `EventHandler` 전략 패턴 추가

### 9.3 웹사이트 연동 강화

- **내용**: 동기화된 파일을 yjlaser_website의 주문/접수 시스템과 자동 연결
- **고려사항**: 파일명에서 업체명 추출 → 거래처 자동 매칭 → 주문 건 자동 생성
- **아키텍처 영향**: 웹사이트 API 연동 모듈 추가, Webhook 수신 기능

### 9.4 다중 외부웹하드 지원

- **내용**: LGU+ 외에 다른 웹하드 서비스 지원 (ex. 센드애니웨어, 네이버 MYBOX 등)
- **아키텍처 영향**: `IExternalWebhardApi` 인터페이스 정의, 외부웹하드도 어댑터 패턴 적용

### 9.5 모니터링 대시보드 (웹)

- **내용**: 외부에서 동기화 상태를 모니터링할 수 있는 웹 대시보드
- **고려사항**: yjlaser_website에 관리자 전용 모니터링 페이지 추가
- **아키텍처 영향**: Main 프로세스에 REST API 서버 추가 (또는 yjlaser_website에 상태 리포팅)

### 9.6 모바일 알림

- **내용**: 심각한 오류 발생 시 모바일(카카오톡, SMS 등)로 알림
- **고려사항**: Popbill API 또는 Slack Webhook 활용
- **아키텍처 영향**: 알림 인터페이스 확장, 알림 채널 어댑터 추가

## 10. 위험 요소 및 대응 방안

### 10.1 기술적 위험

| 위험 | 발생 확률 | 영향도 | 대응 방안 |
|------|-----------|--------|-----------|
| **LGU+ API 변경/차단** | 중간 | 치명적 | (1) API 호출 패턴을 최소화하여 감지 위험 축소 (2) Playwright 폴백 유지 (3) API 응답 형식 변경 감지 자동 알림 구축 (4) 대응 가능한 모듈 격리 설계 |
| **LGU+ 세션 정책 변경** | 중간 | 높음 | (1) 세션 갱신 주기를 보수적으로 설정 (10분) (2) 복수 인증 전략 (HTTP + Playwright) 유지 (3) 세션 상태 모니터링 강화 |
| **Electron 번들 크기** | 높음 | 낮음 | (1) Playwright Chromium과 Electron Chromium 공유 검토 (2) 설치 파일 500MB 이하 목표 (3) 델타 업데이트로 패치 크기 최소화 |
| **SQLite 동시성 한계** | 낮음 | 중간 | (1) WAL 모드 사용 (2) 쓰기 직렬화 (3) 읽기/쓰기 커넥션 분리 |
| **메모리 누수 (장시간 운영)** | 중간 | 중간 | (1) 24시간 이상 운영 테스트 필수 (2) 메모리 사용량 모니터링 + 임계치 초과 시 자동 재시작 (3) 이벤트 리스너 정리 철저 |
| **파일명 인코딩 이슈** | 중간 | 중간 | (1) 모든 파일명을 UTF-8로 정규화 (2) Windows 경로 길이 제한(260자) 대응 (3) 특수문자 치환 규칙 정의 |

### 10.2 비즈니스 위험

| 위험 | 발생 확률 | 영향도 | 대응 방안 |
|------|-----------|--------|-----------|
| **LGU+ 웹하드 서비스 종료** | 낮음 | 치명적 | (1) 외부웹하드 API를 인터페이스로 추상화 (2) 다른 웹하드 서비스 어댑터 추가가 용이한 구조 (3) 웹사이트 직접 업로드 기능을 대체 경로로 확보 |
| **거래처 웹하드 사용 거부** | 낮음 | 높음 | (1) yjlaser_website 직접 업로드 기능 병행 개발 |
| **내부웹하드 API 지연 제공** | 높음 | 중간 | (1) Mock 어댑터로 개발/테스트 진행 (2) v1 API 호환 어댑터 우선 구현 (3) 인터페이스 확정 후 실 어댑터로 교체 |
| **사용자 적응 실패** | 낮음 | 중간 | (1) 최소한의 UI로 단순화 (2) 초기 설정 마법사 제공 (3) 기본값을 최적으로 설정하여 조작 최소화 |

### 10.3 운영 위험

| 위험 | 발생 확률 | 영향도 | 대응 방안 |
|------|-----------|--------|-----------|
| **네트워크 장시간 단절** | 중간 | 높음 | (1) 오프라인 큐잉: 단절 중 이벤트를 큐에 보관 (2) 네트워크 복구 시 자동 일괄 처리 (3) 단절 1시간 초과 시 사용자 알림 |
| **PC 비정상 종료 (정전 등)** | 중간 | 중간 | (1) 체크포인트 기반 이어받기 (2) SQLite WAL 모드로 DB 무결성 보장 (3) 부분 다운로드 파일 자동 정리 |
| **디스크 공간 부족** | 중간 | 중간 | (1) 임시 파일 자동 정리 (업로드 완료 후 삭제) (2) 디스크 잔여 공간 주기적 검사 (3) 임계치(1GB) 이하 시 경고 + 동기화 일시중지 |
| **Windows 업데이트 재시작** | 높음 | 낮음 | (1) 자동 시작 기능으로 재부팅 후 자동 복귀 (2) 체크포인트로 이어받기 |

### 10.4 위험 완화 우선순위

```
[치명적 영향]
  1. LGU+ API 변경/차단 → 인터페이스 격리 + 이중 인증 전략
  2. LGU+ 웹하드 서비스 종료 → 외부웹하드 추상화 + 대체 경로

[높은 영향]
  3. LGU+ 세션 정책 변경 → 보수적 갱신 주기 + 복수 전략
  4. 네트워크 장시간 단절 → 오프라인 큐잉 + 자동 복구
  5. 내부웹하드 API 지연 → Mock 어댑터 선행 개발

[중간 영향]
  6. 메모리 누수 → 장시간 운영 테스트
  7. 파일명 인코딩 → 정규화 규칙 정의
  8. SQLite 동시성 → WAL + 직렬화
  9. PC 비정상 종료 → 체크포인트 + WAL
  10. 디스크 공간 부족 → 자동 정리 + 모니터링
```
