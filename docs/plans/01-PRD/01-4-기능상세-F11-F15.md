# 기능별 상세 요구사항 (F-11 ~ F-15)

> 이 문서는 PRD 전체 문서를 분할한 것입니다. 전체 목차는 [인덱스](../01-PRD-제품요구사항정의서.md)를 참조하세요.

### F-11. 로그 뷰어

#### 설명

프로그램의 동작 로그를 실시간으로 확인하고 검색할 수 있는 뷰어를 제공한다.

#### 사용자 스토리

- 사용자로서, 오류 발생 시 로그를 확인하여 원인을 파악하기를 원한다.
- 사용자로서, 특정 파일의 동기화 이력을 검색하기를 원한다.
- 사용자로서, 로그를 파일로 내보내어 개발자에게 전달하기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-11-AC01 | 실시간 로그 스트림을 표시한다 (자동 스크롤) | 동기화 진행 중 로그가 실시간으로 추가되는지 확인 |
| F-11-AC02 | 로그 레벨 필터: DEBUG, INFO, WARN, ERROR | 각 레벨 필터 적용 후 해당 레벨만 표시되는지 확인 |
| F-11-AC03 | 텍스트 검색으로 로그를 필터링할 수 있다 | 파일명 검색 시 관련 로그만 표시되는지 확인 |
| F-11-AC04 | 날짜/시간 범위 필터를 제공한다 | 특정 날짜 범위 선택 후 해당 기간 로그만 표시 확인 |
| F-11-AC05 | 로그를 텍스트 파일로 내보내기할 수 있다 | 내보내기 후 파일 내용이 화면 표시와 일치하는지 확인 |
| F-11-AC06 | 오래된 로그를 자동 삭제한다 (기본 30일 보관) | 30일 이상 된 로그가 자동 정리되는지 확인 |
| F-11-AC07 | 로그 항목 클릭 시 상세 정보(스택 트레이스, 컨텍스트 등)를 펼쳐 표시한다 | 오류 로그 클릭 후 상세 정보 표시 확인 |

#### 엣지 케이스

- 로그가 초당 100건 이상 발생하는 대량 동기화 상황 (UI 성능)
- 매우 긴 로그 메시지
- 로그 DB 파일 크기가 매우 큰 경우 (1GB 이상)

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 로그 DB 조회 실패 | "로그를 불러올 수 없습니다" 메시지 표시 |
| 내보내기 경로에 쓰기 권한 없음 | 다른 경로 선택 유도 |
| 로그 DB 파일 손상 | 새 DB 파일 생성 + 이전 DB 백업 |

### F-12. 파일 탐색기 UI

#### 설명

외부웹하드의 파일/폴더 구조를 GUI에서 탐색하고 동기화 상태를 확인할 수 있는 파일 탐색기를 제공한다.

#### 사용자 스토리

- 사용자로서, 외부웹하드의 폴더 구조와 파일 목록을 GUI에서 탐색하기를 원한다.
- 사용자로서, 각 파일의 동기화 상태(완료/대기/실패)를 확인하기를 원한다.
- 사용자로서, 특정 파일을 선택하여 수동으로 동기화를 시작하기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-12-AC01 | 좌측에 폴더 트리, 우측에 파일 목록을 표시하는 2패널 레이아웃을 제공한다 | 폴더 클릭 시 해당 폴더의 파일이 우측에 표시되는지 확인 |
| F-12-AC02 | 파일별 동기화 상태 아이콘을 표시한다: 완료(체크), 대기(시계), 진행중(화살표), 실패(X) | 각 상태의 파일이 올바른 아이콘으로 표시되는지 확인 |
| F-12-AC03 | 파일명, 크기, 수정일, 동기화 시간을 컬럼으로 표시한다 | 모든 컬럼에 올바른 데이터가 표시되는지 확인 |
| F-12-AC04 | 파일 선택 후 "지금 동기화" 버튼으로 해당 파일만 즉시 동기화할 수 있다 | 선택 파일이 즉시 동기화 큐에 추가되는지 확인 |
| F-12-AC05 | 컬럼 헤더 클릭으로 정렬할 수 있다 (이름, 크기, 날짜 등) | 각 컬럼 정렬이 올바르게 동작하는지 확인 |
| F-12-AC06 | 파일명 검색 기능을 제공한다 | 검색어 입력 시 일치하는 파일만 표시되는지 확인 |

#### 엣지 케이스

- 폴더에 파일이 1,000개 이상인 경우 (가상 스크롤 필요)
- 폴더 트리 로딩 중 네트워크 오류
- 파일 목록 표시 중 동기화 상태 변경

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 폴더 내용 조회 실패 | "폴더를 불러올 수 없습니다" 메시지 + 재시도 버튼 |
| 수동 동기화 실패 | 해당 파일에 실패 상태 표시 + 재시도 버튼 |

### F-13. 알림 시스템

#### 설명

인앱 알림과 Windows 토스트 알림으로 중요 이벤트를 사용자에게 전달한다.

#### 사용자 스토리

- 사용자로서, 동기화 오류 발생 시 즉시 알림을 받기를 원한다.
- 사용자로서, 새 파일이 동기화 완료되면 알림을 받기를 원한다 (선택적).
- 사용자로서, 알림의 유형별로 켜기/끄기를 설정하기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-13-AC01 | 인앱 알림: 프로그램 내 알림 센터에 이벤트가 시간순으로 표시된다 | 이벤트 발생 후 알림 센터에 추가되는지 확인 |
| F-13-AC02 | Windows 토스트 알림: 오류 발생 시 Windows 알림 센터에 토스트가 표시된다 | 오류 시뮬레이션 후 토스트 알림 확인 |
| F-13-AC03 | 토스트 알림 클릭 시 프로그램이 열리고 관련 화면으로 이동한다 | 오류 알림 클릭 후 로그 뷰어 표시 확인 |
| F-13-AC04 | 알림 유형별 설정: 오류 알림(기본 ON), 동기화 완료 알림(기본 OFF), 세션 만료 알림(기본 ON) | 각 설정 토글 후 해당 유형 알림 발생/미발생 확인 |
| F-13-AC05 | 인앱 알림에 읽음/안읽음 상태가 있고, 배지(미읽음 수)를 표시한다 | 알림 확인 후 배지 수 감소 확인 |
| F-13-AC06 | 오래된 알림을 자동 정리한다 (기본 7일 보관) | 7일 이상 된 알림이 자동 삭제되는지 확인 |

#### 엣지 케이스

- 단시간에 대량 알림 발생 (100건 이상)
- Windows 집중 모드/방해 금지 모드에서 토스트 알림
- 알림 센터가 가득 찬 상태

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 토스트 알림 API 실패 | 인앱 알림으로 폴백 |
| 대량 알림으로 인한 성능 저하 | 동일 유형 알림 그룹핑 (예: "5건의 파일 동기화 완료") |

### F-14. 동기화 통계 및 차트

#### 설명

일별/주별/월별 동기화 통계를 차트로 시각화한다.

#### 사용자 스토리

- 사용자로서, 일별 동기화 건수를 차트로 확인하기를 원한다.
- 사용자로서, 업체별 동기화 현황을 파악하기를 원한다.
- 사용자로서, 전송 용량 추이를 확인하기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-14-AC01 | 일별 동기화 건수 막대 차트를 표시한다 (최근 30일) | 차트 데이터가 SQLite 이력과 일치하는지 확인 |
| F-14-AC02 | 일별 전송 용량 라인 차트를 표시한다 (최근 30일) | 차트 수치가 실제 전송 용량과 일치하는지 확인 |
| F-14-AC03 | 폴더(업체)별 동기화 건수 도넛/파이 차트를 표시한다 | 각 업체의 비율이 올바르게 표시되는지 확인 |
| F-14-AC04 | 기간 선택 (오늘, 7일, 30일, 전체)을 제공한다 | 기간 변경 시 차트가 갱신되는지 확인 |
| F-14-AC05 | 성공률(성공 건수 / 전체 건수) 지표를 표시한다 | 실제 성공률과 표시 수치 일치 확인 |

#### 엣지 케이스

- 통계 데이터가 없는 초기 상태
- 매우 큰 데이터셋에서 차트 렌더링 성능

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 통계 데이터 조회 실패 | "통계를 불러올 수 없습니다" 메시지 |
| 차트 렌더링 오류 | 테이블 형태로 폴백 표시 |

### F-15. 체크포인트 기반 이어받기

#### 설명

프로그램 종료/크래시 후 재시작 시 마지막 처리 지점부터 이어서 동기화한다.

#### 사용자 스토리

- 사용자로서, 프로그램이 비정상 종료되어도 이미 처리된 파일을 다시 처리하지 않기를 원한다.
- 사용자로서, 재시작 후 누락 없이 나머지 파일부터 이어서 동기화하기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-15-AC01 | 처리 완료된 이벤트의 historyNo를 SQLite에 원자적으로 기록한다 | DB에 체크포인트가 기록되는지 확인 |
| F-15-AC02 | 프로그램 재시작 시 마지막 체크포인트부터 폴링을 재개한다 | 재시작 후 마지막 처리 이후 이벤트만 처리되는지 확인 |
| F-15-AC03 | 전체 동기화 중단 시 진행 상황(처리된 폴더/파일 목록)을 저장하여 이어받기한다 | 전체 동기화 50% 중단 → 재시작 → 나머지 50% 처리 확인 |
| F-15-AC04 | 3단계 중복 검사: historyNo 비교 → 메모리 캐시 → SQLite DB 조회 | 동일 이벤트가 중복 처리되지 않음을 확인 |
| F-15-AC05 | 비정상 종료 시 처리 중이던 파일(부분 다운로드)을 정리한다 | 재시작 후 불완전 파일이 삭제 + 재다운로드되는지 확인 |

#### 엣지 케이스

- SQLite DB 파일 손상
- 체크포인트 기록 직전 프로세스 Kill
- 재시작 사이에 LGU+ 웹하드에서 대량 변경 발생

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| SQLite DB 손상 | 새 DB 생성 + 전체 동기화 자동 시작 + 사용자 알림 |
| 체크포인트 불일치 | 보수적으로 이전 체크포인트부터 재처리 (중복 허용) |
