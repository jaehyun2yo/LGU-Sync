## 5. 기능별 상세 요구사항

### F-01. LGU+ 웹하드 로그인 및 세션 관리

#### 설명

LGU+ 웹하드(`only.webhard.co.kr`)에 관리자 계정으로 로그인하여 세션 쿠키를 획득하고, 세션이 만료되지 않도록 자동 갱신한다. LGU+ 웹하드는 공식 API가 없으므로 역공학으로 분석한 비공식 API를 사용한다.

#### 사용자 스토리

- 사용자로서, 프로그램에 LGU+ 웹하드 아이디와 비밀번호를 한 번만 입력하면 이후 자동으로 로그인이 유지되기를 원한다.
- 사용자로서, 세션이 만료되었을 때 자동으로 재로그인되어 동기화가 중단되지 않기를 원한다.
- 사용자로서, 로그인 실패 시 알림을 받고 비밀번호를 재입력할 수 있기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-01-AC01 | HTTP API 직접 로그인(`POST /login-process`)으로 세션 쿠키를 획득할 수 있다 | 로그인 후 폴더 목록 API 호출이 성공하는지 확인 |
| F-01-AC02 | Playwright 브라우저 로그인이 HTTP 로그인 실패 시 폴백으로 작동한다 | HTTP 로그인을 의도적으로 실패시킨 후 브라우저 로그인이 시도되는지 확인 |
| F-01-AC03 | 15분 간격으로 세션 유효성을 검증하고, 만료 시 자동 재로그인한다 | 15분 타이머 동작 및 세션 만료 후 재로그인 성공 확인 |
| F-01-AC04 | API 호출 중 세션 만료 감지(302 리다이렉트, 401, RESULT_CODE 9999) 시 즉시 재로그인 후 API를 재시도한다 | 세션 만료 시뮬레이션 후 자동 복구 확인 |
| F-01-AC05 | 자격증명을 안전하게 저장한다 (평문 저장 금지) | 저장소에 암호화된 자격증명이 기록되는지 확인 |
| F-01-AC06 | 로그인 실패가 3회 연속 발생하면 사용자에게 알림을 보내고 재입력을 요청한다 | 3회 실패 후 알림 및 설정 화면 이동 확인 |
| F-01-AC07 | CAPTCHA 감지 시 사용자에게 수동 로그인을 안내한다 | CAPTCHA 감지 후 알림 내용 확인 |

#### 엣지 케이스

- LGU+ 웹하드 서버 점검 시간(새벽 시간대) 동안 로그인 불가 상황 처리
- 비밀번호 변경 후 기존 세션 무효화 감지
- 동시에 여러 곳에서 같은 계정 로그인 시 세션 충돌
- LGU+ 웹하드 로그인 페이지 HTML 구조 변경 시 Playwright 셀렉터 실패

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 네트워크 타임아웃 | 5초, 10초, 30초 간격으로 3회 재시도 |
| 잘못된 자격증명 | 사용자에게 알림 + 설정 화면으로 유도 |
| CAPTCHA 출현 | 사용자에게 수동 로그인 안내 (브라우저 창 표시) |
| LGU+ 서버 다운 | 지수 백오프로 재시도 (최대 30분 간격), 사용자 알림 |
| 쿠키 파싱 실패 | 로그 기록 + Playwright 폴백 시도 |

### F-02. 실시간 파일 변경 감지

#### 설명

LGU+ 웹하드에 새로운 파일이 업로드되면 이를 실시간으로 감지한다. v1에서 사용한 Upload History API 폴링 방식을 기반으로 하되, 감지 신뢰성을 개선한다.

#### 사용자 스토리

- 사용자로서, 거래처가 외부웹하드에 파일을 올리면 1분 이내에 자동으로 동기화가 시작되기를 원한다.
- 사용자로서, 어떤 파일도 누락 없이 모두 동기화되기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-02-AC01 | Upload History API를 5초 간격으로 폴링하여 신규 파일 업로드(operCode: UP)를 감지한다 | 외부웹하드에 파일 업로드 후 5초 이내 이벤트 생성 확인 |
| F-02-AC02 | historyNo 기반 체크포인트로 중복 감지를 방지한다 | 동일 historyNo가 2회 이상 처리되지 않음을 확인 |
| F-02-AC03 | 감지된 이벤트를 SyncQueue에 추가하여 순서대로 처리한다 | 이벤트 발생 순서와 처리 순서가 일치하는지 확인 |
| F-02-AC04 | 이벤트 감지 후 500ms 이내 즉시 재폴링하여 연속 업로드를 놓치지 않는다 (최대 3회 연속) | 연속 3개 파일 업로드 시 모두 감지되는지 확인 |
| F-02-AC05 | 폴링 실패 시 자동 재시도하며, 실패가 지속되면 간격을 늘린다 (지수 백오프) | 네트워크 단절 시뮬레이션 후 재시도 패턴 확인 |
| F-02-AC06 | 보조 감지로 주기적 스냅샷 비교(10분 간격)를 수행하여 폴링이 놓친 변경을 보완한다 | 폴링 비활성 상태에서 10분 후 스냅샷 비교로 파일 감지 확인 |
| F-02-AC07 | 동기화 대상으로 선택된 폴더의 파일만 감지한다 (선택되지 않은 폴더는 무시) | 비대상 폴더의 파일 업로드 시 이벤트가 생성되지 않음을 확인 |
| F-02-AC08 | 프로그램 재시작 후에도 마지막 체크포인트부터 이어서 감지한다 | 프로그램 종료 → 외부 파일 업로드 → 재시작 → 해당 파일 감지 확인 |

#### 엣지 케이스

- 같은 파일명으로 다른 폴더에 동시 업로드
- 대량 업로드 (100개 이상 파일을 한 번에 업로드)
- 업로드 직후 삭제 (감지 전에 삭제됨)
- 날짜 경계(자정) 시점의 이력 조회
- 폴링 중 세션 만료
- LGU+ 서버 측 Upload History API 페이지네이션 변동

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| API 응답 타임아웃 | 5초 후 재시도, 3회 실패 시 30초 대기 후 재시도 |
| 세션 만료 (RESULT_CODE: 9999) | 자동 재로그인 후 폴링 재개 |
| 빈 응답 반환 | 정상으로 간주하고 다음 폴링 진행 |
| historyNo 역전/중복 | 메모리 캐시 + SQLite 이중 검증으로 필터링 |
| LGU+ API 형식 변경 | 파싱 오류 로그 + 사용자 알림 |

### F-03. 전체 동기화 (초기/수동)

#### 설명

프로그램 최초 실행 시 또는 사용자가 수동 요청 시, 선택된 폴더의 전체 파일을 스캔하여 아직 동기화되지 않은 파일을 모두 동기화한다. 델타 비교(파일명 + 크기)로 이미 동기화된 파일은 건너뛴다.

#### 사용자 스토리

- 사용자로서, 프로그램을 처음 설치하면 기존 파일을 모두 자동 동기화하기를 원한다.
- 사용자로서, "전체 동기화" 버튼을 눌러 누락된 파일을 수동으로 보충할 수 있기를 원한다.
- 사용자로서, 전체 동기화 진행률(스캔 중, 비교 중, 다운로드 중, 업로드 중)을 실시간으로 확인하기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-03-AC01 | 선택된 폴더의 전체 파일/폴더 구조를 재귀적으로 스캔한다 | 스캔 완료 후 파일 수가 외부웹하드 실제 파일 수와 일치하는지 확인 |
| F-03-AC02 | 로컬 상태(SQLite)와 비교하여 신규/변경 파일만 동기화 대상으로 선별한다 | 이미 동기화된 파일이 재다운로드되지 않음을 확인 |
| F-03-AC03 | 전체 동기화 진행률(단계별: 스캔 → 비교 → 다운로드 → 업로드)을 UI에 실시간 표시한다 | UI에 단계, 진행률 퍼센트, 현재 파일명이 표시되는지 확인 |
| F-03-AC04 | 전체 동기화 중에도 실시간 감지가 병행된다 (새 파일 놓치지 않음) | 전체 동기화 중 새 파일 업로드 시 해당 파일도 동기화되는지 확인 |
| F-03-AC05 | 전체 동기화를 중간에 취소할 수 있으며, 재시작 시 이어받기가 가능하다 | 50% 진행 후 취소 → 재시작 시 나머지 50%만 처리하는지 확인 |
| F-03-AC06 | 빈 폴더도 자체웹하드에 동일 구조로 생성한다 | 외부웹하드의 빈 폴더가 자체웹하드에도 존재하는지 확인 |
| F-03-AC07 | 전체 동기화 시작 시 startHistoryNo를 기록하여 갭 없는 동기화를 보장한다 | 전체 동기화 시작~실시간 감지 시작 사이에 올라온 파일이 누락 없이 처리되는지 확인 |

#### 엣지 케이스

- 매우 큰 폴더 (10,000개 이상 파일)
- 전체 동기화 중 네트워크 단절
- 전체 동기화 중 LGU+ 웹하드에서 파일 삭제
- 동일 파일명이 다른 폴더에 존재
- 특수문자가 포함된 파일명/폴더명
- 0바이트 파일

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 스캔 중 API 실패 | 실패한 폴더 건너뛰기 + 나중에 재시도 |
| 디스크 공간 부족 | 사전 검사 + 용량 부족 시 알림 후 중단 |
| 다운로드 실패 (개별 파일) | 실패 파일 DLQ에 추가 + 나머지 계속 진행 |
| 업로드 실패 (개별 파일) | 3회 재시도 후 실패 파일 기록 + 나머지 계속 진행 |

### F-04. 파일 다운로드

#### 설명

LGU+ 웹하드에서 파일을 다운로드한다. 2단계 프로세스를 따른다: (1) 다운로드 URL 정보 획득, (2) 실제 파일 다운로드.

#### 사용자 스토리

- 사용자로서, 외부웹하드의 파일이 빠르게 다운로드되기를 원한다.
- 사용자로서, 대용량 파일도 안정적으로 다운로드되기를 원한다.
- 사용자로서, 다운로드 중인 파일의 진행 상황을 알 수 있기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-04-AC01 | `GET /downloads/{fileId}/server?fileStatus=1`로 다운로드 URL 정보를 획득한다 | API 응답에 session, nonce, url 등 필수 필드가 포함되는지 확인 |
| F-04-AC02 | 획득한 정보로 다운로드 URL을 구성하여 파일을 다운로드한다 | 다운로드된 파일이 원본과 동일한지 크기 비교 확인 |
| F-04-AC03 | 파일 크기에 따른 동적 타임아웃을 적용한다 (10MB 미만: 2분, 100MB 미만: 5분, 1GB 미만: 15분, 1GB 이상: 30분) | 대용량 파일 다운로드 시 타임아웃이 적절히 설정되는지 확인 |
| F-04-AC04 | 배치 다운로드 시 최대 동시 다운로드 수를 제한한다 (기본값: 5) | 6개 이상 파일 동시 요청 시 5개만 병렬 진행하는지 확인 |
| F-04-AC05 | 다운로드 실패 시 최대 3회 재시도한다 (1초 → 3초 → 10초 간격) | 네트워크 오류 시뮬레이션 후 재시도 횟수 및 간격 확인 |
| F-04-AC06 | 서킷 브레이커로 연속 실패 시 일시적으로 다운로드를 중단한다 (실패율 40% 초과 시 10초 차단) | 연속 실패 시 차단 → 복구 패턴 확인 |
| F-04-AC07 | 다운로드된 파일명에서 특수문자를 안전하게 처리한다 (경로 구분자, 제어문자 등 제거) | 특수문자 파일명 다운로드 후 파일 시스템에 정상 저장 확인 |
| F-04-AC08 | 파일별 다운로드 진행률(바이트 기반)을 UI에 전달한다 | 대용량 파일 다운로드 시 진행률 업데이트가 UI에 반영되는지 확인 |

#### 엣지 케이스

- 다운로드 중 LGU+ 서버에서 파일 삭제
- 다운로드 URL 만료 (시간 초과)
- 0바이트 파일
- 파일명이 매우 긴 경우 (255자 초과)
- 한글/일본어/중국어 파일명
- 동시 다운로드 중 하나만 실패

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 다운로드 URL 획득 실패 (404) | 파일 삭제로 판단, 건너뛰기 + 로그 기록 |
| 다운로드 URL 획득 실패 (500) | 3회 재시도 후 DLQ에 추가 |
| 다운로드 중 네트워크 단절 | 부분 파일 삭제 + 재시도 |
| 다운로드 파일 크기 불일치 | 파일 삭제 + 재다운로드 |
| 로컬 디스크 공간 부족 | 즉시 중단 + 사용자 알림 |

### F-05. 자체웹하드 업로드

#### 설명

다운로드된 파일을 자체웹하드(yjlaser_website API)에 업로드한다. 자체웹하드 API는 추후 변경될 수 있으므로 인터페이스/어댑터 패턴으로 교체 가능하게 설계한다.

#### 사용자 스토리

- 사용자로서, 다운로드된 파일이 자동으로 자체웹하드에 업로드되어 웹사이트에서 확인할 수 있기를 원한다.
- 사용자로서, 외부웹하드의 폴더 구조가 자체웹하드에 그대로 보존되기를 원한다.

#### 인수 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| F-05-AC01 | 다운로드 완료된 파일을 자체웹하드 API로 업로드한다 | 업로드 후 자체웹하드 웹 UI에서 파일 확인 |

| F-05-AC03 | 외부웹하드의 폴더 경로를 자체웹하드에 동일하게 재현한다 (폴더 자동 생성) | `/올리기전용/업체A/서브폴더/` 경로가 그대로 생성되는지 확인 |
| F-05-AC04 | 100MB 이상 파일은 멀티파트 업로드를 사용한다 | 150MB 파일 업로드 시 멀티파트 API가 호출되는지 확인 |
| F-05-AC05 | 중복 파일(체크섬 동일)은 업로드를 건너뛴다 | 동일 파일 재업로드 시도 시 스킵되는지 확인 |
| F-05-AC06 | 업로드 실패 시 3회 재시도한다 (지수 백오프: 1초 → 2초 → 4초) | 서버 오류 시뮬레이션 후 재시도 패턴 확인 |
| F-05-AC07 | 내부웹하드 API 인터페이스가 정의되어 있어, 다른 구현체로 교체 가능하다 | 인터페이스 정의서 검토 + Mock 구현체 테스트 |
| F-05-AC08 | 폴더 404 오류 시 폴더 재생성 후 업로드를 재시도한다 (타이밍 이슈 대응) | 폴더 생성 직후 업로드 시도 시 정상 처리 확인 |

#### 엣지 케이스

- 자체웹하드 서버 다운/점검 시
- 업로드 중 자체웹하드 서버 재시작
- Presigned URL 만료
- 동일 파일명이 이미 자체웹하드에 존재 (체크섬은 다름)
- 매우 깊은 폴더 경로 (10단계 이상)

#### 오류 시나리오

| 오류 | 처리 방안 |
|------|-----------|
| 자체웹하드 서버 연결 불가 | 30초 간격으로 재시도, 5분 후에도 불가하면 사용자 알림 |
| 업로드 타임아웃 | 재시도 (멀티파트의 경우 실패 파트만 재업로드) |
| 인증 실패 (API Key 오류) | 사용자에게 API Key 확인 알림 |
| 폴더 생성 실패 | 상위 폴더부터 순차 생성 재시도 |
| 체크섬 불일치 | 파일 재다운로드 후 재업로드 |
