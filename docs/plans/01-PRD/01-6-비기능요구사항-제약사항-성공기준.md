## 6. 비기능 요구사항

### 6.1 성능

| 항목 | 요구사항 | 측정 방법 |
|------|----------|-----------|
| **파일 감지 지연** | 외부웹하드에 파일 업로드 후 60초 이내 감지 | 타임스탬프 비교 |
| **단일 파일 동기화** | 10MB 이하 파일: 감지 후 2분 이내 자체웹하드 업로드 완료 | 타임스탬프 비교 |
| **대량 파일 처리** | 100개 파일 동시 업로드 시 30분 이내 전체 완료 | 배치 테스트 |
| **UI 응답성** | 모든 UI 인터랙션 100ms 이내 응답 | 사용자 클릭 → 화면 변경 시간 측정 |
| **메모리 사용량** | 유휴 시 200MB 이하, 동기화 시 500MB 이하 | Task Manager 모니터링 |
| **CPU 사용률** | 유휴 시 1% 이하, 폴링 시 5% 이하 | Task Manager 모니터링 |
| **시작 시간** | 프로그램 실행 후 10초 이내 트레이 아이콘 표시 + 로그인 시작 | 타이머 측정 |

### 6.2 신뢰성

| 항목 | 요구사항 | 측정 방법 |
|------|----------|-----------|
| **동기화 완전성** | 24시간 무중단 운영 시 파일 감지 누락 0건 | 외부웹하드 파일 수 vs 자체웹하드 파일 수 비교 |
| **데이터 무결성** | 동기화된 파일의 100%가 원본과 동일 | 체크섬(MD5) 비교 |
| **자동 복구율** | 일시적 오류의 95% 이상이 자동 복구됨 | 오류 로그 vs 자동 복구 로그 비율 |
| **가동 시간** | 월간 99% 이상 가동 (PC 부팅 시간 기준) | 가동/비가동 시간 로그 |
| **데이터 보존** | 프로세스 크래시 시 처리 완료된 데이터 손실 0건 | 크래시 시뮬레이션 후 DB 무결성 검증 |

### 6.3 보안

| 항목 | 요구사항 |
|------|----------|
| **자격증명 저장** | LGU+ 비밀번호, API Key를 OS 자격증명 관리자 또는 AES-256 암호화로 저장 |
| **통신 암호화** | 모든 외부 통신은 HTTPS(TLS 1.2 이상) |
| **로그 민감정보** | 로그에 비밀번호, API Key, 세션 쿠키를 기록하지 않음 |
| **파일 시스템 접근** | 지정된 다운로드 디렉토리와 프로그램 데이터 디렉토리만 접근 |
| **자동 업데이트 검증** | 업데이트 파일의 디지털 서명 또는 체크섬 검증 |

### 6.4 사용성

| 항목 | 요구사항 |
|------|----------|
| **초기 설정** | 최초 실행 시 설정 마법사(Wizard)로 5분 이내 설정 완료 가능 |
| **일상 운영** | 설정 완료 후 별도 조작 없이 자동 운영 |
| **언어** | 한국어 전용 (다국어 미지원) |
| **접근성** | 기본 Windows 고대비 모드 지원, 최소 폰트 크기 12px |
| **오류 메시지** | 모든 오류 메시지는 한국어로, 비개발자가 이해할 수 있는 문장으로 표시 |
| **도움말** | 각 설정 항목에 툴팁 설명 제공 |

### 6.5 유지보수성

| 항목 | 요구사항 |
|------|----------|
| **아키텍처** | 3계층 분리: Renderer(React) → Main(Electron IPC) → Core(비즈니스 로직). Core는 Electron 의존성 없이 단독 테스트 가능 |
| **테스트 커버리지** | Core 계층 80% 이상 단위 테스트 커버리지 |
| **코드 품질** | TypeScript strict 모드, ESLint, Prettier 적용 |
| **의존성 주입** | 모든 외부 의존성은 인터페이스로 추상화 + 생성자 주입 |
| **문서화** | 모든 public API에 JSDoc 주석 |
| **로그** | Winston 기반 구조화 로깅 (JSON 형식), 콘텍스트 객체 필수 |

## 7. 제약사항

### 7.1 기술적 제약사항

| 제약 | 설명 | 영향 |
|------|------|------|
| **LGU+ 비공식 API** | LGU+ 웹하드는 공식 API를 제공하지 않음. 역공학으로 분석한 HTTP API를 사용함 | API 형식이 예고 없이 변경될 수 있으며, 이 경우 긴급 패치가 필요함 |
| **쿠키 기반 인증** | LGU+ 웹하드는 세션 쿠키 기반 인증만 지원. OAuth 등 표준 인증 미지원 | 세션 만료 관리, 쿠키 파싱 등 추가 로직 필요 |
| **폴링 방식 감지** | LGU+ 웹하드는 WebSocket/Webhook을 지원하지 않음. Upload History API 폴링만 가능 | 실시간성에 한계 (최소 5초 지연), 서버 부하 고려 필요 |
| **단일 계정** | LGU+ 웹하드 관리자 계정 1개만 사용 | 다중 세션/병렬 접근 제한 |
| **Windows 전용** | 사용 환경이 Windows 데스크톱으로 한정됨 | Electron으로 크로스 플랫폼 가능하나 Windows만 지원/테스트 |
| **Playwright 의존** | CAPTCHA 등 HTTP 로그인 실패 시 Playwright 브라우저 자동화가 필요 | Chromium 브라우저 번들 (추가 용량 약 200MB) |
| **내부웹하드 API 미확정** | 자체웹하드 API 사양이 확정되지 않았으며 추후 변경 가능 | 인터페이스/어댑터 패턴으로 교체 가능하게 설계 필수 |

### 7.2 비즈니스 제약사항

| 제약 | 설명 |
|------|------|
| **단일 사용자** | 유진레이저 내부에서 1명만 사용. 다중 사용자 고려 불필요 |
| **단방향 전용** | LGU+ → 자체웹하드 방향만. 역방향은 별도 프로세스(수동)로 처리 |
| **파일 유형 무제한** | 도면 파일(DXF, DWG) 외에도 모든 파일 유형을 동기화해야 함 (PDF, AI, ZIP 등) |
| **폴더 구조 보존** | 외부웹하드의 폴더 구조를 자체웹하드에 그대로 재현해야 함 |
| **무중단 운영** | 업무 시간(09:00~18:00) 중에는 동기화가 중단되면 안 됨 |

### 7.3 인프라 제약사항

| 제약 | 설명 |
|------|------|
| **네트워크** | 사무실 유선 네트워크 (100Mbps 이상). 간헐적 불안정 가능 |
| **PC 사양** | Windows 10 Home, 일반 사무용 PC (RAM 8GB+, SSD) |
| **디스크 용량** | 임시 다운로드용으로 최소 10GB 여유 공간 필요 |
| **방화벽** | `only.webhard.co.kr`, `whfile1.webhard.co.kr`, 자체웹하드 URL에 대한 HTTPS 아웃바운드 허용 필요 |

## 8. 성공 기준

### 8.1 정량적 성공 기준

| 기준 | 목표값 | 측정 방법 | 달성 기한 |
|------|--------|-----------|-----------|
| **파일 감지율** | 100% (누락 0건) | 외부웹하드에 100개 파일 업로드 → 100개 모두 자체웹하드에 존재 확인 | 출시 전 |
| **동기화 정확도** | 100% (체크섬 일치) | 동기화된 파일 100개 무작위 샘플링 → 원본과 MD5 비교 | 출시 전 |
| **평균 동기화 지연** | 60초 이내 | 업로드 시간 ~ 자체웹하드 등록 시간 차이 (10MB 이하 파일 기준) | 출시 전 |
| **자동 복구 성공률** | 95% 이상 | 일시적 오류(네트워크, 세션) 중 자동 복구된 비율 | 출시 후 1개월 |
| **프로그램 가동률** | 99% (업무 시간 기준) | 업무 시간 중 정상 운영 시간 비율 (월간 측정) | 출시 후 1개월 |
| **사용자 개입 빈도** | 주 1회 이하 | 사용자가 수동으로 조치를 취해야 하는 횟수 | 출시 후 1개월 |
| **UI 응답 시간** | 100ms 이내 | 버튼 클릭 → 화면 변경 지연 시간 | 출시 전 |

### 8.2 정성적 성공 기준

| 기준 | 목표 | 검증 방법 |
|------|------|-----------|
| **사용 편의성** | 비개발자가 도움 없이 설정/운영 가능 | 실 사용자 테스트 |
| **안정성 체감** | "알아서 잘 동기화된다"는 사용자 인식 | 1개월 운영 후 피드백 |
| **오류 대응** | 대부분의 오류가 자동 복구되어 사용자가 인지하지 못함 | 오류 로그 vs 사용자 보고 비교 |
| **v1 대비 개선** | v1의 모든 알려진 문제점이 해결됨 | v1 이슈 목록 대비 체크 |
