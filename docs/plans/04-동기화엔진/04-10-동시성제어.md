### 10.1 동기화 엔진 싱글 인스턴스 보장

같은 시스템에서 동기화 엔진이 중복 실행되는 것을 방지한다.

```
┌──────────────────────────────────────────────────────────────┐
│                  싱글 인스턴스 보장 메커니즘                    │
└──────────────────────────────────────────────────────────────┘

  프로그램 시작
       │
       ▼
  ┌─────────────────────────┐
  │ Lock 파일 확인           │
  │ ./sync-engine.lock      │
  └────────┬────────────────┘
           │
     ┌─────┴──────┐
     │            │
  Lock 없음    Lock 있음
     │            │
     │            ▼
     │   ┌──────────────────────────┐
     │   │ Lock 파일의 PID 확인      │
     │   │                          │
     │   │ 해당 PID 프로세스가       │
     │   │ 실행 중인가?              │
     │   └────┬─────────────────────┘
     │        │
     │   ┌────┴────┐
     │   │         │
     │ 실행중    종료됨
     │   │      (비정상 종료)
     │   │         │
     │   ▼         ▼
     │  에러:    Lock 삭제
     │  "이미    → 새 Lock
     │   실행    파일 생성
     │   중"
     │
     ▼
  Lock 파일 생성
  (PID + 시작 시각 기록)
       │
       ▼
  엔진 시작
```

### 10.2 파일별 Lock (같은 파일 동시 처리 방지)

같은 파일에 대한 이벤트가 동시에 처리되는 것을 방지한다.

```typescript
class FileLockManager {
  private locks: Map<string, { holder: string; acquiredAt: number }>;
  private lockTimeoutMs: 300000;  // 5분 타임아웃 (무한 잠금 방지)

  /**
   * 파일 Lock 획득 시도
   * @param filePath - 잠글 파일 경로
   * @param holder - Lock 소유자 (이벤트 ID)
   * @returns Lock 획득 성공 여부
   */
  async acquire(filePath: string, holder: string): Promise<boolean> {
    const existing = this.locks.get(filePath);

    if (existing) {
      // 기존 Lock 타임아웃 확인
      if (Date.now() - existing.acquiredAt > this.lockTimeoutMs) {
        // 타임아웃 → 강제 해제 후 새 Lock 획득
        logger.warn('파일 Lock 타임아웃, 강제 해제', {
          filePath,
          previousHolder: existing.holder
        });
        this.locks.delete(filePath);
      } else {
        return false;  // Lock 획득 실패
      }
    }

    this.locks.set(filePath, { holder, acquiredAt: Date.now() });
    return true;
  }

  /**
   * 파일 Lock 해제
   */
  release(filePath: string, holder: string): void {
    const existing = this.locks.get(filePath);
    if (existing && existing.holder === holder) {
      this.locks.delete(filePath);
    }
  }
}
```

**Lock 사용 패턴:**

```
  이벤트 처리 시작
       │
       ▼
  ┌─────────────────────────┐
  │ fileLock.acquire(path)  │
  └────────┬────────────────┘
           │
     ┌─────┴─────┐
     │           │
  획득 성공    획득 실패
     │           │
     ▼           ▼
  ┌─────────┐  이벤트 큐에
  │ 다운로드 │  재추가
  │ + 업로드 │  (1초 후)
  │ + 검증   │
  └────┬────┘
       │
       ▼
  fileLock.release(path)
```

### 10.3 DB 트랜잭션 전략

SQLite WAL(Write-Ahead Logging) 모드를 사용하여 읽기/쓰기 동시성을 확보한다.

```typescript
interface DBConfig {
  // WAL 모드 활성화 (읽기 중에 쓰기 가능)
  journal_mode: 'WAL';

  // 동시 쓰기 시 대기 시간 (5초)
  busy_timeout: 5000;

  // 자동 체크포인트 페이지 수
  wal_autocheckpoint: 1000;

  // 캐시 크기 (페이지 수, 음수면 KB)
  cache_size: -8000;  // 8MB

  // 동기화 모드 (NORMAL: 성능과 안정성 절충)
  synchronous: 'NORMAL';
}
```

**트랜잭션 사용 규칙:**

| 작업 | 트랜잭션 범위 | 격리 수준 |
|------|-------------|----------|
| 이벤트 기록 | 이벤트 단위 | IMMEDIATE |
| 체크포인트 저장 | 배치 단위 (변경된 키 전체) | IMMEDIATE |
| 상태 조회 | 트랜잭션 없음 (WAL 읽기) | READ |
| DLQ 이동 | 이벤트 단위 | IMMEDIATE |
| 스냅샷 저장 | 전체 스냅샷 | EXCLUSIVE (드물게) |

### 10.4 큐 처리 동시성 (p-queue 설정)

```typescript
// 메인 이벤트 큐
const eventQueue = new PQueue({
  concurrency: 5,         // 동시 처리 이벤트 수
  interval: 100,          // 레이트 리밋 간격 (ms)
  intervalCap: 3,         // interval 내 최대 시작 수
  timeout: 600000,        // 단일 이벤트 처리 타임아웃 (10분)
  throwOnTimeout: true,
  autoStart: true,
});

// 다운로드 큐
const downloadQueue = new PQueue({
  concurrency: 5,         // 동시 다운로드 수
  timeout: 300000,        // 단일 다운로드 타임아웃 (5분)
});

// 업로드 큐
const uploadQueue = new PQueue({
  concurrency: 3,         // 동시 업로드 수
  timeout: 600000,        // 단일 업로드 타임아웃 (10분)
});

// 재시도 큐
const retryQueue = new PQueue({
  concurrency: 2,         // 동시 재시도 수 (메인보다 낮음)
  interval: 1000,         // 재시도 간격
  intervalCap: 1,
});
```
