# 4. 다운로드 관리

### 4.1 다운로드 아키텍처 개요

다운로드 관리자는 이벤트 큐에서 처리 대상 이벤트를 가져와 LGU+ 웹하드에서 파일을 다운로드하는 역할을 담당한다.

```
┌──────────────────────────────────────────────────────────────────┐
│                      다운로드 관리자 구조                          │
└──────────────────────────────────────────────────────────────────┘

                    ┌───────────────────┐
                    │    이벤트 큐       │
                    │  (Priority Queue) │
                    └────────┬──────────┘
                             │ dequeue()
                             ▼
                    ┌───────────────────┐
                    │  다운로드 관리자    │
                    │ DownloadManager   │
                    └────────┬──────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              ▼              ▼
       ┌───────────┐  ┌───────────┐  ┌───────────┐
       │ Worker #1 │  │ Worker #2 │  │ Worker #3 │   ... 최대 5개
       │ (슬롯 1)  │  │ (슬롯 2)  │  │ (슬롯 3)  │
       └─────┬─────┘  └─────┬─────┘  └─────┬─────┘
             │              │              │
             ▼              ▼              ▼
       ┌───────────────────────────────────────┐
       │           임시 디렉토리                 │
       │  ./temp/downloads/                    │
       │    ├── {uuid1}.tmp                    │
       │    ├── {uuid2}.tmp                    │
       │    └── {uuid3}.tmp                    │
       └───────────────────────────────────────┘
             │ 검증 완료 후
             ▼
       ┌───────────────────────────────────────┐
       │           완료 디렉토리                 │
       │  ./downloads/올리기전용/업체A/          │
       │    ├── file1.dxf                      │
       │    └── file2.dxf                      │
       └───────────────────────────────────────┘
```

### 4.2 동시 다운로드 수 제한

p-queue 라이브러리를 사용하여 동시 다운로드 수를 제한한다.

```typescript
interface DownloadManagerConfig {
  concurrency: 5;              // 동시 다운로드 수 (기본값: 5)
  maxConcurrency: 10;          // 최대 동시 다운로드 수 (설정 상한)
  minConcurrency: 1;           // 최소 동시 다운로드 수
  intervalMs: 100;             // 다운로드 시작 간격 (레이트 리밋 방지)
  intervalCap: 3;              // interval 내 최대 시작 수
  timeoutMs: 300000;           // 단일 다운로드 타임아웃 (5분)
  tempDir: './temp/downloads'; // 임시 다운로드 디렉토리
  outputDir: './downloads';    // 최종 다운로드 디렉토리
}
```

**동시성 자동 조절:**

```
  성공률 모니터링 기반 동시성 자동 조절

  성공률 > 95%   → concurrency + 1 (최대 maxConcurrency까지)
  성공률 < 80%   → concurrency - 1 (최소 minConcurrency까지)
  성공률 < 50%   → concurrency = 1 (최소값으로 급감)

  ┌──────────────────────────────────────────────────────┐
  │  성공률    │  동시성 조절        │  상태              │
  ├───────────┼────────────────────┼───────────────────┤
  │  100%     │  concurrency = 5   │  정상              │
  │   90%     │  concurrency = 4   │  약간 불안정        │
  │   70%     │  concurrency = 2   │  불안정             │
  │   40%     │  concurrency = 1   │  심각               │
  └──────────────────────────────────────────────────────┘
```

### 4.3 다운로드 진행률 추적

각 다운로드 작업의 진행률을 실시간으로 추적하여 GUI에 표시한다.

```typescript
interface DownloadProgress {
  eventId: string;           // 이벤트 ID
  fileName: string;          // 파일명
  totalBytes: number;        // 전체 파일 크기
  downloadedBytes: number;   // 다운로드 완료 바이트
  percentage: number;        // 진행률 (0-100)
  speedBps: number;          // 다운로드 속도 (bytes/sec)
  estimatedRemainingMs: number; // 예상 남은 시간 (ms)
  status: 'pending' | 'downloading' | 'verifying' | 'completed' | 'failed';
  startedAt: number;
  error?: string;
}

// 진행률 이벤트 발행 (IPC → GUI)
interface DownloadProgressEvent {
  type: 'download:progress';
  payload: DownloadProgress;
}

// 전체 다운로드 상태 요약
interface DownloadSummary {
  type: 'download:summary';
  payload: {
    activeDownloads: number;     // 진행 중인 다운로드 수
    queuedDownloads: number;     // 대기 중인 다운로드 수
    completedDownloads: number;  // 완료된 다운로드 수
    failedDownloads: number;     // 실패한 다운로드 수
    totalBytesDownloaded: number; // 총 다운로드 바이트
    averageSpeedBps: number;     // 평균 다운로드 속도
  };
}
```

### 4.4 대용량 파일 처리 (스트림 기반)

파일 크기에 관계없이 메모리 효율적으로 다운로드하기 위해 스트림 기반 다운로드를 사용한다.

```
┌──────────────────────────────────────────────────────────┐
│              스트림 기반 다운로드 흐름                      │
└──────────────────────────────────────────────────────────┘

  LGU+ API 응답 (HTTP Response)
       │
       ▼
  ┌─────────────────────┐
  │ ReadableStream      │   HTTP 응답 바디 (청크 단위)
  │ (response.body)     │
  └────────┬────────────┘
           │ pipe
           ▼
  ┌─────────────────────┐
  │ TransformStream     │   진행률 추적 + 해시 계산
  │ (ProgressTracker)   │   - downloadedBytes 업데이트
  │                     │   - SHA-256 해시 누적 계산
  └────────┬────────────┘
           │ pipe
           ▼
  ┌─────────────────────┐
  │ WritableStream      │   임시 파일에 기록
  │ (FileWriter)        │   ./temp/downloads/{uuid}.tmp
  └─────────────────────┘
```

**파일 크기별 처리 전략:**

| 파일 크기 | 전략 | 메모리 사용량 |
|----------|------|-------------|
| < 10MB | 스트림 (기본) | ~64KB (버퍼 크기) |
| 10MB ~ 100MB | 스트림 + 진행률 표시 | ~64KB |
| 100MB ~ 1GB | 스트림 + 청크 검증 | ~64KB |
| > 1GB | 스트림 + 경고 로그 | ~64KB |

```typescript
const STREAM_CONFIG = {
  highWaterMark: 64 * 1024,    // 64KB 버퍼 크기
  progressIntervalMs: 500,      // 진행률 업데이트 간격 (500ms)
};
```

### 4.5 임시 파일에서 최종 위치로 이동 (Atomic Write)

다운로드가 중간에 실패해도 불완전한 파일이 최종 위치에 남지 않도록 원자적 쓰기 패턴을 사용한다.

```
┌──────────────────────────────────────────────────────────────┐
│                   Atomic Write 패턴                           │
└──────────────────────────────────────────────────────────────┘

  1단계: 임시 파일에 다운로드
  ┌──────────────────────────────────────────┐
  │ ./temp/downloads/a1b2c3d4.tmp            │ ← 다운로드 중
  │ (불완전한 파일이 여기에 쓰여짐)            │
  └──────────────────────────────────────────┘
           │ 다운로드 완료 + 검증 통과
           ▼
  2단계: 최종 디렉토리 확인/생성
  ┌──────────────────────────────────────────┐
  │ ./downloads/올리기전용/업체A/              │ ← mkdir -p (없으면 생성)
  └──────────────────────────────────────────┘
           │
           ▼
  3단계: 원자적 이동 (rename)
  ┌──────────────────────────────────────────┐
  │ fs.rename(                               │
  │   './temp/downloads/a1b2c3d4.tmp',       │
  │   './downloads/올리기전용/업체A/file1.dxf' │
  │ )                                        │
  └──────────────────────────────────────────┘

  ※ rename은 같은 파일시스템 내에서 원자적(atomic) 연산
  ※ 임시 디렉토리와 최종 디렉토리를 같은 드라이브에 배치
```

**동일 파일명 충돌 처리:**

```typescript
async function resolveFilePath(targetPath: string): Promise<string> {
  if (!await fileExists(targetPath)) {
    return targetPath;  // 충돌 없음
  }

  // 기존 파일과 새 파일의 크기/해시 비교
  const existingHash = await computeFileHash(targetPath);
  const newHash = await computeFileHash(tempPath);

  if (existingHash === newHash) {
    // 동일 파일 → 다운로드 스킵 (이미 존재)
    return null;
  }

  // 다른 파일 → 파일명에 타임스탬프 추가
  const ext = path.extname(targetPath);
  const base = path.basename(targetPath, ext);
  const dir = path.dirname(targetPath);
  const timestamp = Date.now();
  return path.join(dir, `${base}_${timestamp}${ext}`);
}
```

### 4.6 다운로드 검증

다운로드된 파일의 무결성을 검증한다.

```
┌──────────────────────────────────────────────────────────┐
│                    다운로드 검증 흐름                      │
└──────────────────────────────────────────────────────────┘

  다운로드 완료된 임시 파일
       │
       ▼
  ┌─────────────────────────────┐
  │ 1. 파일 크기 검증            │
  │    downloaded.size           │
  │    === expected.size ?       │
  │                              │
  │    불일치 → 재다운로드        │
  └────────┬────────────────────┘
           │ 일치
           ▼
  ┌─────────────────────────────┐
  │ 2. SHA-256 해시 검증         │
  │    (API에서 해시를 제공하는   │
  │     경우에만 수행)            │
  │                              │
  │    불일치 → 재다운로드        │
  └────────┬────────────────────┘
           │ 일치 (또는 해시 없음)
           ▼
  ┌─────────────────────────────┐
  │ 3. 파일 읽기 가능 검증       │
  │    fs.open(path, 'r') 성공?  │
  │                              │
  │    실패 → 재다운로드          │
  └────────┬────────────────────┘
           │ 성공
           ▼
       검증 통과 → Atomic Write 진행
```

```typescript
interface DownloadVerification {
  checkFileSize: true;       // 항상 수행
  checkHash: boolean;        // API가 해시를 제공하면 수행
  checkReadable: true;       // 항상 수행
  maxRetryOnFailure: 2;      // 검증 실패 시 최대 재다운로드 횟수
}
```

### 4.7 다운로드 디렉토리 관리

```typescript
interface DirectoryConfig {
  // 임시 다운로드 디렉토리 (다운로드 중인 파일)
  tempDir: './temp/downloads';

  // 최종 다운로드 디렉토리 (완료된 파일, 업로드 대기)
  outputDir: './downloads';

  // 업로드 완료 후 파일 처리
  completedFilePolicy: 'keep' | 'delete' | 'archive';

  // 아카이브 디렉토리 (completedFilePolicy가 'archive'일 때)
  archiveDir: './archive';

  // 임시 파일 정리 주기
  tempCleanupIntervalMs: 3600000;  // 1시간

  // 임시 파일 최대 보존 시간
  tempMaxAgeMs: 86400000;          // 24시간
}
```

**임시 파일 정리 로직:**

프로그램 시작 시와 1시간 간격으로 임시 디렉토리를 스캔하여 24시간 이상 된 `.tmp` 파일을 삭제한다. 이는 비정상 종료 시 남겨진 불완전한 다운로드 파일을 정리하기 위함이다.
