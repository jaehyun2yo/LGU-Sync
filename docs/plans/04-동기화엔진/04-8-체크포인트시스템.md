# 8. 체크포인트 시스템

### 8.1 개요

체크포인트 시스템은 동기화 엔진의 진행 상태를 영속적으로 저장하여, 프로그램 재시작이나 비정상 종료 후에도 마지막 처리 지점부터 이어서 작업할 수 있게 한다.

### 8.2 체크포인트 키-값 목록

```typescript
interface CheckpointStore {
  // Primary 감지 전략 체크포인트
  'primary.last_history_no': number;        // 마지막 처리한 히스토리 번호
  'primary.last_poll_at': number;           // 마지막 폴링 시각 (ms)
  'primary.consecutive_failures': number;   // 연속 실패 횟수

  // Secondary 감지 전략 체크포인트
  'secondary.last_snapshot_at': number;     // 마지막 스냅샷 시각 (ms)
  'secondary.snapshot_hash': string;        // 마지막 스냅샷 해시 (변경 감지용)
  'secondary.total_entries': number;        // 마지막 스냅샷의 총 항목 수

  // Tertiary 감지 전략 체크포인트
  'tertiary.last_checked_id': number;       // 마지막 검증한 레코드 ID
  'tertiary.last_check_at': number;         // 마지막 검증 시각 (ms)
  'tertiary.missing_count': number;         // 마지막 검증에서 발견된 누락 수

  // 이벤트 처리 체크포인트
  'events.last_processed_id': string;       // 마지막 처리한 이벤트 ID
  'events.pending_count': number;           // 미처리 이벤트 수
  'events.total_processed': number;         // 총 처리된 이벤트 수

  // 다운로드 체크포인트
  'download.active_count': number;          // 진행 중인 다운로드 수
  'download.total_bytes': number;           // 총 다운로드 바이트 수
  'download.total_files': number;           // 총 다운로드 파일 수

  // 업로드 체크포인트
  'upload.active_count': number;            // 진행 중인 업로드 수
  'upload.total_bytes': number;             // 총 업로드 바이트 수
  'upload.total_files': number;             // 총 업로드 파일 수

  // 서킷 브레이커 상태
  'circuit.state': 'CLOSED' | 'OPEN' | 'HALF_OPEN';
  'circuit.opened_at': number;              // 서킷 열린 시각
  'circuit.failure_rate': number;           // 현재 실패율

  // 엔진 상태
  'engine.state': string;                   // 엔진 상태
  'engine.started_at': number;              // 엔진 시작 시각
  'engine.last_heartbeat': number;          // 마지막 하트비트 시각

  // 인증 상태
  'auth.last_login_at': number;             // 마지막 로그인 시각
  'auth.cookie_expires_at': number;         // 쿠키 만료 시각
  'auth.reauth_count': number;              // 재인증 횟수
}
```

### 8.3 저장 타이밍

체크포인트는 다음 시점에 저장된다:

```
┌──────────────────────────────────────────────────────────────┐
│                  체크포인트 저장 타이밍                         │
└──────────────────────────────────────────────────────────────┘

  1. 이벤트 단위 저장 (매 이벤트 처리 완료 후)
     ┌────────────┐     ┌────────────┐     ┌────────────┐
     │ 이벤트 처리 │────▶│ 결과 기록  │────▶│ 체크포인트  │
     │   완료     │     │  (DB)     │     │   저장     │
     └────────────┘     └────────────┘     └────────────┘

  2. 주기적 저장 (30초 간격)
     ┌─────────────────┐
     │ 30초 타이머 발동  │────▶ 전체 체크포인트 플러시
     └─────────────────┘

  3. 상태 전이 시 저장 (엔진 상태가 변경될 때)
     ┌────────────────────────┐
     │ 상태 변경 감지           │────▶ 관련 체크포인트 저장
     │ (RUNNING → PAUSED 등)  │
     └────────────────────────┘

  4. 종료 시 저장 (정상 종료)
     ┌──────────────────┐
     │ 엔진 종료 시그널  │────▶ 전체 체크포인트 플러시 + DB sync
     └──────────────────┘
```

**체크포인트 저장 구현:**

```typescript
class CheckpointManager {
  private store: Map<string, unknown>;
  private dirty: Set<string>;      // 변경된 키 추적
  private db: Database;
  private flushIntervalMs: 30000;

  // 단일 키 저장 (더티 마킹만, 실제 DB 기록은 플러시 시)
  set(key: string, value: unknown): void {
    this.store.set(key, value);
    this.dirty.add(key);
  }

  // 즉시 DB에 기록 (중요한 체크포인트)
  async setImmediate(key: string, value: unknown): Promise<void> {
    this.store.set(key, value);
    await this.db.upsertCheckpoint(key, value);
    this.dirty.delete(key);
  }

  // 더티 체크포인트 일괄 플러시
  async flush(): Promise<void> {
    if (this.dirty.size === 0) return;

    const entries = Array.from(this.dirty).map(key => ({
      key,
      value: this.store.get(key)
    }));

    await this.db.batchUpsertCheckpoints(entries);
    this.dirty.clear();
  }

  // DB에서 복원
  async restore(): Promise<void> {
    const entries = await this.db.getAllCheckpoints();
    for (const { key, value } of entries) {
      this.store.set(key, value);
    }
  }
}
```

### 8.4 복구 시나리오별 동작

#### 시나리오 1: 정상 종료 후 재시작

```
  종료 시: 모든 체크포인트 플러시됨
  재시작 시:
    1. DB에서 체크포인트 복원
    2. primary.last_history_no부터 폴링 재개
    3. 미처리 이벤트(events.pending_count > 0) 확인 → 큐에 재추가
    4. 정상 동작 시작
```

#### 시나리오 2: 비정상 종료 (강제 종료, 전원 차단)

```
  종료 시: 마지막 플러시 시점까지만 저장됨 (최대 30초 데이터 손실)
  재시작 시:
    1. DB에서 체크포인트 복원
    2. primary.last_history_no부터 폴링 재개
       → 일부 이벤트가 중복 감지될 수 있으나 중복 제거기가 처리
    3. 임시 다운로드 디렉토리 정리 (불완전한 .tmp 파일 삭제)
    4. DB의 status='processing' 레코드를 'pending'으로 되돌림
    5. 정상 동작 시작
```

#### 시나리오 3: 긴 오프라인 기간 후 재시작

```
  종료 시: 정상 체크포인트 저장
  1일+ 후 재시작 시:
    1. DB에서 체크포인트 복원
    2. primary.last_history_no 이후의 히스토리 조회
       → 히스토리 보관 기간 초과 시 빈 결과 반환 가능
    3. Secondary(스냅샷 비교)를 즉시 실행하여 전체 현재 상태 파악
    4. Tertiary(무결성 검증)를 즉시 실행하여 누락 파일 확인
    5. 정상 동작 시작
```

#### 시나리오 4: DB 파일 손상/삭제

```
  재시작 시:
    1. DB 파일 없음 감지 → 새 DB 생성 + 마이그레이션 실행
    2. 모든 체크포인트 초기값으로 시작
    3. Secondary(스냅샷 비교)를 즉시 실행하여 현재 상태를 초기 스냅샷으로 저장
    4. 이후 변경분부터 동기화 시작
    ※ 기존 동기화 이력이 사라지므로 일부 파일이 재동기화될 수 있음
       (멱등성 보장으로 문제 없음)
```

       │
       ▼
