# 2. 변경 감지 전략 (핵심)

> **SDD Level 2** — 이 인터페이스는 스펙 파일 `src/core/types/file-detector.types.ts`에 정의되며, 구현체는 이 계약을 준수해야 한다.

### 2.1 개요

v1의 가장 큰 문제점은 단일 폴링 전략에만 의존하여 간헐적으로 파일 변경을 감지하지 못하는 것이었다. v2에서는 **3중 감지 전략**을 도입하여 100% 감지를 보장한다.

```
┌───────────────────────────────────────────────────────────────────┐
│                     3중 감지 전략 아키텍처                          │
└───────────────────────────────────────────────────────────────────┘

  시간축 ──────────────────────────────────────────────────────────▶

  Primary (5초)
  ▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░
  │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │
  ▓ = 업로드 히스토리 폴링 (5초마다)

  Secondary (10분)
  ▓─────────────────────────▓─────────────────────────▓───────────
  │                         │                         │

  ▓───────────────────────────────────────────────────────────────▓

  ┌─────────┐  ┌─────────┐  ┌─────────┐
  │ Primary │  │Secondary│  │Tertiary │
  │ 폴링    │  │ 스냅샷  │  │ 무결성  │

  │                                  │
```

### 2.2 Primary: 업로드 히스토리 폴링 (5초 간격)

**목적**: 거의 실시간으로 새 파일 업로드를 감지한다.

**동작 원리:**

1. LGU+ 웹하드의 비공식 API 엔드포인트를 호출하여 업로드 히스토리를 조회한다
2. 마지막으로 처리한 `history_no` 이후의 새 이벤트만 가져온다
3. 새 이벤트가 있으면 이벤트 큐에 추가한다
4. 처리한 마지막 `history_no`를 체크포인트에 저장한다

```
┌──────────────────────────────────────────────────────────┐
│              업로드 히스토리 폴링 흐름                      │
└──────────────────────────────────────────────────────────┘

  ┌─────────────────┐
  │ 5초 타이머 발동   │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────────┐
  │ 체크포인트에서        │
  │ last_history_no 조회 │
  └────────┬────────────┘
           │
           ▼
  ┌─────────────────────────────────┐
  │ LGU+ API 호출:                  │
  │ GET /api/history                │
  │   ?type=upload                  │
  │   &since_no={last_history_no}   │
  │   &limit=100                    │
  └────────┬────────────────────────┘
           │
           ▼
  ┌─────────────────────┐     아니오
  │ 새 이벤트가 있는가?  │──────────▶ (종료, 다음 주기 대기)
  └────────┬────────────┘
           │ 예
           ▼
  ┌─────────────────────────────┐
  │ 각 이벤트를 SyncEvent로 변환 │
  │ ─ path, size, mtime 추출    │
  │ ─ 이벤트 타입 결정           │
  └────────┬────────────────────┘
           │
           ▼
  ┌─────────────────────────────┐
  │ 중복 제거기(Deduplicator)로  │
  │ 이벤트 전달                  │
  └────────┬────────────────────┘
           │
           ▼
  ┌─────────────────────────────────┐
  │ last_history_no 체크포인트 갱신  │
  └─────────────────────────────────┘
```

**API 호출 상세:**

```typescript
interface HistoryPollingConfig {
  intervalMs: 5000;           // 폴링 간격: 5초
  maxEventsPerPoll: 100;      // 한 번에 가져올 최대 이벤트 수
  checkpointKey: 'primary.last_history_no';
  timeoutMs: 10000;           // API 호출 타임아웃: 10초
}

interface HistoryEntry {
  history_no: number;         // 히스토리 고유 번호 (단조 증가)
  action: 'upload' | 'modify' | 'delete' | 'mkdir' | 'rmdir';
  file_path: string;          // 전체 파일 경로
  file_name: string;          // 파일명
  file_size: number;          // 파일 크기 (bytes)
  modified_at: string;        // 수정 시각 (ISO 8601)
  user_id: string;            // 업로드한 사용자 ID
}
```

**장점:**
- 거의 실시간 감지 (5초 이내)
- API 호출량이 적음 (변경분만 조회)
- `history_no` 기반으로 누락 없이 순차 처리 가능

**단점:**
- LGU+ API가 비공식이므로 API 변경 시 깨질 수 있음
- API 서버 장애 시 감지 불가
- 히스토리 보관 기간이 제한될 수 있음 (장기간 오프라인 후 복구 어려움)

**위험 완화:**
- Secondary(스냅샷 비교)가 Primary 실패를 보완
- API 응답 형식 변경 감지 로직 포함
- 연속 3회 실패 시 경고 로그 + GUI 알림

### 2.3 Secondary: 스냅샷 비교 (10분 간격)

**목적**: Primary 폴링이 놓친 파일을 잡아내는 안전망(safety net) 역할.

**동작 원리:**

1. LGU+ 웹하드의 전체 폴더 트리를 재귀적으로 조회하여 스냅샷을 생성한다
2. 이전 스냅샷과 현재 스냅샷을 비교(diff)하여 변경사항을 추출한다
3. 변경사항을 이벤트로 변환하여 중복 제거기로 전달한다

```
┌──────────────────────────────────────────────────────────┐
│                스냅샷 비교 흐름                            │
└──────────────────────────────────────────────────────────┘

  ┌───────────────────┐
  │ 10분 타이머 발동    │
  └─────────┬─────────┘
            │
            ▼
  ┌───────────────────────────────────┐
  │ LGU+ API: 전체 폴더 트리 조회      │
  │                                   │
  │ 올리기전용/                        │
  │   ├── 업체A/                      │
  │   │   ├── file1.dxf  (1.2MB, t1) │
  │   │   └── file2.dxf  (0.8MB, t2) │
  │   ├── 업체B/                      │
  │   │   └── file3.pdf  (2.1MB, t3) │
  │   └── ...                        │
  └─────────┬─────────────────────────┘
            │
            ▼
  ┌───────────────────────────────────┐
  │ 현재 스냅샷 생성                   │
  │                                   │
  │ Map<string, SnapshotEntry>        │
  │ key = 파일 전체 경로               │
  │ value = { size, mtime, type }    │
  └─────────┬─────────────────────────┘
            │
            ▼
  ┌────────────────────────────────────────────────────┐
  │              스냅샷 비교 (Diff)                      │
  │                                                    │
  │  이전 스냅샷 (prev)    vs    현재 스냅샷 (curr)      │
  │                                                    │
  │  curr에만 있음 ─────▶ FILE_CREATED / FOLDER_CREATED │
  │  prev에만 있음 ─────▶ FILE_DELETED / FOLDER_DELETED │
  │  양쪽 다 있지만       ─────▶ FILE_MODIFIED           │
  │  size/mtime 다름                                    │
  └─────────┬──────────────────────────────────────────┘
            │
            ▼
  ┌───────────────────────────────────┐
  │ 변경 이벤트를 중복 제거기로 전달   │
  └─────────┬─────────────────────────┘
            │
            ▼
  ┌───────────────────────────────────┐
  │ 현재 스냅샷을 "이전 스냅샷"으로    │
  │ 저장 (다음 비교를 위해)            │
  └───────────────────────────────────┘
```

**스냅샷 자료구조:**

```typescript
interface SnapshotEntry {
  path: string;           // 전체 경로 (예: "/올리기전용/업체A/file1.dxf")
  name: string;           // 파일명
  type: 'file' | 'dir';   // 파일 또는 폴더
  size: number;           // 파일 크기 (bytes), 폴더는 0
  mtime: number;          // 수정 시각 (Unix timestamp ms)
  parentPath: string;     // 부모 폴더 경로
}

interface Snapshot {
  capturedAt: number;      // 스냅샷 캡처 시각
  entries: Map<string, SnapshotEntry>;  // path → entry
  totalFiles: number;
  totalFolders: number;
}

interface SnapshotDiff {
  created: SnapshotEntry[];   // 새로 생성된 항목
  modified: SnapshotEntry[];  // 수정된 항목
  deleted: SnapshotEntry[];   // 삭제된 항목
}
```

**폴더 트리 조회 최적화:**

```
올리기전용/
├── 업체A/          ← 1차 API 호출: 루트 폴더 목록
│   ├── file1.dxf   ← 2차 API 호출: 업체A 폴더 내용
│   └── file2.dxf
├── 업체B/
│   └── sub/        ← 3차 API 호출: sub 폴더 내용
│       └── file3.pdf
└── 업체C/
    └── file4.ai    ← 2차 API 호출: 업체C 폴더 내용
```

- 폴더 깊이별로 순차 조회 (BFS 방식)
- 각 폴더당 1회 API 호출 필요
- 동시 요청 수를 3개로 제한하여 API 부하 방지
- 폴더 수가 많을 경우 전체 스냅샷에 수 초 소요 가능

**장점:**
- 폴링이 놓친 파일을 100% 잡아냄
- API 히스토리 기능에 의존하지 않음
- 파일 삭제도 감지 가능

**단점:**
- API 호출량이 많음 (폴더 수 × 1회)
- 10분 간격이므로 실시간성은 떨어짐
- 대량 파일 시 비교 연산 비용 증가

### 2.4 Tertiary: 무결성 검증 (1시간 간격)

**목적**: 동기화가 완료되었다고 기록된 파일이 실제로 내부 웹하드에 존재하는지 확인하고, 누락된 파일을 재동기화한다.

```
┌──────────────────────────────────────────────────────────┐
│                무결성 검증 흐름                            │
└──────────────────────────────────────────────────────────┘

  ┌───────────────────┐
  │ 1시간 타이머 발동   │
  └─────────┬─────────┘
            │
            ▼
  ┌─────────────────────────────────────┐
  │ DB에서 sync_status='completed'인    │
  │ 모든 레코드 조회                     │
  └─────────┬───────────────────────────┘
            │
            ▼
  ┌─────────────────────────────────────┐
  │ 내부 웹하드에서 해당 파일 존재 확인  │
  │ (HEAD 요청으로 경량 확인)            │
  └─────────┬───────────────────────────┘
            │
            ▼
  ┌─────────────────────┐      예
  │ 누락된 파일이 있는가? │────────────┐
  └─────────┬───────────┘            │
            │ 아니오                  ▼
            ▼              ┌──────────────────────┐
     (검증 완료, 로그)      │ 누락 파일을 재동기화   │
                           │ 이벤트로 생성하여      │
                           │ 이벤트 큐에 추가       │
                           └──────────────────────┘
```

**검증 대상:**

```typescript
interface IntegrityCheckConfig {
  intervalMs: 3600000;          // 1시간
  batchSize: 50;                // 한 번에 검증할 파일 수
  checkpointKey: 'tertiary.last_checked_id';
  maxFileAge: 30 * 24 * 3600000;  // 30일 이내 파일만 검증
}
```

**검증 로직:**

1. DB에서 `sync_status = 'completed'`이고 `created_at`이 30일 이내인 레코드를 50건씩 조회
2. 각 레코드의 `target_path`로 내부 웹하드에 HEAD 요청
3. 파일이 없거나 크기가 다르면 `FILE_CREATED` 이벤트를 생성하여 재동기화
4. 마지막 검증한 레코드 ID를 체크포인트에 저장
5. 모든 레코드를 검증한 후 처음부터 다시 시작 (순환)

```
  │  ┌─────────────────────────────────────────────┐             │
  │  │ 처리 완료 이벤트 캐시 (LRU, 최대 10,000건)   │             │
  │  │                                             │             │
  │  │  file_key_1 → { processedAt, eventId }     │             │
  │  │  file_key_2 → { processedAt, eventId }     │             │
  │  │  ...                                        │             │
  │  └─────────────────────────────────────────────┘             │
  │                                                              │
  │  새 이벤트 수신 시:                                           │
  │  1. file_key 계산                                            │
  │  2. 캐시에 존재하면 → 폐기 (이미 처리됨)                      │
  │  3. DB에 동일 file_key + completed 레코드 있으면 → 폐기       │
  │  4. 캐시에 없고 DB에도 없으면 → 이벤트 큐에 추가              │
  └──────────────────────────────────────────────────────────────┘
```

**중복 제거 알고리즘:**

```typescript
class EventDeduplicator {
  private processedCache: LRUCache<string, ProcessedEntry>;  // 메모리 캐시
  private db: Database;  // SQLite

  async isDuplicate(event: SyncEvent): Promise<boolean> {
    const fileKey = this.computeFileKey(event);

    // 1단계: 메모리 캐시 확인 (가장 빠름)
    if (this.processedCache.has(fileKey)) {
      return true;
    }

    // 2단계: DB 확인 (캐시 미스 시)
    const dbRecord = await this.db.findByFileKey(fileKey);
    if (dbRecord && dbRecord.status === 'completed') {
      // 캐시에 추가하여 다음 조회 가속
      this.processedCache.set(fileKey, {
        processedAt: dbRecord.completedAt,
        eventId: dbRecord.id
      });
      return true;
    }

    return false;
  }

  private computeFileKey(event: SyncEvent): string {
    // path + size + mtime 조합으로 고유 키 생성
    const raw = `${event.path}|${event.size}|${event.mtime}`;
    return createHash('sha256').update(raw).digest('hex').slice(0, 16);
  }
}
```

### 2.6 각 전략의 실행 타이밍과 우선순위

| 전략 | 간격 | 우선순위 | CPU 부하 | API 호출량 | 감지 범위 |
|------|------|---------|---------|-----------|----------|
| Primary (폴링) | 5초 | 최우선 | 낮음 | 1회/주기 | 새 업로드만 |
| Secondary (스냅샷) | 10분 | 보조 | 중간 | 폴더수/주기 | 전체 변경 |
| Tertiary (무결성) | 1시간 | 최저 | 낮음 | 파일수/50/주기 | 동기화 누락 |

**실행 규칙:**

1. Primary는 항상 실행된다 (엔진이 RUNNING 상태인 한)
2. Secondary는 Primary가 연속 3회 이상 실패했을 때 간격을 5분으로 단축한다
3. Tertiary는 Secondary 실행 직후에는 건너뛴다 (중복 방지)
4. 모든 전략은 서킷 브레이커가 OPEN 상태이면 실행을 건너뛴다
