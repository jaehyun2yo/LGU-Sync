## 1. 동기화 엔진 개요

### 1.1 엔진의 역할과 책임

동기화 엔진은 외부웹하드동기화프로그램 v2의 핵심 컴포넌트로서, LGU+ 웹하드(only.webhard.co.kr)의 `올리기전용/` 폴더에 거래처가 업로드한 파일을 감지하여 내부 웹하드(Cloudflare R2 기반)로 자동 동기화하는 단방향(one-way) 동기화를 담당한다.

**핵심 책임:**

| 책임 | 설명 |
|------|------|
| 변경 감지 | LGU+ 웹하드의 파일 변경사항을 100% 누락 없이 감지 |
| 이벤트 처리 | 감지된 변경사항을 이벤트로 변환하고 우선순위에 따라 처리 |
| 파일 전송 | LGU+ 웹하드에서 파일을 다운로드하여 내부 웹하드로 업로드 |
| 무결성 보장 | 전송된 파일의 무결성을 검증하고 누락 방지 |
| 오류 복구 | 네트워크 오류, 인증 만료 등 장애 상황에서 자동 복구 |
| 상태 관리 | 동기화 진행 상태를 체크포인트로 기록하여 재시작 시 이어서 처리 |
| 인증 유지 | LGU+ 쿠키 기반 인증 세션을 자동 갱신 |

**v1 대비 v2의 핵심 개선점:**

- v1 문제: 간헐적 감지 누락 (폴링 단일 전략 의존)
- v2 해결: 다중 감지 전략 (폴링 + 스냅샷 비교 + 무결성 검증) 으로 100% 감지 보장

**설계 원칙:**

1. **감지 무결성**: 어떤 파일도 누락되어서는 안 된다 (최소 1회 감지 보장)
2. **멱등성**: 같은 이벤트를 여러 번 처리해도 결과가 동일해야 한다
3. **내결함성**: 일시적 장애에서 자동 복구되어야 한다
4. **관찰 가능성**: 모든 동작이 로깅되고 모니터링 가능해야 한다
5. **단순성**: 단일 사용자 데스크톱 앱에 맞는 적절한 복잡도 유지

### 1.2 동기화 파이프라인 전체 흐름도

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        동기화 파이프라인 전체 흐름                         │
└─────────────────────────────────────────────────────────────────────────┘

  LGU+ 웹하드                                              내부 웹하드
  (only.webhard.co.kr)                                    (Cloudflare R2)
       │                                                        ▲
       ▼                                                        │
  ┌─────────┐    ┌──────────┐    ┌────────┐    ┌──────────┐    ┌─────────┐
  │  감 지   │───▶│  필터링   │───▶│  큐잉   │───▶│ 다운로드  │───▶│  업로드  │
  │ Detect  │    │  Filter  │    │ Queue  │    │ Download │    │ Upload  │
  └─────────┘    └──────────┘    └────────┘    └──────────┘    └─────────┘
       │              │              │              │               │
       │              │              │              │               ▼
       │              │              │              │          ┌─────────┐
       │              │              │              │          │  검 증   │
       │              │              │              │          │ Verify  │
       │              │              │              │          └─────────┘
       │              │              │              │               │
       ▼              ▼              ▼              ▼               ▼
  ┌──────────────────────────────────────────────────────────────────────┐
  │                       기 록 (Record to DB)                          │
  │                   SQLite - sync_events 테이블                       │
  └──────────────────────────────────────────────────────────────────────┘
```

**각 단계 상세:**

| 단계 | 입력 | 출력 | 실패 시 |
|------|------|------|---------|
| 감지 (Detect) | LGU+ API 응답 | 원시 변경 이벤트 | 다음 주기에 재감지 |
| 필터링 (Filter) | 원시 이벤트 | 유효한 이벤트 | 이벤트 폐기 + 로그 |
| 큐잉 (Queue) | 유효한 이벤트 | 우선순위 정렬된 큐 | 메모리 큐 → DB 백업 |
| 다운로드 (Download) | 큐의 이벤트 | 로컬 임시 파일 | 재시도 큐 이동 |
| 업로드 (Upload) | 로컬 파일 | 내부 웹하드 파일 | 재시도 큐 이동 |
| 검증 (Verify) | 업로드 결과 | 검증 완료 상태 | 재다운로드/업로드 |
| 기록 (Record) | 검증 결과 | DB 레코드 | WAL 모드로 보장 |

### 1.3 엔진 생명주기

동기화 엔진은 다음의 생명주기 상태를 가진다:

```
                    ┌──────────────────────────────┐
                    │        엔진 생명주기           │
                    └──────────────────────────────┘

                         initialize()
                              │
                              ▼
                    ┌──────────────────┐
                    │   INITIALIZING   │
                    │   (초기화 중)     │
                    └────────┬─────────┘
                             │ 인증 성공 + DB 준비 완료
                             ▼
              ┌──────────────────────────────┐
     ┌───────▶│           RUNNING            │◀──────────┐
     │        │          (실행 중)            │           │
     │        └──────┬────────────┬──────────┘           │
     │               │            │                      │
     │     pause()   │            │  stop()      resume()│
     │               ▼            ▼                      │
     │   ┌────────────────┐  ┌──────────────┐            │
     │   │    PAUSED      │  │  STOPPING    │            │
     │   │  (일시정지)     │  │  (종료 중)   │            │
     │   └───────┬────────┘  └──────┬───────┘            │
     │           │                  │                    │
     │   resume()│                  │ 모든 작업 완료      │
     │           │                  ▼                    │
     │           │         ┌──────────────┐              │
     │           └────────▶│   STOPPED    │              │
     │                     │   (중지됨)    │              │
     │                     └──────┬───────┘              │
     │                            │                      │
     │                    start() │                      │
     │                            └──────────────────────┘
     │
     │  에러 발생 시 자동 복구 시도
     │         ┌──────────────┐
     └─────────│    ERROR     │
               │  (오류 상태)  │
               └──────────────┘
                     │ 복구 불가 시
                     ▼
              ┌──────────────┐
              │    FATAL     │
              │ (치명적 오류) │
              └──────────────┘
```

**각 상태 설명:**

| 상태 | 설명 | 허용되는 전이 |
|------|------|--------------|
| INITIALIZING | 엔진 초기화 중. DB 연결, 인증, 체크포인트 복원 수행 | → RUNNING, → ERROR |
| RUNNING | 정상 동작 중. 모든 감지 전략이 스케줄에 따라 실행됨 | → PAUSED, → STOPPING, → ERROR |
| PAUSED | 일시 정지. 진행 중인 다운로드/업로드는 완료하지만 새 감지는 중단 | → RUNNING, → STOPPING |
| STOPPING | 종료 진행 중. 진행 중인 작업 완료를 기다리고 체크포인트 저장 | → STOPPED |
| STOPPED | 완전 중지. 모든 리소스 해제됨 | → RUNNING (재시작) |
| ERROR | 오류 발생. 자동 복구를 시도하며, 복구 성공 시 RUNNING으로 전이 | → RUNNING, → FATAL |
| FATAL | 복구 불가능한 치명적 오류. 사용자 개입 필요 | → STOPPED (수동) |

**상태 전이 시 수행되는 작업:**

```typescript
// 상태 전이 훅 (의사코드)
interface LifecycleHooks {
  onInitializing(): Promise<void>;  // DB 마이그레이션, 인증, 체크포인트 복원
  onRunning(): Promise<void>;       // 감지 스케줄러 시작, 큐 프로세서 시작
  onPausing(): Promise<void>;       // 새 감지 중단, 진행 중 작업 완료 대기
  onResuming(): Promise<void>;      // 감지 스케줄러 재시작, 놓친 이벤트 확인
  onStopping(): Promise<void>;      // 모든 타이머 해제, 큐 비우기, 체크포인트 저장
  onStopped(): Promise<void>;       // DB 연결 종료, 리소스 해제
  onError(error: Error): Promise<void>;  // 에러 로깅, 복구 시도
}
```

## 2. 변경 감지 전략 (핵심)

> **SDD Level 2** — 이 인터페이스는 스펙 파일 `src/core/types/file-detector.types.ts`에 정의되며, 구현체는 이 계약을 준수해야 한다.

### 2.1 개요

v1의 가장 큰 문제점은 단일 폴링 전략에만 의존하여 간헐적으로 파일 변경을 감지하지 못하는 것이었다. v2에서는 **3중 감지 전략**을 도입하여 100% 감지를 보장한다.

```
┌───────────────────────────────────────────────────────────────────┐
│                     3중 감지 전략 아키텍처                          │
└───────────────────────────────────────────────────────────────────┘

  시간축 ──────────────────────────────────────────────────────────▶

  Primary (5초)
  ▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░▓░
  │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │
  ▓ = 업로드 히스토리 폴링 (5초마다)

  Secondary (10분)
  ▓─────────────────────────▓─────────────────────────▓───────────
  │                         │                         │
  ▓ = 스냅샷 비교 (10분마다)

  Tertiary (1시간)
  ▓───────────────────────────────────────────────────────────────▓
  │                                                               │
  ▓ = 무결성 검증 (1시간마다)

  ┌─────────┐  ┌─────────┐  ┌─────────┐
  │ Primary │  │Secondary│  │Tertiary │
  │ 폴링    │  │ 스냅샷  │  │ 무결성  │
  └────┬────┘  └────┬────┘  └────┬────┘
       │            │            │
       ▼            ▼            ▼
  ┌──────────────────────────────────┐
  │     이벤트 중복 제거기            │
  │   (Deduplication Engine)        │
  │                                  │
  │  file_key = path + size + mtime │
  │  이미 처리된 이벤트는 폐기       │
  └──────────────┬───────────────────┘
                 │
                 ▼
  ┌──────────────────────────────────┐
  │       이벤트 큐 (Event Queue)    │
  └──────────────────────────────────┘
```

### 2.2 Primary: 업로드 히스토리 폴링 (5초 간격)

**목적**: 거의 실시간으로 새 파일 업로드를 감지한다.

**동작 원리:**

1. LGU+ 웹하드의 비공식 API 엔드포인트를 호출하여 업로드 히스토리를 조회한다
2. 마지막으로 처리한 `history_no` 이후의 새 이벤트만 가져온다
3. 새 이벤트가 있으면 이벤트 큐에 추가한다
4. 처리한 마지막 `history_no`를 체크포인트에 저장한다

```
┌──────────────────────────────────────────────────────────┐
│              업로드 히스토리 폴링 흐름                      │
└──────────────────────────────────────────────────────────┘

  ┌─────────────────┐
  │ 5초 타이머 발동   │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────────┐
  │ 체크포인트에서        │
  │ last_history_no 조회 │
  └────────┬────────────┘
           │
           ▼
  ┌─────────────────────────────────┐
  │ LGU+ API 호출:                  │
  │ GET /api/history                │
  │   ?type=upload                  │
  │   &since_no={last_history_no}   │
  │   &limit=100                    │
  └────────┬────────────────────────┘
           │
           ▼
  ┌─────────────────────┐     아니오
  │ 새 이벤트가 있는가?  │──────────▶ (종료, 다음 주기 대기)
  └────────┬────────────┘
           │ 예
           ▼
  ┌─────────────────────────────┐
  │ 각 이벤트를 SyncEvent로 변환 │
  │ ─ path, size, mtime 추출    │
  │ ─ 이벤트 타입 결정           │
  └────────┬────────────────────┘
           │
           ▼
  ┌─────────────────────────────┐
  │ 중복 제거기(Deduplicator)로  │
  │ 이벤트 전달                  │
  └────────┬────────────────────┘
           │
           ▼
  ┌─────────────────────────────────┐
  │ last_history_no 체크포인트 갱신  │
  └─────────────────────────────────┘
```

**API 호출 상세:**

```typescript
interface HistoryPollingConfig {
  intervalMs: 5000;           // 폴링 간격: 5초
  maxEventsPerPoll: 100;      // 한 번에 가져올 최대 이벤트 수
  checkpointKey: 'primary.last_history_no';
  timeoutMs: 10000;           // API 호출 타임아웃: 10초
}

interface HistoryEntry {
  history_no: number;         // 히스토리 고유 번호 (단조 증가)
  action: 'upload' | 'modify' | 'delete' | 'mkdir' | 'rmdir';
  file_path: string;          // 전체 파일 경로
  file_name: string;          // 파일명
  file_size: number;          // 파일 크기 (bytes)
  modified_at: string;        // 수정 시각 (ISO 8601)
  user_id: string;            // 업로드한 사용자 ID
}
```

**장점:**
- 거의 실시간 감지 (5초 이내)
- API 호출량이 적음 (변경분만 조회)
- `history_no` 기반으로 누락 없이 순차 처리 가능

**단점:**
- LGU+ API가 비공식이므로 API 변경 시 깨질 수 있음
- API 서버 장애 시 감지 불가
- 히스토리 보관 기간이 제한될 수 있음 (장기간 오프라인 후 복구 어려움)

**위험 완화:**
- Secondary(스냅샷 비교)가 Primary 실패를 보완
- API 응답 형식 변경 감지 로직 포함
- 연속 3회 실패 시 경고 로그 + GUI 알림

### 2.3 Secondary: 스냅샷 비교 (10분 간격)

**목적**: Primary 폴링이 놓친 파일을 잡아내는 안전망(safety net) 역할.

**동작 원리:**

1. LGU+ 웹하드의 전체 폴더 트리를 재귀적으로 조회하여 스냅샷을 생성한다
2. 이전 스냅샷과 현재 스냅샷을 비교(diff)하여 변경사항을 추출한다
3. 변경사항을 이벤트로 변환하여 중복 제거기로 전달한다

```
┌──────────────────────────────────────────────────────────┐
│                스냅샷 비교 흐름                            │
└──────────────────────────────────────────────────────────┘

  ┌───────────────────┐
  │ 10분 타이머 발동    │
  └─────────┬─────────┘
            │
            ▼
  ┌───────────────────────────────────┐
  │ LGU+ API: 전체 폴더 트리 조회      │
  │                                   │
  │ 올리기전용/                        │
  │   ├── 업체A/                      │
  │   │   ├── file1.dxf  (1.2MB, t1) │
  │   │   └── file2.dxf  (0.8MB, t2) │
  │   ├── 업체B/                      │
  │   │   └── file3.pdf  (2.1MB, t3) │
  │   └── ...                        │
  └─────────┬─────────────────────────┘
            │
            ▼
  ┌───────────────────────────────────┐
  │ 현재 스냅샷 생성                   │
  │                                   │
  │ Map<string, SnapshotEntry>        │
  │ key = 파일 전체 경로               │
  │ value = { size, mtime, type }    │
  └─────────┬─────────────────────────┘
            │
            ▼
  ┌────────────────────────────────────────────────────┐
  │              스냅샷 비교 (Diff)                      │
  │                                                    │
  │  이전 스냅샷 (prev)    vs    현재 스냅샷 (curr)      │
  │                                                    │
  │  curr에만 있음 ─────▶ FILE_CREATED / FOLDER_CREATED │
  │  prev에만 있음 ─────▶ FILE_DELETED / FOLDER_DELETED │
  │  양쪽 다 있지만       ─────▶ FILE_MODIFIED           │
  │  size/mtime 다름                                    │
  └─────────┬──────────────────────────────────────────┘
            │
            ▼
  ┌───────────────────────────────────┐
  │ 변경 이벤트를 중복 제거기로 전달   │
  └─────────┬─────────────────────────┘
            │
            ▼
  ┌───────────────────────────────────┐
  │ 현재 스냅샷을 "이전 스냅샷"으로    │
  │ 저장 (다음 비교를 위해)            │
  └───────────────────────────────────┘
```

**스냅샷 자료구조:**

```typescript
interface SnapshotEntry {
  path: string;           // 전체 경로 (예: "/올리기전용/업체A/file1.dxf")
  name: string;           // 파일명
  type: 'file' | 'dir';   // 파일 또는 폴더
  size: number;           // 파일 크기 (bytes), 폴더는 0
  mtime: number;          // 수정 시각 (Unix timestamp ms)
  parentPath: string;     // 부모 폴더 경로
}

interface Snapshot {
  capturedAt: number;      // 스냅샷 캡처 시각
  entries: Map<string, SnapshotEntry>;  // path → entry
  totalFiles: number;
  totalFolders: number;
}

interface SnapshotDiff {
  created: SnapshotEntry[];   // 새로 생성된 항목
  modified: SnapshotEntry[];  // 수정된 항목
  deleted: SnapshotEntry[];   // 삭제된 항목
}
```

**폴더 트리 조회 최적화:**

```
올리기전용/
├── 업체A/          ← 1차 API 호출: 루트 폴더 목록
│   ├── file1.dxf   ← 2차 API 호출: 업체A 폴더 내용
│   └── file2.dxf
├── 업체B/
│   └── sub/        ← 3차 API 호출: sub 폴더 내용
│       └── file3.pdf
└── 업체C/
    └── file4.ai    ← 2차 API 호출: 업체C 폴더 내용
```

- 폴더 깊이별로 순차 조회 (BFS 방식)
- 각 폴더당 1회 API 호출 필요
- 동시 요청 수를 3개로 제한하여 API 부하 방지
- 폴더 수가 많을 경우 전체 스냅샷에 수 초 소요 가능

**장점:**
- 폴링이 놓친 파일을 100% 잡아냄
- API 히스토리 기능에 의존하지 않음
- 파일 삭제도 감지 가능

**단점:**
- API 호출량이 많음 (폴더 수 × 1회)
- 10분 간격이므로 실시간성은 떨어짐
- 대량 파일 시 비교 연산 비용 증가

### 2.4 Tertiary: 무결성 검증 (1시간 간격)

**목적**: 동기화가 완료되었다고 기록된 파일이 실제로 내부 웹하드에 존재하는지 확인하고, 누락된 파일을 재동기화한다.

```
┌──────────────────────────────────────────────────────────┐
│                무결성 검증 흐름                            │
└──────────────────────────────────────────────────────────┘

  ┌───────────────────┐
  │ 1시간 타이머 발동   │
  └─────────┬─────────┘
            │
            ▼
  ┌─────────────────────────────────────┐
  │ DB에서 sync_status='completed'인    │
  │ 모든 레코드 조회                     │
  └─────────┬───────────────────────────┘
            │
            ▼
  ┌─────────────────────────────────────┐
  │ 내부 웹하드에서 해당 파일 존재 확인  │
  │ (HEAD 요청으로 경량 확인)            │
  └─────────┬───────────────────────────┘
            │
            ▼
  ┌─────────────────────┐      예
  │ 누락된 파일이 있는가? │────────────┐
  └─────────┬───────────┘            │
            │ 아니오                  ▼
            ▼              ┌──────────────────────┐
     (검증 완료, 로그)      │ 누락 파일을 재동기화   │
                           │ 이벤트로 생성하여      │
                           │ 이벤트 큐에 추가       │
                           └──────────────────────┘
```

**검증 대상:**

```typescript
interface IntegrityCheckConfig {
  intervalMs: 3600000;          // 1시간
  batchSize: 50;                // 한 번에 검증할 파일 수
  checkpointKey: 'tertiary.last_checked_id';
  maxFileAge: 30 * 24 * 3600000;  // 30일 이내 파일만 검증
}
```

**검증 로직:**

1. DB에서 `sync_status = 'completed'`이고 `created_at`이 30일 이내인 레코드를 50건씩 조회
2. 각 레코드의 `target_path`로 내부 웹하드에 HEAD 요청
3. 파일이 없거나 크기가 다르면 `FILE_CREATED` 이벤트를 생성하여 재동기화
4. 마지막 검증한 레코드 ID를 체크포인트에 저장
5. 모든 레코드를 검증한 후 처음부터 다시 시작 (순환)

```
┌────────────────────────────────────────────────────────────────────┐
│                    3중 감지 전략 협력 다이어그램                      │
└────────────────────────────────────────────────────────────────────┘

           │                             │                    │
  Second.  │                             ●                    ●
  (10분)   │                          스냅샷 비교           스냅샷 비교
           │                             │                    │
           │               Primary가 놓친 1% 파일을 여기서 감지  │
           │                                                  │
  Tertiary │                                                  ●
  (1시간)  │                                              무결성 검증
           │                                                  │
           │                         동기화 완료 후 누락 파일 재동기화
           │
           ▼
  ┌──────────────────────────────────────────────────────────────┐
  │                    이벤트 중복 제거기                          │
  │                                                              │
  │  중복 판별 기준: file_key = sha256(path + size + mtime)      │
  │                                                              │
  │  ┌─────────────────────────────────────────────┐             │
  │  │ 처리 완료 이벤트 캐시 (LRU, 최대 10,000건)   │             │
  │  │                                             │             │
  │  │  file_key_1 → { processedAt, eventId }     │             │
  │  │  file_key_2 → { processedAt, eventId }     │             │
  │  │  ...                                        │             │
  │  └─────────────────────────────────────────────┘             │
  │                                                              │
  │  새 이벤트 수신 시:                                           │
  │  1. file_key 계산                                            │
  │  2. 캐시에 존재하면 → 폐기 (이미 처리됨)                      │
  │  3. DB에 동일 file_key + completed 레코드 있으면 → 폐기       │
  │  4. 캐시에 없고 DB에도 없으면 → 이벤트 큐에 추가              │
  └──────────────────────────────────────────────────────────────┘
```

**중복 제거 알고리즘:**

```typescript
class EventDeduplicator {
  private processedCache: LRUCache<string, ProcessedEntry>;  // 메모리 캐시
  private db: Database;  // SQLite

  async isDuplicate(event: SyncEvent): Promise<boolean> {
    const fileKey = this.computeFileKey(event);

    // 1단계: 메모리 캐시 확인 (가장 빠름)
    if (this.processedCache.has(fileKey)) {
      return true;
    }

    // 2단계: DB 확인 (캐시 미스 시)
    const dbRecord = await this.db.findByFileKey(fileKey);
    if (dbRecord && dbRecord.status === 'completed') {
      // 캐시에 추가하여 다음 조회 가속
      this.processedCache.set(fileKey, {
        processedAt: dbRecord.completedAt,
        eventId: dbRecord.id
      });
      return true;
    }

    return false;
  }

  private computeFileKey(event: SyncEvent): string {
    // path + size + mtime 조합으로 고유 키 생성
    const raw = `${event.path}|${event.size}|${event.mtime}`;
    return createHash('sha256').update(raw).digest('hex').slice(0, 16);
  }
}
```

### 2.6 각 전략의 실행 타이밍과 우선순위

| 전략 | 간격 | 우선순위 | CPU 부하 | API 호출량 | 감지 범위 |
|------|------|---------|---------|-----------|----------|
| Primary (폴링) | 5초 | 최우선 | 낮음 | 1회/주기 | 새 업로드만 |
| Secondary (스냅샷) | 10분 | 보조 | 중간 | 폴더수/주기 | 전체 변경 |
| Tertiary (무결성) | 1시간 | 최저 | 낮음 | 파일수/50/주기 | 동기화 누락 |

**실행 규칙:**

1. Primary는 항상 실행된다 (엔진이 RUNNING 상태인 한)
2. Secondary는 Primary가 연속 3회 이상 실패했을 때 간격을 5분으로 단축한다
3. Tertiary는 Secondary 실행 직후에는 건너뛴다 (중복 방지)
4. 모든 전략은 서킷 브레이커가 OPEN 상태이면 실행을 건너뛴다

### 3.1 이벤트 타입 정의

```typescript
// 이벤트 타입 열거형
enum SyncEventType {
  FILE_CREATED = 'FILE_CREATED',     // 새 파일이 업로드됨
  FILE_MODIFIED = 'FILE_MODIFIED',   // 기존 파일이 수정됨 (같은 경로, 다른 내용)
  FILE_DELETED = 'FILE_DELETED',     // 파일이 삭제됨
  FOLDER_CREATED = 'FOLDER_CREATED', // 새 폴더가 생성됨
  FOLDER_DELETED = 'FOLDER_DELETED', // 폴더가 삭제됨 (하위 파일 포함)
}

// 이벤트 인터페이스
interface SyncEvent {
  id: string;                    // UUID v7 (시간 순서 보장)
  type: SyncEventType;           // 이벤트 타입
  source: 'primary' | 'secondary' | 'tertiary';  // 감지 전략
  path: string;                  // LGU+ 웹하드 상의 전체 경로
  fileName: string;              // 파일명
  fileSize: number;              // 파일 크기 (bytes)
  mtime: number;                 // 수정 시각 (Unix timestamp ms)
  parentPath: string;            // 부모 폴더 경로
  fileKey: string;               // 중복 제거용 키
  priority: number;              // 처리 우선순위 (낮을수록 높음)
  createdAt: number;             // 이벤트 생성 시각
  retryCount: number;            // 재시도 횟수
  metadata: Record<string, unknown>;  // 추가 메타데이터
}

// 이벤트 처리 결과
interface SyncEventResult {
  eventId: string;
  status: 'completed' | 'failed' | 'skipped';
  error?: string;
  downloadedBytes?: number;
  uploadedBytes?: number;
  durationMs: number;
  completedAt: number;
}
```

### 3.2 이벤트 필터링 규칙

감지된 이벤트 중 처리가 불필요한 이벤트를 걸러내는 필터링 단계이다.

```
┌──────────────────────────────────────────────────────────┐
│                이벤트 필터링 파이프라인                     │
└──────────────────────────────────────────────────────────┘

  원시 이벤트
       │
       ▼
  ┌─────────────────────────┐
  │ 1. 경로 필터             │  "올리기전용/" 하위만 허용
  │    허용: 올리기전용/**   │  "내리기전용/", 기타 → 폐기
  └────────┬────────────────┘
           │ 통과
           ▼
  ┌─────────────────────────┐
  │ 2. 파일 확장자 필터       │  .tmp, .partial, .crdownload → 폐기
  │    제외: 임시 파일       │  (다운로드 중인 임시 파일)
  └────────┬────────────────┘
           │ 통과
           ▼
  ┌─────────────────────────┐
  │ 3. 파일명 패턴 필터       │  ~$*, .~lock.*, Thumbs.db → 폐기
  │    제외: 시스템 파일     │  desktop.ini, .DS_Store → 폐기

  │    제외: 빈 파일         │  > 10GB → 경고 + 수동 처리
  └────────┬────────────────┘
           │ 통과
           ▼
  ┌─────────────────────────┐
  │ 5. 중복 제거             │  fileKey가 이미 처리된
  │    제외: 이미 처리됨     │  이벤트면 폐기
  └────────┬────────────────┘
           │ 통과
           ▼
  유효한 이벤트 → 큐잉
```

### 3.3 이벤트 우선순위

이벤트는 타입에 따라 우선순위가 부여되며, 우선순위가 높은 이벤트가 먼저 처리된다.

```
  우선순위 (숫자가 낮을수록 높은 우선순위)

  ┌──────────────────────────────────────────────────┐
  │  1 (최고)  │  FOLDER_CREATED                     │
  │            │  폴더가 먼저 생성되어야 파일을 넣을   │
  │            │  수 있으므로 최우선                   │
  ├────────────┼─────────────────────────────────────┤
  │  2         │  FILE_CREATED                       │
  │            │  새 파일 동기화가 핵심 기능           │
  ├────────────┼─────────────────────────────────────┤
  │  3         │  FILE_MODIFIED                      │
  │            │  기존 파일 업데이트                   │
  ├────────────┼─────────────────────────────────────┤
  │  4         │  FOLDER_DELETED                     │
  │            │  폴더 삭제는 하위 파일 삭제 후 처리   │
  ├────────────┼─────────────────────────────────────┤
  │  5 (최저)  │  FILE_DELETED                       │
  │            │  삭제는 가장 낮은 우선순위            │
  └──────────────────────────────────────────────────┘
```

**동일 우선순위 내 정렬:**

1. 이벤트 생성 시각 (`createdAt`) 오름차순 (먼저 감지된 것 먼저)
2. 재시도 횟수 (`retryCount`) 오름차순 (처음 시도하는 것 먼저)
3. 파일 크기 (`fileSize`) 오름차순 (작은 파일 먼저 → 빠른 처리)

## 4. 다운로드 관리

### 4.1 다운로드 아키텍처 개요

다운로드 관리자는 이벤트 큐에서 처리 대상 이벤트를 가져와 LGU+ 웹하드에서 파일을 다운로드하는 역할을 담당한다.

```
┌──────────────────────────────────────────────────────────────────┐
│                      다운로드 관리자 구조                          │
└──────────────────────────────────────────────────────────────────┘

                    ┌───────────────────┐
                    │    이벤트 큐       │
                    │  (Priority Queue) │
                    └────────┬──────────┘
                             │ dequeue()
                             ▼
                    ┌───────────────────┐
                    │  다운로드 관리자    │
                    │ DownloadManager   │
                    └────────┬──────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              ▼              ▼
       ┌───────────┐  ┌───────────┐  ┌───────────┐
       │ Worker #1 │  │ Worker #2 │  │ Worker #3 │   ... 최대 5개
       │ (슬롯 1)  │  │ (슬롯 2)  │  │ (슬롯 3)  │
       └─────┬─────┘  └─────┬─────┘  └─────┬─────┘
             │              │              │
             ▼              ▼              ▼
       ┌───────────────────────────────────────┐
       │           임시 디렉토리                 │
       │  ./temp/downloads/                    │
       │    ├── {uuid1}.tmp                    │
       │    ├── {uuid2}.tmp                    │
       │    └── {uuid3}.tmp                    │
       └───────────────────────────────────────┘
             │ 검증 완료 후
             ▼
       ┌───────────────────────────────────────┐
       │           완료 디렉토리                 │
       │  ./downloads/올리기전용/업체A/          │
       │    ├── file1.dxf                      │
       │    └── file2.dxf                      │
       └───────────────────────────────────────┘
```

### 4.2 동시 다운로드 수 제한

p-queue 라이브러리를 사용하여 동시 다운로드 수를 제한한다.


**동시성 자동 조절:**

```
  성공률 모니터링 기반 동시성 자동 조절

  성공률 > 95%   → concurrency + 1 (최대 maxConcurrency까지)
  성공률 < 80%   → concurrency - 1 (최소 minConcurrency까지)
  성공률 < 50%   → concurrency = 1 (최소값으로 급감)

  ┌──────────────────────────────────────────────────────┐
  │  성공률    │  동시성 조절        │  상태              │
  ├───────────┼────────────────────┼───────────────────┤
  │  100%     │  concurrency = 5   │  정상              │
  │   90%     │  concurrency = 4   │  약간 불안정        │
  │   70%     │  concurrency = 2   │  불안정             │
  │   40%     │  concurrency = 1   │  심각               │
  └──────────────────────────────────────────────────────┘
```

### 4.4 대용량 파일 처리 (스트림 기반)

파일 크기에 관계없이 메모리 효율적으로 다운로드하기 위해 스트림 기반 다운로드를 사용한다.

```
┌──────────────────────────────────────────────────────────┐
│              스트림 기반 다운로드 흐름                      │
└──────────────────────────────────────────────────────────┘

  LGU+ API 응답 (HTTP Response)
       │
       ▼
  ┌─────────────────────┐
  │ ReadableStream      │   HTTP 응답 바디 (청크 단위)
  │ (response.body)     │
  └────────┬────────────┘
           │ pipe
           ▼
  ┌─────────────────────┐
  │ TransformStream     │   진행률 추적 + 해시 계산
  │ (ProgressTracker)   │   - downloadedBytes 업데이트
  │                     │   - SHA-256 해시 누적 계산
  └────────┬────────────┘
           │ pipe
           ▼
  ┌─────────────────────┐
  │ WritableStream      │   임시 파일에 기록
  │ (FileWriter)        │   ./temp/downloads/{uuid}.tmp
  └─────────────────────┘
```

**파일 크기별 처리 전략:**

| 파일 크기 | 전략 | 메모리 사용량 |
|----------|------|-------------|
| < 10MB | 스트림 (기본) | ~64KB (버퍼 크기) |
| 10MB ~ 100MB | 스트림 + 진행률 표시 | ~64KB |
| 100MB ~ 1GB | 스트림 + 청크 검증 | ~64KB |
| > 1GB | 스트림 + 경고 로그 | ~64KB |


### 4.5 임시 파일에서 최종 위치로 이동 (Atomic Write)

다운로드가 중간에 실패해도 불완전한 파일이 최종 위치에 남지 않도록 원자적 쓰기 패턴을 사용한다.

```
┌──────────────────────────────────────────────────────────────┐
│                   Atomic Write 패턴                           │
└──────────────────────────────────────────────────────────────┘

  1단계: 임시 파일에 다운로드
  ┌──────────────────────────────────────────┐
  │ ./temp/downloads/a1b2c3d4.tmp            │ ← 다운로드 중
  │ (불완전한 파일이 여기에 쓰여짐)            │
  └──────────────────────────────────────────┘
           │ 다운로드 완료 + 검증 통과
           ▼
  2단계: 최종 디렉토리 확인/생성
  ┌──────────────────────────────────────────┐
  │ ./downloads/올리기전용/업체A/              │ ← mkdir -p (없으면 생성)
  └──────────────────────────────────────────┘
           │
           ▼
  3단계: 원자적 이동 (rename)
  ┌──────────────────────────────────────────┐
  │ fs.rename(                               │
  │   './temp/downloads/a1b2c3d4.tmp',       │
  │   './downloads/올리기전용/업체A/file1.dxf' │
  │ )                                        │
  └──────────────────────────────────────────┘

  ※ rename은 같은 파일시스템 내에서 원자적(atomic) 연산
  ※ 임시 디렉토리와 최종 디렉토리를 같은 드라이브에 배치
```

**동일 파일명 충돌 처리:**

```typescript
async function resolveFilePath(targetPath: string): Promise<string> {
  if (!await fileExists(targetPath)) {
    return targetPath;  // 충돌 없음
  }

  // 기존 파일과 새 파일의 크기/해시 비교
  const existingHash = await computeFileHash(targetPath);
  const newHash = await computeFileHash(tempPath);

  if (existingHash === newHash) {
    // 동일 파일 → 다운로드 스킵 (이미 존재)
    return null;
  }

  // 다른 파일 → 파일명에 타임스탬프 추가
  const ext = path.extname(targetPath);
  const base = path.basename(targetPath, ext);
  const dir = path.dirname(targetPath);
  const timestamp = Date.now();
  return path.join(dir, `${base}_${timestamp}${ext}`);
}
```

## 5. 업로드 관리 (인터페이스 설계)

### 5.1 개요

업로드 관리자는 LGU+ 웹하드에서 다운로드한 파일을 내부 웹하드(Cloudflare R2)로 업로드하는 역할을 담당한다. 내부 웹하드의 구현은 추후 제공되므로, 인터페이스를 먼저 정의하고 개발/테스트용 MockUploader를 제공한다.

### 5.2 IWebhardUploader 인터페이스 정의

```typescript
/**
 * 내부 웹하드 업로더 인터페이스
 *
 * 이 인터페이스를 구현하여 다양한 웹하드 백엔드에 파일을 업로드할 수 있다.
 * - MockUploader: 개발/테스트용 (로컬 파일시스템에 복사)
 * - R2Uploader: Cloudflare R2 업로드 (실제 운영용)
 * - ApiUploader: yjlaser_website API를 통한 업로드
 */
interface IWebhardUploader {
  /**
   * 업로더 초기화
   * 인증, 연결 설정 등 사전 작업 수행
   */
  initialize(): Promise<void>;

  /**
   * 단일 파일 업로드
   * @param request - 업로드 요청 정보
   * @returns 업로드 결과
   */
  upload(request: UploadRequest): Promise<UploadResult>;

  /**
   * 여러 파일 일괄 업로드
   * @param requests - 업로드 요청 배열
   * @returns 업로드 결과 배열
   */
  uploadBatch(requests: UploadRequest[]): Promise<UploadResult[]>;

  /**
   * 파일 존재 여부 확인
   * @param remotePath - 내부 웹하드 경로
   * @returns 파일 정보 (없으면 null)
   */
  checkExists(remotePath: string): Promise<FileInfo | null>;

  /**
   * 폴더 생성
   * @param remotePath - 생성할 폴더 경로
   */
  createFolder(remotePath: string): Promise<void>;

  /**
   * 파일 삭제
   * @param remotePath - 삭제할 파일 경로
   */
  deleteFile(remotePath: string): Promise<void>;

  /**
   * 업로더 종료
   * 연결 해제, 리소스 정리
   */
  dispose(): Promise<void>;

  /**
   * 업로더 상태 확인
   */
  healthCheck(): Promise<HealthCheckResult>;
}

interface UploadRequest {
  localPath: string;         // 로컬 파일 경로
  remotePath: string;        // 내부 웹하드 대상 경로
  fileName: string;          // 파일명
  fileSize: number;          // 파일 크기
  mimeType?: string;         // MIME 타입
  metadata?: Record<string, string>;  // 추가 메타데이터
  onProgress?: (progress: UploadProgress) => void;  // 진행률 콜백
}

interface UploadResult {
  success: boolean;
  remotePath: string;
  uploadedBytes: number;
  durationMs: number;
  etag?: string;             // 서버 측 파일 식별자
  error?: string;
}

interface UploadProgress {
  uploadedBytes: number;
  totalBytes: number;
  percentage: number;
}

interface FileInfo {
  path: string;
  size: number;
  lastModified: number;
  etag?: string;
}

interface HealthCheckResult {
  healthy: boolean;
  latencyMs: number;
  message?: string;
}
```

### 5.3 MockUploader (개발/테스트용)

```typescript
/**
 * 개발 및 테스트용 Mock 업로더
 *
 * 실제 외부 서비스에 연결하지 않고 로컬 파일시스템에
 * 파일을 복사하여 업로드를 시뮬레이션한다.
 */
class MockUploader implements IWebhardUploader {
  private mockStorageDir: string;  // 예: ./mock-storage/

  constructor(config: { storageDir: string }) {
    this.mockStorageDir = config.storageDir;
  }

  async initialize(): Promise<void> {
    await fs.mkdir(this.mockStorageDir, { recursive: true });
    logger.info('[MockUploader] 초기화 완료', { storageDir: this.mockStorageDir });
  }

  async upload(request: UploadRequest): Promise<UploadResult> {
    const startTime = Date.now();
    const targetPath = path.join(this.mockStorageDir, request.remotePath);

    // 디렉토리 생성
    await fs.mkdir(path.dirname(targetPath), { recursive: true });

    // 파일 복사 (스트림 기반)
    await pipeline(
      createReadStream(request.localPath),
      createWriteStream(targetPath)
    );

    const durationMs = Date.now() - startTime;
    logger.info('[MockUploader] 업로드 완료', {
      fileName: request.fileName,
      remotePath: request.remotePath,
      durationMs
    });

    return {
      success: true,
      remotePath: request.remotePath,
      uploadedBytes: request.fileSize,
      durationMs,
      etag: `mock-${Date.now()}`
    };
  }

  async uploadBatch(requests: UploadRequest[]): Promise<UploadResult[]> {
    return Promise.all(requests.map(req => this.upload(req)));
  }

  async checkExists(remotePath: string): Promise<FileInfo | null> {
    const fullPath = path.join(this.mockStorageDir, remotePath);
    try {
      const stat = await fs.stat(fullPath);
      return {
        path: remotePath,
        size: stat.size,
        lastModified: stat.mtimeMs,
        etag: `mock-${stat.mtimeMs}`
      };
    } catch {
      return null;
    }
  }

  async createFolder(remotePath: string): Promise<void> {
    await fs.mkdir(path.join(this.mockStorageDir, remotePath), { recursive: true });
  }

  async deleteFile(remotePath: string): Promise<void> {
    await fs.unlink(path.join(this.mockStorageDir, remotePath));
  }

  async dispose(): Promise<void> {
    logger.info('[MockUploader] 종료');
  }
```

## 6. 재시도 전략

### 6.1 에러 분류 체계

모든 에러를 5가지 카테고리로 분류하고, 각 카테고리별로 다른 재시도 정책을 적용한다.

```
┌──────────────────────────────────────────────────────────────────┐
│                     에러 분류 및 재시도 정책                       │
└──────────────────────────────────────────────────────────────────┘

  에러 발생
       │
       ▼
  ┌─────────────────────┐
  │ 에러 분류기           │
  │ (ErrorClassifier)    │
  └────────┬────────────┘
           │
     ┌─────┼─────┬─────────┬───────────┐
     │     │     │         │           │
     ▼     ▼     ▼         ▼           ▼
  ┌──────┐ ┌──────┐ ┌──────────┐ ┌────────┐ ┌──────────┐
  │Netwrk│ │ Auth │ │RateLimit │ │FileSys │ │ Server  │
  │Error │ │Error │ │Error     │ │Error   │ │ Error   │
  └──┬───┘ └──┬───┘ └────┬─────┘ └───┬────┘ └────┬────┘
     │        │          │           │            │
     ▼        ▼          ▼           ▼            ▼
  지수백오프  재인증후   Retry-After  즉시실패    고정간격
  1→2→4→8→16 1회재시도  헤더 대기    (재시도X)    30초
  max 5회    max 1회    max 3회                  max 3회
```

### 6.2 에러별 상세 재시도 정책

#### 6.2.1 NetworkError (네트워크 오류)

DNS 실패, 연결 타임아웃, 연결 거부, 소켓 끊김 등 네트워크 수준의 오류.

```typescript
interface NetworkRetryPolicy {
  type: 'exponential_backoff';
  initialDelayMs: 1000;     // 첫 재시도: 1초 후
  multiplier: 2;             // 배수: 2배
  maxDelayMs: 16000;         // 최대 대기: 16초
  maxRetries: 5;             // 최대 재시도: 5회
  jitter: true;              // 지터 추가 (동시 재시도 분산)
}
```

**재시도 타이밍:**

```
  시도 1 (원본)  → 실패
       │
       ├── 1초 대기 (±0.5초 지터)
       ▼
  시도 2 (재시도 1) → 실패
       │
       ├── 2초 대기 (±1초 지터)
       ▼
  시도 3 (재시도 2) → 실패
       │
       ├── 4초 대기 (±2초 지터)
       ▼
  시도 4 (재시도 3) → 실패
       │
       ├── 8초 대기 (±4초 지터)
       ▼
  시도 5 (재시도 4) → 실패
       │
       ├── 16초 대기 (±8초 지터)
       ▼
  시도 6 (재시도 5) → 실패 → DLQ 이동
```

#### 6.2.2 AuthError (인증 오류)

HTTP 401/403, 쿠키 만료, 세션 무효 등 인증 관련 오류.

```typescript
interface AuthRetryPolicy {
  type: 'reauth_then_retry';
  maxRetries: 1;              // 재인증 후 1회만 재시도
  reauthTimeoutMs: 30000;     // 재인증 타임아웃: 30초
}
```

**재인증 흐름:**

```
  API 요청 → 401 Unauthorized
       │
       ▼
  ┌─────────────────────────┐
  │ 인증 관리자에 재인증 요청 │
  │ (AuthManager.reauth())  │
  └────────┬────────────────┘
           │
     ┌─────┴─────┐
     │            │
  재인증 성공   재인증 실패
     │            │
     ▼            ▼
  원래 요청     DLQ 이동
  1회 재시도    + 사용자 알림
     │         (수동 로그인 필요)
     │
  ┌──┴──┐
  │     │
  성공  실패
  │     │
  ▼     ▼
 완료  DLQ 이동
```

#### 6.2.3 RateLimitError (API 제한 초과)

HTTP 429 Too Many Requests 또는 API 사용량 제한 초과.

```typescript
interface RateLimitRetryPolicy {
  type: 'retry_after';
  maxRetries: 3;                    // 최대 3회 재시도
  defaultDelayMs: 60000;            // Retry-After 헤더 없을 시 기본 60초
  maxDelayMs: 300000;               // 최대 대기: 5분
  respectRetryAfterHeader: true;    // Retry-After 헤더 값 사용
}
```

**동작 흐름:**

```
  API 요청 → 429 Too Many Requests
       │       Retry-After: 45
       │
       ├── 45초 대기 (Retry-After 헤더 값)
       ▼
  시도 2 → 429
       │    Retry-After: 30
       │
       ├── 30초 대기
       ▼
  시도 3 → 429
       │
       ├── 60초 대기 (기본값)
       ▼
  시도 4 → 실패 → DLQ 이동
```

#### 6.2.4 FileSystemError (파일시스템 오류)

디스크 공간 부족, 권한 오류, 파일 잠금 등 로컬 파일시스템 오류.

```typescript
interface FileSystemRetryPolicy {
  type: 'no_retry';
  maxRetries: 0;                    // 재시도 없음
  action: 'immediate_fail';         // 즉시 실패 처리
  notifyUser: true;                 // 사용자에게 알림 표시
}
```

**이유**: 파일시스템 오류는 재시도해도 해결되지 않는 경우가 대부분이다. 디스크 공간 확보, 권한 설정 등 사용자 개입이 필요하다.

#### 6.2.5 ServerError (서버 오류, 5xx)

HTTP 500, 502, 503, 504 등 서버 측 오류.

```typescript
interface ServerRetryPolicy {
  type: 'fixed_interval';
  delayMs: 30000;                   // 고정 30초 간격
  maxRetries: 3;                    // 최대 3회 재시도
}
```

**동작 흐름:**

```
  API 요청 → 500 Internal Server Error
       │
       ├── 30초 대기
       ▼
  시도 2 → 502 Bad Gateway
       │
       ├── 30초 대기
       ▼
  시도 3 → 503 Service Unavailable
       │
       ├── 30초 대기
       ▼
  시도 4 → 실패 → DLQ 이동
```

**재시도 큐 처리 루프:**

```
┌──────────────────────────────────────────────────────────┐

└──────────────────────────────────────────────────────────┘

  ┌─────────────────┐

           ▼

  │ 각 항목에 대해:                  │
```

### 6.4 Dead Letter Queue (DLQ) 정책

재시도 횟수를 모두 소진한 이벤트를 보관하는 DLQ이다.

```
┌──────────────────────────────────────────────────────────┐
│                Dead Letter Queue (DLQ) 구조               │
└──────────────────────────────────────────────────────────┘

  ┌──────────────────────────────────────────────────┐
  │ DLQ (SQLite dead_letter_queue 테이블)            │
  │                                                  │
  │ ┌────┬───────────┬──────────┬─────────┬────────┐ │
  │ │ ID │ 파일명     │ 에러유형  │ 시도횟수 │ 시각   │ │
  │ ├────┼───────────┼──────────┼─────────┼────────┤ │
  │ │ 1  │ file1.dxf │ Network  │ 5/5    │ 10:30  │ │
  │ │ 2  │ file2.pdf │ Auth     │ 2/2    │ 10:45  │ │
  │ │ 3  │ file3.ai  │ Server   │ 4/4    │ 11:00  │ │
  │ └────┴───────────┴──────────┴─────────┴────────┘ │
  │                                                  │
  │ 수동 액션:                                        │
  │  [재시도] [삭제] [전체 재시도] [전체 삭제]          │
  └──────────────────────────────────────────────────┘
```

**DLQ 관리 정책:**

```typescript
interface DLQConfig {
  // DLQ에 이벤트가 추가될 때 GUI 알림 표시
  notifyOnAdd: true;

  // DLQ 항목 최대 보존 기간 (30일)
  maxRetentionMs: 30 * 24 * 3600000;

  // DLQ 최대 항목 수 (1000개 초과 시 가장 오래된 항목 자동 삭제)
  maxEntries: 1000;

  // DLQ 알림 반복 간격 (DLQ에 항목이 있을 때 1시간마다 GUI 알림)
  notifyIntervalMs: 3600000;
}
```

**DLQ 수동 재처리 (GUI에서):**

1. **개별 재시도**: 특정 DLQ 항목을 선택하여 재시도 큐로 이동 (retryCount 초기화)
2. **전체 재시도**: DLQ의 모든 항목을 재시도 큐로 이동
3. **개별 삭제**: 특정 DLQ 항목을 영구 삭제
4. **전체 삭제**: DLQ 비우기
5. **상세 조회**: DLQ 항목의 에러 상세 정보 확인 (에러 메시지, 스택 트레이스)

### 7.5 슬라이딩 윈도우 구현

```typescript
class SlidingWindow {
  private results: Array<{ success: boolean; timestamp: number }>;
  private readonly windowSize: number;

  constructor(windowSize: number) {
    this.results = [];
    this.windowSize = windowSize;
  }

  record(success: boolean): void {
    this.results.push({ success, timestamp: Date.now() });
    // 윈도우 크기 초과 시 가장 오래된 결과 제거
    if (this.results.length > this.windowSize) {
      this.results.shift();
    }
  }

  getFailureRate(): number {
    if (this.results.length === 0) return 0;
    const failures = this.results.filter(r => !r.success).length;
    return (failures / this.results.length) * 100;
  }

  reset(): void {
    this.results = [];
  }
}
```

## 8. 체크포인트 시스템

### 8.1 개요

체크포인트 시스템은 동기화 엔진의 진행 상태를 영속적으로 저장하여, 프로그램 재시작이나 비정상 종료 후에도 마지막 처리 지점부터 이어서 작업할 수 있게 한다.

### 8.3 저장 타이밍

체크포인트는 다음 시점에 저장된다:

```
┌──────────────────────────────────────────────────────────────┐
│                  체크포인트 저장 타이밍                         │
└──────────────────────────────────────────────────────────────┘

  1. 이벤트 단위 저장 (매 이벤트 처리 완료 후)
     ┌────────────┐     ┌────────────┐     ┌────────────┐
     │ 이벤트 처리 │────▶│ 결과 기록  │────▶│ 체크포인트  │
     │   완료     │     │  (DB)     │     │   저장     │
     └────────────┘     └────────────┘     └────────────┘

  2. 주기적 저장 (30초 간격)
     ┌─────────────────┐
     │ 30초 타이머 발동  │────▶ 전체 체크포인트 플러시
     └─────────────────┘

  3. 상태 전이 시 저장 (엔진 상태가 변경될 때)
     ┌────────────────────────┐
     │ 상태 변경 감지           │────▶ 관련 체크포인트 저장
     │ (RUNNING → PAUSED 등)  │
     └────────────────────────┘

  4. 종료 시 저장 (정상 종료)
     ┌──────────────────┐
     │ 엔진 종료 시그널  │────▶ 전체 체크포인트 플러시 + DB sync
     └──────────────────┘
```

**체크포인트 저장 구현:**

```typescript
class CheckpointManager {
  private store: Map<string, unknown>;
  private dirty: Set<string>;      // 변경된 키 추적
  private db: Database;
  private flushIntervalMs: 30000;

  // 단일 키 저장 (더티 마킹만, 실제 DB 기록은 플러시 시)
  set(key: string, value: unknown): void {
    this.store.set(key, value);
    this.dirty.add(key);
  }

  // 즉시 DB에 기록 (중요한 체크포인트)
  async setImmediate(key: string, value: unknown): Promise<void> {
    this.store.set(key, value);
    await this.db.upsertCheckpoint(key, value);
    this.dirty.delete(key);
  }

  // 더티 체크포인트 일괄 플러시
  async flush(): Promise<void> {
    if (this.dirty.size === 0) return;

    const entries = Array.from(this.dirty).map(key => ({
      key,
      value: this.store.get(key)
    }));

    await this.db.batchUpsertCheckpoints(entries);
    this.dirty.clear();
  }

  // DB에서 복원
  async restore(): Promise<void> {
    const entries = await this.db.getAllCheckpoints();
    for (const { key, value } of entries) {
      this.store.set(key, value);
    }
  }
}
```

### 8.4 복구 시나리오별 동작

#### 시나리오 1: 정상 종료 후 재시작

```
  종료 시: 모든 체크포인트 플러시됨
  재시작 시:
    1. DB에서 체크포인트 복원
    2. primary.last_history_no부터 폴링 재개
    3. 미처리 이벤트(events.pending_count > 0) 확인 → 큐에 재추가
    4. 정상 동작 시작
```

#### 시나리오 2: 비정상 종료 (강제 종료, 전원 차단)

```
  종료 시: 마지막 플러시 시점까지만 저장됨 (최대 30초 데이터 손실)
  재시작 시:
    1. DB에서 체크포인트 복원
    2. primary.last_history_no부터 폴링 재개
       → 일부 이벤트가 중복 감지될 수 있으나 중복 제거기가 처리
    3. 임시 다운로드 디렉토리 정리 (불완전한 .tmp 파일 삭제)
    4. DB의 status='processing' 레코드를 'pending'으로 되돌림
    5. 정상 동작 시작
```

#### 시나리오 3: 긴 오프라인 기간 후 재시작

```
  종료 시: 정상 체크포인트 저장
  1일+ 후 재시작 시:
    1. DB에서 체크포인트 복원
    2. primary.last_history_no 이후의 히스토리 조회
       → 히스토리 보관 기간 초과 시 빈 결과 반환 가능
    3. Secondary(스냅샷 비교)를 즉시 실행하여 전체 현재 상태 파악
    4. Tertiary(무결성 검증)를 즉시 실행하여 누락 파일 확인
    5. 정상 동작 시작
```

#### 시나리오 4: DB 파일 손상/삭제

```
  재시작 시:
    1. DB 파일 없음 감지 → 새 DB 생성 + 마이그레이션 실행
    2. 모든 체크포인트 초기값으로 시작
    3. Secondary(스냅샷 비교)를 즉시 실행하여 현재 상태를 초기 스냅샷으로 저장
    4. 이후 변경분부터 동기화 시작
    ※ 기존 동기화 이력이 사라지므로 일부 파일이 재동기화될 수 있음
       (멱등성 보장으로 문제 없음)
```

### 10.2 파일별 Lock (같은 파일 동시 처리 방지)

같은 파일에 대한 이벤트가 동시에 처리되는 것을 방지한다.

```typescript
class FileLockManager {
  private locks: Map<string, { holder: string; acquiredAt: number }>;
  private lockTimeoutMs: 300000;  // 5분 타임아웃 (무한 잠금 방지)

  /**
   * 파일 Lock 획득 시도
   * @param filePath - 잠글 파일 경로
   * @param holder - Lock 소유자 (이벤트 ID)
   * @returns Lock 획득 성공 여부
   */
  async acquire(filePath: string, holder: string): Promise<boolean> {
    const existing = this.locks.get(filePath);

    if (existing) {
      // 기존 Lock 타임아웃 확인
      if (Date.now() - existing.acquiredAt > this.lockTimeoutMs) {
        // 타임아웃 → 강제 해제 후 새 Lock 획득
        logger.warn('파일 Lock 타임아웃, 강제 해제', {
          filePath,
          previousHolder: existing.holder
        });
        this.locks.delete(filePath);
      } else {
        return false;  // Lock 획득 실패
      }
    }

    this.locks.set(filePath, { holder, acquiredAt: Date.now() });
    return true;
  }

  /**
   * 파일 Lock 해제
   */
  release(filePath: string, holder: string): void {
    const existing = this.locks.get(filePath);
    if (existing && existing.holder === holder) {
      this.locks.delete(filePath);
    }
  }
}
```

**Lock 사용 패턴:**

```
  이벤트 처리 시작
       │
       ▼
  ┌─────────────────────────┐
  │ fileLock.acquire(path)  │
  └────────┬────────────────┘
           │
     ┌─────┴─────┐
     │           │
  획득 성공    획득 실패
     │           │
     ▼           ▼
  ┌─────────┐  이벤트 큐에
  │ 다운로드 │  재추가
  │ + 업로드 │  (1초 후)
  │ + 검증   │
  └────┬────┘
       │
       ▼

```

```typescript
// 메인 → 렌더러 (단방향)
interface IPCChannels {
  // 전체 메트릭 업데이트 (1초 간격)
  'sync:metrics': SyncMetrics;

  // 개별 파일 진행률 (실시간)
  'sync:progress': DownloadProgress | UploadProgress;

  // 에러 알림 (발생 시 즉시)
  'sync:error': {
    type: string;
    message: string;
    details?: unknown;
    timestamp: number;
  };

  // 엔진 상태 변경 (상태 전이 시)
  'sync:status': {
    previousState: string;
    currentState: string;
    reason: string;
    timestamp: number;
  };

  // DLQ 알림 (항목 추가 시)
  'sync:dlq': {
    action: 'added' | 'removed' | 'cleared';
    entry?: DLQEntry;
    totalCount: number;
  };

  // 인증 상태 변경
  'sync:auth': {
    status: 'authenticated' | 'expired' | 'reauth_required' | 'reauth_success' | 'reauth_failed';
    message: string;
  };
}

// 렌더러 → 메인 (명령)
interface IPCCommands {
  // 엔진 제어
  'sync:start': void;
  'sync:pause': void;
  'sync:resume': void;
  'sync:stop': void;

  // DLQ 관리
  'sync:dlq:retry': { eventId: string };
  'sync:dlq:retryAll': void;
  'sync:dlq:delete': { eventId: string };
  'sync:dlq:deleteAll': void;

  // 설정 변경
  'sync:config:update': Partial<SyncEngineConfig>;

  // 수동 트리거
```
